系统日志里频繁出现“Timed out waiting for time from RTC”的警告，或者设备重启后时间总是回到1970年——这通常是RTC驱动有问题或电池没电了。RTC（Real Time Clock）是嵌入式系统的“电子手表”，即使在系统断电时，也能依靠备用电池（如CR2032）维持计时。本章我们深入内核，看看如何让这块“手表”准确工作。

RTC子系统架构与核心结构

Linux内核将RTC功能抽象为一个独立的子系统，对上提供统一的字符设备接口（如/dev/rtc0），对下则通过struct rtc_class_ops来适配不同的硬件芯片。这个设计模式，和我们在第8章学习的输入子系统非常相似。

驱动开发者的主要工作就是填充一个rtc_class_ops结构体，并向内核注册一个rtc_device。这个结构体定义了驱动必须实现的一组回调函数。下表列出了最关键的几个操作：

操作函数作用是否必须典型实现内容
read_time从硬件读取当前时间是读取芯片的秒、分、时、日、月、年寄存器，并转换为struct rtc_time。
set_time向硬件设置时间是将struct rtc_time分解，写入芯片对应寄存器。
read_alarm读取设置的闹钟时间否（支持闹钟时必需）读取闹钟寄存器。
set_alarm设置闹钟时间否（支持闹钟时必需）配置闹钟寄存器并启用中断。
alarm_irq_enable启用/禁用闹钟中断否（支持闹钟时必需）操作芯片的中断使能位。
proc在/proc/driver/rtc显示信息否提供芯片特定信息，如电压状态。

关键点：时间格式的转换

芯片寄存器存储的时间格式五花八门：BCD码（0x59表示59秒）、12/24小时制、世纪位。而内核使用的struct rtc_time是十进制表示。你必须仔细阅读芯片数据手册，在驱动中正确进行格式转换。我在一个DS1337驱动项目里，就因为忽略了它的“世纪位”（第7个月寄存器的最高位），导致时间在2000年前后计算错误。

编写一个I2C RTC驱动

市面上大多数RTC芯片（如Microchip的DS1307、NXP的PCF8563）通过I2C总线连接。结合第9章的I2C总线知识，编写驱动的流程非常清晰。我们以一款虚拟的“VendorRTC”芯片为例，它的I2C地址是0x68。

首先，驱动需要声明它支持的设备。在设备树普及的今天，OF匹配表是标准做法：

/* 设备树节点示例 */
// i2c1: &i2c1 {
//     rtc@68 {
//         compatible = "vendor,rtc-001";
//         reg = <0x68>;
//         interrupt-parent = <&gpio0>;
//         interrupts = <5 IRQ_TYPE_EDGE_FALLING>; // 闹钟中断引脚
//     };
// };

static const struct of_device_id vendor_rtc_of_match[] = {
    { .compatible = "vendor,rtc-001" },
    { }
};
MODULE_DEVICE_TABLE(of, vendor_rtc_of_match);

static struct i2c_driver vendor_rtc_driver = {
    .driver = {
        .name   = "vendor-rtc",
        .of_match_table = vendor_rtc_of_match,
    },
    .probe    = vendor_rtc_probe,
    .remove   = vendor_rtc_remove,
    .id_table = vendor_rtc_id, // 用于非设备树匹配
};

接下来是驱动核心，实现read_time和set_time。注意寄存器地址和BCD转换：

#define REG_SECONDS   0x00
#define REG_MINUTES   0x01
#define REG_HOURS     0x02 // 位6表示12/24小时制

static int vendor_rtc_read_time(struct device *dev, struct rtc_time *tm)
{
    struct i2c_client *client = to_i2c_client(dev);
    u8 regs[7];
    int err;

    err = i2c_smbus_read_i2c_block_data(client, REG_SECONDS, 7, regs);
    if (err < 0) {
        dev_err(dev, "read error %d\n", err);
        return err;
    }

    tm->tm_sec  = bcd2bin(regs[0] & 0x7F); // 忽略最高位时钟停止位
    tm->tm_min  = bcd2bin(regs[1] & 0x7F);
    tm->tm_hour = bcd2bin(regs[2] & 0x3F); // 假设为24小时制
    // ... 解析日、月、年、星期
    tm->tm_year += 100; // RTC年通常从1900算起，需要偏移

    return 0;
}

常见误区

  • 忽略时钟停止位：许多RTC芯片（如DS1307）的秒寄存器最高位（BIT7）是时钟停止位（CH）。读取时间时需要屏蔽此位，设置时间时如果CH=1，时钟会停止运行。这是一个经典的坑。
  • 时制混淆：芯片可能支持12/24小时制，寄存器格式不同。驱动必须统一转换为24小时制再填充rtc_time。
  • 年偏移错误：内核rtc_time.tm_year是从1900年开始的偏移年数。而芯片寄存器可能只存0-99，表示2000-2099年。忘记做“+100”或“+2000”的偏移，会导致时间显示错误。

实现闹钟与周期性中断

闹钟功能让RTC从简单的计时器变为可唤醒系统的“闹钟”。实现它需要三个步骤：支持set_alarm/read_alarm、实现alarm_irq_enable，以及正确的中断处理。

首先，在probe函数中申请中断线。实际开发中，大部分团队会使用devm_request_threaded_irq来管理中断资源：

static int vendor_rtc_probe(struct i2c_client *client)
{
    struct vendor_rtc *rtc;
    int ret;

    rtc = devm_kzalloc(&client->dev, sizeof(*rtc), GFP_KERNEL);
    // ... 初始化

    if (client->irq > 0) {
        ret = devm_request_threaded_irq(&client->dev, client->irq,
                NULL, vendor_rtc_alarm_irq,
                IRQF_TRIGGER_FALLING | IRQF_ONESHOT,
                "vendor-rtc-alarm", rtc);
        if (ret) {
            dev_warn(&client->dev, "IRQ %d not available, alarm disabled\n", client->irq);
            client->irq = 0;
        }
    }

    rtc->rtc_dev = devm_rtc_device_register(&client->dev, "vendor_rtc",
                        &vendor_rtc_ops, THIS_MODULE);
    return PTR_ERR_OR_ZERO(rtc->rtc_dev);
}

概念：闹钟中断处理

生活类比：就像你设好闹钟后睡觉，闹钟响（中断触发）把你叫醒，你按掉闹钟（清除中断标志）并开始新的一天（执行唤醒任务）。
技术定义：当RTC芯片的计时到达预设的闹钟时间时，会拉低中断引脚电平，触发内核中断。驱动需要在中断处理函数中，清除芯片内部的中断标志位，并通过rtc_update_irq()函数通知RTC核心层，最终可能唤醒等待/dev/rtc的进程或触发系统唤醒。

中断处理函数的核心是调用rtc_update_irq()：

static irqreturn_t vendor_rtc_alarm_irq(int irq, void *dev_id)
{
    struct vendor_rtc *rtc = dev_id;
    int status;

    // 1. 读取并清除芯片的中断标志位
    status = i2c_smbus_read_byte_data(rtc->client, REG_CONTROL);
    i2c_smbus_write_byte_data(rtc->client, REG_CONTROL, status & ~ALARM_INT_FLAG);

    // 2. 通知RTC核心层发生了闹钟中断
    rtc_update_irq(rtc->rtc_dev, 1, RTC_IRQF | RTC_AF);

    return IRQ_HANDLED;
}

编者提示：中断处理中的资源竞争和同步问题至关重要。关于中断上下文的限制、如何安全地访问共享数据，请回顾第5章《中断处理程序设计》的详细讲解。

除了闹钟，一些高端RTC芯片（如DS3231）还支持可编程方波输出（1Hz, 1kHz等）或周期性中断（如每秒一次）。这通过实现rtc_class_ops中的irq_set_freq或read_offset/set_offset（用于校准）来支持。对于时间精度要求高的应用（如基站同步），ppm（百万分之一）级别的校准是必要的。

消费电子

  • 使用低成本I2C RTC (DS1307)
  • 主要需求：保持日期时间
  • 精度要求：±2分钟/月

工业控制

  • 使用带温补的RTC (DS3231)
  • 需求：事件时间戳、定时唤醒
  • 精度要求：±2分钟/年

通信设备

  • 使用高精度RTC模块 (RX-8025SA)
  • 需求：网络时间同步基准
  • 精度要求：±3.8ppm (约±2秒/周)

调试与测试你的驱动

驱动编写完成后，不要急于集成到系统。先进行模块化测试。加载驱动后，检查/proc/driver/rtc和/sys/class/rtc/rtc0/下的信息是否正确。

使用用户空间工具hwclock是最直接的测试方法：

# 将系统时间写入RTC硬件
$ hwclock -w -f /dev/rtc0
# 从RTC硬件读取时间到系统
$ hwclock -s -f /dev/rtc0
# 查看RTC详细信息
$ hwclock --debug -f /dev/rtc0

如果hwclock操作失败，查看dmesg输出。常见的错误信息包括：

  • "rtc core: registered rtc_name as rtcX" – 注册成功。
  • "invalid date" – 从芯片读取的时间数据转换失败，检查BCD转换和寄存器掩码。
  • "i2c i2c-0: sendbytes: error -121" – I2C通信失败，检查地址、上拉电阻或设备树配置。

对于闹钟测试，你可以编写一个小程序，使用ioctl（RTC_ALM_SET, RTC_AIE_ON）设置一个几十秒后的闹钟，然后read(fd, &data, sizeof(data))进行阻塞读取，观察闹钟触发时程序是否被唤醒。

动手试一试

1. 虚拟RTC驱动框架：在不依赖真实硬件的情况下，创建一个虚拟的“dummy_rtc”内核模块。实现read_time和操作，让它们读写一个内核内存中的struct rtc_time变量。使用misc_register（参考第30章）或rtc_device_register将其注册为RTC设备。加载后，尝试用hwclock命令进行读写。

2. 闹钟模拟测试：在上述虚拟驱动中，添加一个内核定时器（参考第27章）。在set_alarm中启动这个定时器，在定时器到期函数中调用rtc_update_irq来模拟闹钟中断。编写一个用户空间C程序，设置闹钟并等待中断，验证整个通路是否工作。

检验你的理解

  1. 当用户程序调用ioctl(fd, RTC_SET_TIME, &tm)时，内核空间的数据流经过了哪些关键函数和结构体，最终如何到达RTC芯片的寄存器？
  2. 如果一款RTC芯片的“年”寄存器存储的是0-99的BCD码，代表2000-2099年。在read_time函数中，你需要如何正确设置tm->tm_year？如果芯片还有一个独立的“世纪位”（Century Bit）在另一个寄存器，方案又该如何调整？
  3. 在闹钟中断处理函数vendor_rtc_alarm_irq中，为什么必须先清除芯片的中断标志位，再调用rtc_update_irq？顺序反过来会导致什么问题？

本章小结

  • Linux RTC子系统采用“核心-驱动”分层模型，驱动通过实现struct rtc_class_ops中的回调函数来适配具体硬件。
  • 编写RTC驱动的核心是正确实现时间读写（read_time/set_time），关键在于处理BCD码、时制、时钟停止位和年偏移等格式转换。
  • 闹钟功能需要实现set_alarm、alarm_irq_enable并处理中断，在中断处理中清除芯片标志并调用rtc_update_irq()通知上层。
  • 使用hwclock工具和检查/proc/driver/rtc、内核日志是调试RTC驱动的基本方法。
  • 根据应用场景（消费电子、工业、通信）对精度的不同要求，选择合适的RTC芯片并可能实现校准功能。
