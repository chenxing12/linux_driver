核心概念性能剖析缓存友好

一个网络驱动在千兆网卡上只能跑到300Mbps，一个传感器驱动导致系统响应延迟超过100ms。这些都不是功能bug，而是性能缺陷。内核驱动作为硬件与系统的桥梁，其性能直接决定了整个设备的体验上限。本章不讲理论，只讲实战中能立刻见效的优化技巧。

性能分析：先测量，再优化

优化驱动的第一原则是“不要猜”。你需要数据来告诉你瓶颈在哪里。内核提供了丰富的性能剖析工具，我习惯先用perf和ftrace定位热点。

性能问题现象
→
perf top/stat
→
定位热点函数
↓
ftrace function_graph
⇄
分析调用链与耗时
↓
针对性优化
→
验证性能提升

比如，使用perf record -g -p <pid> -o perf.data采集驱动进程的性能数据，然后用perf report查看。如果发现kmalloc或copy_to_user占用大量时间，那优化方向就很明确了。

关键点：perf 常用命令速查

  • perf top: 实时查看系统函数热点，像任务管理器。
  • perf stat <command>: 统计命令执行期间的CPU周期、缓存命中率等硬件事件。
  • perf record/report: 记录并生成详细报告，支持符号解析。
  • echo function_graph > /sys/kernel/debug/tracing/current_tracer: 启用ftrace函数图跟踪，能看清函数调用关系和耗时。

内存访问优化：让数据跑得更快

现代CPU的速度远快于内存。一次缓存未命中（Cache Miss）带来的延迟可能相当于执行几十条指令。驱动中不当的内存访问是性能的隐形杀手。

优化策略具体做法预期收益适用场景
结构体对齐与紧凑使用__packed或手动排列成员，减少填充字节减少内存占用，提高缓存行利用率大量实例化的结构体（如sk_buff、设备私有数据）
预取数据使用prefetch()或prefetchw()提示CPU隐藏内存访问延迟遍历链表或大数据块前的指针解引用
使用SLAB缓存为频繁分配的对象创建专用kmem_cache分配/释放速度提升2-5倍驱动中高频创建销毁的固定大小对象
批量操作使用copy_to_user的向量版本copy_to_user_iter减少用户/内核态切换开销大量数据在用户空间和内核空间之间拷贝

我在一个视频采集驱动项目中踩过坑：驱动中每个视频帧都用kmalloc分配一个frame_buffer结构体，在1080p@60fps下性能急剧下降。后来改为使用kmem_cache_create创建专用缓存，分配耗时从平均1500纳秒降到了300纳秒左右。

// 优化前：频繁的通用分配
struct frame_buf *buf = kmalloc(sizeof(struct frame_buf), GFP_KERNEL);

// 优化后：使用SLAB缓存
static struct kmem_cache *frame_cache;
// 模块初始化时创建缓存
frame_cache = kmem_cache_create("frame_buf",
                               sizeof(struct frame_buf),
                               0,
                               SLAB_HWCACHE_ALIGN,
                               NULL);
// 分配时从缓存获取
struct frame_buf *buf = kmem_cache_alloc(frame_cache, GFP_KERNEL);

常见误区

  • 过度优化冷路径：花费大量精力优化一个每秒只执行几次的函数，不如优化执行数百万次的热点函数。先用工具找到瓶颈。
  • 忽视缓存行伪共享：两个频繁写的变量位于同一个缓存行（通常64字节），分属不同CPU核心，会导致缓存行在核心间反复无效化，性能暴跌。用__cacheline_aligned_in_smp隔离它们。
  • 在中断上下文中进行耗时操作：如在中断处理函数里调用可能阻塞或搜索大型链表。这会导致中断被关闭过久，系统响应延迟。应该将工作推送到工作队列或线程化中断。

中断与延迟优化

中断是响应硬件事件的基石，但处理不当会成为系统延迟的罪魁祸首。优化的核心思想是：中断处理只做最紧急的事，耗时工作后置。

高吞吐场景

  • 策略：NAPI（New API）
  • 做法：中断关闭接收，改为轮询模式批量处理网络包。
  • 效果：减少中断次数，提升吞吐量，适用于网络、块设备驱动。

低延迟场景

  • 策略：线程化中断
  • 做法：request_threaded_irq，硬中断快速应答，处理函数在独立线程运行。
  • 效果：中断处理可被抢占，不阻塞其他中断，系统响应更及时。

混合负载场景

  • 策略：工作队列 + 延迟工作
  • 做法：中断中调度一个work_struct或delayed_work。
  • 效果：将非紧急任务延后执行，平衡吞吐与延迟。

对于GPIO按键这类简单中断，使用request_irq即可。但对于一个USB主机控制器（如dwc3），它可能每微秒就产生一次中断，必须使用NAPI或类似机制来合并中断。我曾经调试一个SDIO Wi-Fi驱动，发现系统在高负载下卡顿，用cat /proc/interrupts发现该中断每秒触发数万次。将其改为线程化中断后，UI响应立刻变得流畅。

编者提示：关于工作队列、线程化中断的具体实现细节和API用法，你可以回顾第28章 内核工作队列机制和第5章 中断处理程序设计，那里有更基础的代码示例。

锁与并发优化

锁是保证正确性的工具，但也是性能的敌人。驱动开发中，锁的优化往往能带来最显著的性能提升。

分析临界区大小与频率
→
选择最合适的锁机制
→
尝试缩小锁粒度或无锁设计

首先，避免在驱动中轻易使用mutex_lock这种睡眠锁。对于非常短、且不会睡眠的临界区，优先考虑自旋锁spin_lock。但自旋锁会忙等待，如果临界区执行时间可能较长（比如包含copy_from_user），或者可能睡眠，就必须用互斥锁。

更高级的优化是缩小锁的粒度。不要用一个锁保护整个设备的所有数据。可以为不同的数据块（如接收队列、发送队列、统计信息）使用不同的锁。读写锁rwlock_t或seqlock_t在读多写少的场景下也很有用。

// 优化前：粗粒度锁，所有操作串行化
static DEFINE_MUTEX(dev_lock);
static struct device_data global_data;

// 优化后：细粒度锁，提升并发度
static DEFINE_SPINLOCK(tx_lock);
static DEFINE_SPINLOCK(rx_lock);
static DEFINE_RWLOCK(stats_lock); // 统计信息读多写少
static struct tx_queue tx_data;
static struct rx_queue rx_data;
static struct device_stats stats;

生活类比：锁与超市收银

类比：整个超市只有一个收银台（粗粒度锁），所有顾客排一队，效率低下。开设多个收银台（细粒度锁），或为只买一件商品的顾客开快速通道（读写锁），整体吞吐量就上去了。
技术定义：锁粒度指锁保护的数据范围大小。细粒度锁通过减少不必要的串行化来提升多核并发性能，但增加了编程复杂度和死锁风险。

编译器优化与内联

内核本身在编译时使用了-O2优化级别。对于驱动中的关键路径小函数，可以手动使用static inline提示编译器内联，消除函数调用开销。但要注意，滥用内联会导致代码膨胀，反而降低指令缓存命中率。

一个实用的技巧是使用likely()和unlikely()宏来优化分支预测。CPU的流水线讨厌分支预测失败。如果你知道某个条件（比如错误检查）很少成立，就用unlikely()包裹它，帮助编译器生成更优的指令顺序。

int process_packet(struct sk_buff *skb) {
    // 假设绝大多数数据包都是有效的，错误包极少
    if (unlikely(skb->len < MIN_PACKET_LEN)) {
        kfree_skb(skb);
        return -EINVAL;
    }
    // 正常处理路径...
    return 0;
}

动手试一试

1. 剖析你的驱动：找一个你之前编写的驱动模块，使用perf record -g -p $(pidof your_module_process) sleep 5（或对内核函数使用perf record -e cycles:u -g -- sleep 5）采集性能数据。用perf report查看，找到最耗时的前三个函数。
2. 优化一个内存访问：检查驱动中频繁分配/释放的、大小固定的数据结构。尝试将其改为使用kmem_cache。使用ktime_get_ns()在分配前后计时，对比优化前后的耗时。
3. 调整锁策略：如果你的驱动有并发访问，分析现有的锁。是否存在一个锁保护了过多不相关的数据？能否将其拆分为多个更细粒度的锁？用压力测试工具（如stress-ng）模拟并发，观察系统负载和吞吐量的变化。

检验你的理解

  1. 使用perf top发现驱动中memcpy函数占用大量CPU时间，这可能暗示什么问题？你应该从哪几个方向着手优化？
  2. 为什么在中断处理函数中调用kmalloc(GFP_KERNEL)可能导致系统死锁或高延迟？正确的做法是什么？
  3. 假设你有一个驱动，需要维护一个读操作远多于写操作的配置参数表。从性能角度考虑，你会选择哪种锁机制（mutex, spinlock, rwlock, seqlock）？为什么？

本章小结

  • 性能优化始于测量：必须使用perf、ftrace等工具准确定位性能热点，避免盲目优化。
  • 内存访问是性能关键：通过结构体紧凑排列、使用SLAB专用缓存、预取数据和批量操作来减少缓存未命中与分配开销。
  • 中断处理要短平快：根据场景选择NAPI、线程化中断或工作队列，将耗时任务剥离出中断上下文，保证系统响应性。
  • 精细控制锁的粒度：用最合适的锁（自旋锁、互斥锁、读写锁），保护最小的必要数据范围，提升多核并发能力。
  • 善用编译提示：对关键小函数使用inline，用likely/unlikely优化分支预测，让生成的机器码更高效。
