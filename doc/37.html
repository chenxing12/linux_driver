核心概念漏洞缓解防御性编程

凌晨三点，你的手机响了。运维同事告诉你，部署在边缘网关的定制网卡驱动崩溃了，内核日志里留下一串“general protection fault”和一堆乱码。你远程登录上去，发现是用户空间程序传了一个超长的字符串到驱动的ioctl命令里，直接冲垮了栈上的缓冲区。这不是性能问题，而是安全问题。驱动运行在内核态，一个微小的编码疏忽就可能导致整个系统崩溃或被提权。

⚠️ 常见误区

  • “我的驱动只给内部用，不需要安全检查。”：内部程序也可能有bug，恶意数据可能通过网络或存储介质间接传入。
  • “用户传错参数，崩溃是他的问题。”：内核驱动崩溃会影响整个系统上所有进程，责任在驱动开发者。
  • “用copy_from_user复制过来就安全了。”：这只是完成了数据拷贝，对数据内容的有效性和边界没有任何检查。

内核驱动的攻击面与漏洞分类

驱动是用户空间与硬件之间的桥梁，也是内核暴露给用户的主要接口之一。攻击者会尝试通过这个接口向内核注入恶意数据或触发异常逻辑。一次不安全的write操作，其破坏力可能远超一次应用层的崩溃。

攻击面典型漏洞潜在后果内核版本影响
ioctl命令分发命令号未验证、参数结构体未检查任意代码执行、信息泄露全版本
read/write/mmap缓冲区溢出（栈/堆）、整数溢出内核崩溃、权限提升全版本，缓解机制随版本增强
中断处理例程未屏蔽中断、重入导致状态不一致系统死锁、数据损坏全版本
Proc/Sysfs接口格式化字符串漏洞、竞态条件内核内存泄露、系统不稳定全版本
自定义Netlink套接字消息长度/类型未校验、权限检查缺失远程或本地提权全版本

我在一个传感器驱动项目中踩过坑：为方便调试，通过sysfs暴露了一个可以动态修改硬件采样率的节点。因为没有对写入值进行范围校验，用户写入一个超大的值导致硬件寄存器配置错误，直接让整个I2C总线锁死，需要重启才能恢复。

输入验证：第一道也是最重要的防线

把用户数据想象成来自外部的快递包裹。你不能假设里面一定是安全的礼物，必须像安检一样，检查包裹的尺寸、形状和内容，确认安全后才能拆开使用。对于驱动，这个“安检”就是输入验证。

🔍 核心原则：绝不信任用户空间数据

所有源自用户空间的数据（通过copy_from_user, get_user, ioctl参数等）在用于任何关键操作（如指针解引用、数组索引、大小计算）之前，都必须经过严格验证。

验证必须覆盖以下几个方面，我推荐你按照这个流程来思考：

接收用户数据
→
验证命令/操作合法性
→
验证参数结构体大小
→
验证具体参数值范围
→
安全拷贝到内核缓冲区
→
执行业务逻辑

让我们看一个反面例子和修正后的代码。这是一个处理设置参数的ioctl实现片段：

// 不安全的版本：完全信任用户数据
static long unsafe_ioctl(struct file *filp, unsigned int cmd, unsigned long arg) {
    struct user_config config;
    if (copy_from_user(&config, (void __user *)arg, sizeof(config)))
        return -EFAULT;

    // 直接使用未经验证的数据！
    my_device->sample_rate = config.rate; // 可能导致整数溢出或非法值
    my_device->buffer_size = config.buf_size; // 可能分配巨大内存
    my_device->channel = config.channel; // 可能数组越界
    return 0;
}
// 安全的版本：逐项验证
static long safe_ioctl(struct file *filp, unsigned int cmd, unsigned long arg) {
    struct user_config config;
    // 1. 验证命令本身 (假设 MY_IOCTL_SET_CONFIG 是合法命令)
    if (cmd != MY_IOCTL_SET_CONFIG)
        return -ENOTTY;

    // 2. 安全拷贝数据
    if (copy_from_user(&config, (void __user *)arg, sizeof(config)))
        return -EFAULT;

    // 3. 验证具体参数值
    // 采样率必须在 1Hz 到 100000Hz 之间
    if (config.rate < 1 || config.rate > 100000)
        return -EINVAL;
    // 缓冲区大小不能超过 1MB，且必须是页面大小的倍数
    if (config.buf_size == 0 || config.buf_size > (1024*1024) || (config.buf_size % PAGE_SIZE) != 0)
        return -EINVAL;
    // 通道号不能超过硬件支持的最大通道数
    if (config.channel >= MAX_HW_CHANNELS)
        return -EINVAL;

    // 4. 所有验证通过后，才应用到设备状态
    my_device->sample_rate = config.rate;
    my_device->buffer_size = config.buf_size;
    my_device->channel = config.channel;
    return 0;
}

编者提示：关于ioctl命令号的定义与规范方法，以及copy_from_user等函数的详细用法和错误处理，可以参考第1章 字符设备驱动框架中的相关章节。

安全的内存管理：杜绝溢出与泄漏

内核内存错误是驱动安全的重灾区。除了验证输入，在内存操作本身也需要遵循安全模式。

栈缓冲区安全

  • 禁用可变长数组（VLA）：如char buf[size];，size若用户可控极危险。
  • 使用固定大小数组并检查拷贝长度：char buf[MAX_LEN]; if (len > MAX_LEN) return -EINVAL;
  • 优先使用内核安全API：如strscpy, strscpy_pad替代strcpy, strncpy。

堆内存安全

  • 检查kmalloc分配结果：永远检查返回指针是否为NULL。
  • 大小计算防整数溢出：if (count > SIZE_MAX / elem_size) return -ENOMEM;
  • 配对释放：在错误路径上也必须释放已分配的资源。

指针与引用安全

  • 用户指针始终用__user标记，提醒自己不能直接解引用。
  • 使用access_ok()验证用户地址（尽管copy_*_user内部会做，显式调用更清晰）。
  • 释放后置空：kfree(ptr); ptr = NULL;防止重复释放（Use-After-Free）。

一个经典的整数溢出导致堆缓冲区溢出的陷阱：

// 危险：`count * elem_size` 可能溢出，导致分配过小缓冲区
void *unsafe_alloc(size_t count, size_t elem_size) {
    return kmalloc(count * elem_size, GFP_KERNEL); // 可能溢出！
}

// 安全：在乘法前检查是否溢出
void *safe_alloc(size_t count, size_t elem_size) {
    size_t total_size;
    if (check_mul_overflow(count, elem_size, &total_size)) // 内核提供的检查函数
        return NULL;
    return kmalloc(total_size, GFP_KERNEL);
}

实际开发中，大部分团队会强制要求使用类似safe_alloc的包装函数，并将check_mul_overflow这类检查作为代码审查的必查项。内核从5.6版本开始，在include/linux/overflow.h中提供了check_add_overflow, check_mul_overflow等辅助宏，请务必使用它们。

利用内核内置的安全机制

现代Linux内核（特别是5.x以后）集成了多种缓解安全漏洞的机制。作为驱动开发者，你需要确保你的驱动能与之协同工作，而不是无意中绕过它们。

首先，在你的驱动Makefile中启用编译器的安全选项。这通常由内核顶层Makefile控制，但你需要了解其作用：

  • -fstack-protector-strong：在函数栈帧中插入“金丝雀”值，检测栈缓冲区溢出。这是现代内核的标准配置，你的驱动自动受益。
  • -D_FORTIFY_SOURCE=2：在编译时对字符串和内存操作函数（如memcpy, strcpy）进行缓冲区大小检查。需要你正确使用sizeof(dst)等。

其次，在开发阶段积极使用内核的运行时检测工具：

  • KASAN (KernelAddressSANitizer)：类似于用户空间的ASan，能检测use-after-free、out-of-bounds访问等内存错误。开发驱动时，在开启KASAN的内核上测试是必须的步骤。它会明显拖慢性能，仅用于测试。
  • LOCKDEP：死锁检测器。如果你在驱动中使用了自旋锁、互斥锁，LOCKDEP可以帮助你发现不正确的锁获取顺序，避免潜在的死锁漏洞。

如果你看到类似“KASAN: slab-out-of-bounds write in my_driver_write”这样的报错，大概率是你的写操作越界了1个字节，需要仔细检查循环边界或拷贝长度。

🔐 权限与能力检查

不是所有用户都有权操作你的设备。在驱动的open函数或关键的ioctl命令开始时，应检查当前进程的权限。例如，一个配置硬件时钟的ioctl可能只允许root用户执行：

if (!capable(CAP_SYS_TIME)) {
    pr_warn("Process %d attempted privileged clock operation without CAP_SYS_TIME\n", current->pid);
    return -EPERM;
}

使用capable()检查特定的能力（Capability），比简单检查current_uid() == 0更精细，符合最小权限原则。

动手试一试

1. 代码审计练习：找一个你之前写过的简单字符设备驱动（或从开源项目找一个），重点审查其ioctl函数和write函数。列出所有来自用户空间的数据输入点，并检查每一项是否都进行了：

  • 命令号/操作合法性验证。
  • 参数结构体大小验证（防止信息泄露）。
  • 具体数值的范围/边界检查。
  • 缓冲区拷贝的长度限制。

2. 安全编译测试：将你的驱动模块放在一个开启了KASAN和LOCKDEP的内核源码树中重新编译并加载（你需要配置内核并开启这些调试选项）。尝试通过用户空间程序传入一些边界或异常值（如超长字符串、负数、超大的size值），观察内核日志是否有KASAN报错或警告信息。

检验你的理解

  1. 为什么在内核驱动中直接使用用户空间传入的整数作为数组索引是危险的？除了越界，还可能引发什么问题？
  2. 假设一个ioctl命令需要传递一个包含指针成员的结构体，这个指针指向用户空间的一个缓冲区。请描述从接收该结构体到安全访问缓冲区数据的完整验证步骤。
  3. copy_from_user执行成功返回0，是否意味着从用户空间拷贝过来的数据就是安全可用的？为什么？

本章小结

  • 内核驱动安全的核心是“零信任”：所有源自用户空间的数据都必须经过严格、多层次的验证（命令、大小、范围）后才能使用。
  • 内存操作是漏洞高发区：必须防范栈/堆缓冲区溢出、整数溢出和Use-After-Free。使用内核安全API（如strscpy）和溢出检查宏（如check_mul_overflow）。
  • 善用内核安全基础设施：在开发阶段利用KASAN、LOCKDEP等工具主动发现漏洞；确保驱动代码兼容栈保护等编译时加固机制。
  • 实施最小权限原则：使用capable()在关键操作前检查进程能力，避免未授权访问。
  • 安全是贯穿始终的过程：从设计、编码、测试到代码审查，每个环节都需要有安全意识的参与，才能构建出健壮的驱动。
