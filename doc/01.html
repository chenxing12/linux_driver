核心概念file_operations设备号cdev

深夜，你接到线上告警：一台负责数据采集的嵌入式设备，其自定义的传感器模块突然无法读取数据。你ssh登录上去，尝试用测试程序直接操作设备文件 /dev/my_sensor，执行 read 操作，返回的不是预期的采样值，而是一个冷冰冰的 “Invalid argument”。更棘手的是，dmesg 里刷出了一行内核错误：“my_sensor: no such device”。问题出在哪里？是硬件坏了，还是驱动加载异常？你意识到，必须深入那个连接用户程序与硬件的桥梁——字符设备驱动，才能找到答案。

本章，我们就从零开始，拆解这个“桥梁”的建造蓝图。你不用关心具体的传感器芯片（比如 MPU6050 或 BME280），我们先聚焦于驱动本身最基础的框架：字符设备驱动。它是 Linux 驱动世界的“Hello World”，也是你理解后续一切复杂子系统（I2C, SPI, Input等）的基石。

1.1 核心问题：用户层的 `read()` 如何抵达硬件？

用户空间
↓ (syscall)
VFS 虚拟文件系统
↓ (file_operations)
字符设备驱动
↓ (硬件操作)
物理硬件
↗
设备文件 `/dev/xxx`
⇄
`cdev` 结构体

上图清晰地展示了当你调用 read(fd, buf, size) 时发生的故事。数据流穿过层层抽象，最终由你编写的驱动函数处理。这里涉及三个关键数据结构，它们的职责对比如下：

数据结构所在层核心职责生命周期
file_operations驱动接口层函数指针集合，定义 `open`, `read`, `write`, `ioctl` 等操作的具体实现。驱动编写时静态定义，全局唯一。
cdev内核设备管理层将 `设备号` 与 `file_operations` 绑定，向内核注册一个字符设备。驱动加载时动态分配并注册，卸载时销毁。
设备文件 (/dev/xxx)用户接口层用户空间的句柄，通过 `主设备号` 关联到对应的 `cdev`。由 `mknod` 或 `udev` 创建，可持久化存在。

很多教程不会告诉你：`/dev` 下的设备文件只是一个“门户”（inode），它本身不包含驱动代码。真正的驱动实体是内核中的 `cdev` 对象。这就是为什么有时设备文件存在，但操作却失败（提示“No such device”）——因为对应的 `cdev` 可能并未成功注册或已注销。我在排查一个 Linux 4.19.y 内核的驱动 bug 时就踩过这个坑，驱动模块卸载后，/dev 下的文件忘了删除，下次加载时因为设备号冲突导致 `cdev_add` 失败，但表象就是用户程序报错“Invalid argument”。

1.2 驱动骨架：一个最小的可运行示例

理论说再多，不如一行代码。下面是一个最精简的字符设备驱动骨架，它创建一个名为 `“vchar”` 的虚拟设备，实现了 `open`、`release` 和 `read` 操作。请在你的实验环境（建议使用 Ubuntu 20.04 或更新版本，内核头文件齐全）中创建文件 `vchar_driver.c`：

// vchar_driver.c - 虚拟字符设备驱动骨架
#include <linux/module.h>
#include <linux/fs.h>
#include <linux/cdev.h>
#include <linux/uaccess.h> // 用于 copy_to_user

#define DEVICE_NAME "vchar"
static int major_num = 0; // 动态分配主设备号
static struct cdev vchar_cdev;

// 1. 定义 file_operations
static int vchar_open(struct inode *inode, struct file *filp) {
    printk(KERN_INFO "vchar device opened.\n");
    return 0;
}

static ssize_t vchar_read(struct file *filp, char __user *buf, size_t count, loff_t *f_pos) {
    char msg[] = "Hello from kernel!\n";
    size_t len = strlen(msg);
    if (*f_pos >= len)
        return 0;
    if (count > len - *f_pos)
        count = len - *f_pos;
    if (copy_to_user(buf, msg + *f_pos, count))
        return -EFAULT;
    *f_pos += count;
    return count;
}

static int vchar_release(struct inode *inode, struct file *filp) {
    printk(KERN_INFO "vchar device closed.\n");
    return 0;
}

static struct file_operations vchar_fops = {
    .owner = THIS_MODULE,
    .open = vchar_open,
    .read = vchar_read,
    .release = vchar_release,
};

// 2. 模块初始化与退出
static int __init vchar_init(void) {
    dev_t dev_num;
    int ret;

    // 动态申请一个主设备号
    ret = alloc_chrdev_region(&dev_num, 0, 1, DEVICE_NAME);
    if (ret < 0) {
        printk(KERN_ERR "Failed to allocate chrdev region.\n");
        return ret;
    }
    major_num = MAJOR(dev_num);
    printk(KERN_INFO "vchar: major number is %d\n", major_num);

    // 初始化并添加 cdev
    cdev_init(&vchar_cdev, &vchar_fops);
    vchar_cdev.owner = THIS_MODULE;
    ret = cdev_add(&vchar_cdev, dev_num, 1);
    if (ret) {
        printk(KERN_ERR "Failed to add cdev.\n");
        unregister_chrdev_region(dev_num, 1);
        return ret;
    }
    printk(KERN_INFO "vchar driver loaded successfully.\n");
    return 0;
}

static void __exit vchar_exit(void) {
    dev_t dev_num = MKDEV(major_num, 0);
    cdev_del(&vchar_cdev);
    unregister_chrdev_region(dev_num, 1);
    printk(KERN_INFO "vchar driver unloaded.\n");
}

module_init(vchar_init);
module_exit(vchar_exit);
MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("A minimal virtual character device driver");

⚠️ 常见误区：内核空间与用户空间的内存隔离

注意 `vchar_read` 函数中的 copy_to_user。这是必须的。内核驱动直接访问的用户空间指针 `buf` 是虚拟地址，在内核上下文无法直接解引用。必须使用内核提供的拷贝函数。我曾见过新手直接使用 `memcpy`，导致内核立刻 oops（内核崩溃），错误信息类似 “Unable to handle kernel paging request at virtual address xxxxxxxx”。记住：永远不要直接解引用来自用户空间的指针。

1.3 从编译到测试：完整的驱动生命周期

有了代码，下一步是让它跑起来。这个过程可以概括为以下流程：

→

→
`make` 编译生成 .ko 文件
→
`insmod` 加载驱动
→
`mknod` 创建设备文件
→
用户程序测试
→
`rmmod` 卸载驱动

你需要一个对应的 `Makefile`。注意，内核模块的编译需要指向你当前运行内核的构建目录（通常是 `/lib/modules/$(uname -r)/build`）。

# Makefile
ifneq ($(KERNELRELEASE),)
    obj-m := vchar_driver.o
else
    KERNELDIR ?= /lib/modules/$(shell uname -r)/build
    PWD := $(shell pwd)
default:
    $(MAKE) -C $(KERNELDIR) M=$(PWD) modules
clean:
    $(MAKE) -C $(KERNELDIR) M=$(PWD) clean
endif

在终端中执行 `make`，成功后会产生 `vchar_driver.ko` 文件。现在，让我们按步骤操作：

步骤命令预期输出/目的关键观察点
1. 加载模块sudo insmod vchar_driver.ko驱动初始化函数 `vchar_init` 被调用。运行 dmesg | tail -5，看到 “vchar: major number is XXX” 和加载成功的消息。记下这个主设备号，假设是 511。
2. 创建设备文件sudo mknod /dev/vchar c 511 0在 `/dev` 下创建节点，`c` 表示字符设备，`511` 是主设备号，`0` 是次设备号。检查 ls -l /dev/vchar，权限应为 `crw-r--r--`。
3. 编写测试程序创建 `test_vchar.c`，调用 `open`, `read`, `close`。从驱动读取 “Hello from kernel!” 字符串。确保编译时加 `-o test_vchar`。
4. 运行测试sudo ./test_vchar程序打印出内核消息。同时用 `dmesg | tail` 观察驱动打印的 open/close 日志。
5. 清理sudo rmmod vchar_driver
sudo rm /dev/vchar		驱动退出函数被调用，资源释放。再次 `dmesg`，应看到卸载成功的消息。

🔧 动手实践：你的第一个测试程序

创建一个简单的测试程序 `test_vchar.c`：

// test_vchar.c
#include <stdio.h>
#include <fcntl.h>
#include <unistd.h>
int main() {
    int fd = open("/dev/vchar", O_RDONLY);
    if (fd < 0) { perror("open failed"); return 1; }
    char buf[100];
    int n = read(fd, buf, sizeof(buf)-1);
    if (n > 0) { buf[n] = '\0'; printf("Read from driver: %s", buf); }
    close(fd);
    return 0;
}

编译并运行：gcc -o test_vchar test_vchar.c && sudo ./test_vchar。你应该看到输出。

1.4 深入 `file_operations`：驱动与应用的契约

`file_operations` 结构体是驱动向 VFS 注册的“方法表”，它定义了驱动能支持哪些操作。对于入门驱动，你需要优先掌握以下几个核心成员：

open/release

  • open: 设备打开时调用，常做初始化、资源申请、引用计数+1。
  • release: 设备关闭时调用，做清理工作。注意不是所有 close 都会触发 release。

read/write

  • 数据读写核心。参数 `__user` 指针需用 `copy_to/from_user`。
  • `loff_t *f_pos` 指向文件偏移指针，驱动需要更新它。

unlocked_ioctl

  • 用于实现设备特定的命令（如设置参数、读取状态）。
  • 在 Linux 2.6.36 后，优先使用它而非旧的 `ioctl`。

llseek

  • 修改文件读写位置。对于顺序访问的设备（如键盘），可设为 `no_llseek`。
  • 需要处理 `SEEK_SET`, `SEEK_CUR`, `SEEK_END`。

一个具体的版本差异坑：在 Linux 5.10+ 内核中，`file_operations` 里的 `read` 和 `write` 函数原型发生了变化，移除了 `struct file *` 参数后的 `char __user *` 修饰符。如果你把为旧内核（如 4.x）编写的驱动直接拿到新内核上编译，会得到类型不匹配的编译错误。解决方案是使用内核头文件提供的 `#if LINUX_VERSION_CODE` 宏来做条件编译，或者直接按照新内核的原型编写。

📘 扩展阅读：本章我们手动使用 `mknod` 创建设备文件。在实际项目中，更常见的做法是配合 udev 规则自动创建设备节点，这涉及到设备模型和 `class_create`、`device_create` 等函数。我们将在第26章《设备模型与kobject》中详细探讨如何让驱动更“现代化”地集成到 sysfs 中，实现设备节点的自动管理。

1.5 静态 vs 动态：设备号分配策略选择

在 `vchar_init` 中，我们使用了 `alloc_chrdev_region` 来动态申请一个未使用的主设备号。这是推荐的做法。但在某些嵌入式产品中，为了固定设备节点号，可能会采用静态分配。两者的对比如下：

分配方式API 函数优点缺点适用场景
动态分配alloc_chrdev_region避免设备号冲突，灵活。驱动可移植性强。主设备号不固定，每次加载可能变化，需要从日志或 `/proc/devices` 读取。绝大多数场景，特别是通用驱动、学习实验。
静态分配register_chrdev_region设备号固定，便于预先编写脚本或配置。可能与其他驱动冲突，需要查阅 `Documentation/devices.txt` 选择未使用的号。产品固化，有明确的设备号规划；或需要兼容老版本脚本。

实际开发中，大部分团队会选择动态分配，并结合 udev 规则或驱动自身通过 `device_create` 自动创建设备节点，完全避免手动执行 `mknod`。如果你必须使用静态号，我建议选择一个较大的主设备号（比如 240-254 范围），这个区域通常预留给本地或实验用途，冲突概率小。

🧪 实战练习：为虚拟设备增加 `write` 功能

目标：修改 `vchar_driver.c`，增加 `file_operations` 中的 `.write` 函数指针，实现一个简单的回显功能——将用户写入的数据存储在内核的一个静态缓冲区中，并在下次 `read` 时返回。

要求：

  1. 在驱动中定义一个静态字符数组 `static char kernel_buffer[1024];` 和长度变量。
  2. 实现 `vchar_write` 函数，使用 `copy_from_user` 将用户数据拷贝到 `kernel_buffer`。
  3. 修改 `vchar_read` 函数，改为从 `kernel_buffer` 中读取数据。
  4. 更新 `vchar_fops`，加入 `.write = vchar_write`。
  5. 重新编译、加载、测试。使用 `echo “test data” | sudo tee /dev/vchar` 写入，再用 `cat /dev/vchar` 读取验证。

提示：注意 `write` 函数的返回值是成功写入的字节数。处理好缓冲区边界，防止溢出。

✅ 自我检查清单

  • 你能在不看代码的情况下，说出用户程序调用 `read()` 到驱动函数被调用的完整路径吗？
  • `cdev_init` 和 `cdev_add` 分别做了什么？它们的调用失败会导致什么现象？
  • 为什么驱动中不能直接用 `memcpy` 拷贝用户空间的数据？正确的函数是什么？
  • 动态分配设备号后，如何确定系统实际分配的主设备号是多少？
  • 驱动模块卸载函数 `vchar_exit` 中，释放资源的顺序应该是怎样的？（提示：先 `cdev_del` 还是先 `unregister_chrdev_region`？）

📦 本章总结

本章我们从一个真实的驱动故障场景切入，构建了一个完整的字符设备驱动认知框架。你学到了：

  1. 核心链路：用户空间系统调用 → VFS → `file_operations` → 驱动函数 → 硬件。
  2. 三大支柱：`file_operations`（操作集）、`cdev`（设备对象）、设备文件（用户接口）。
  3. 关键API：`alloc_chrdev_region`、`cdev_init`、`cdev_add`、`copy_to/from_user`。
  4. 完整流程：编写驱动 → 编译为 `.ko` → `insmod` 加载 → `mknod` 创建设备文件 → 用户程序测试 → `rmmod` 卸载。
  5. 避坑指南：用户空间指针必须通过拷贝函数访问；注意内核版本差异导致的 API 变化；动态分配设备号更通用。
