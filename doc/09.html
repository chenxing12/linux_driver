核心概念I2C适配器I2C客户端

开发板上的温度传感器读数不准，或者EEPROM芯片无法识别？这类问题十有八九出在I2C驱动上。I2C（Inter-Integrated Circuit）是嵌入式领域最常见的低速串行总线之一，用于连接微控制器和众多外围芯片。Linux内核为此提供了一套成熟的分层驱动模型。

I2C总线：设备间的“电话会议”

想象一个电话会议：主持人（主设备）拨通一个号码（从设备地址），然后开始对话（数据传输），其他与会者（其他从设备）保持静默。I2C总线就是如此，它由SCL（时钟线）和SDA（数据线）两根线组成，支持多主多从，通过7位或10位地址寻址设备。

9.1 Linux I2C驱动框架全景

Linux内核将I2C驱动清晰地分为三部分：核心层（I2C Core）、适配器驱动（Adapter Driver，或称控制器驱动）和客户端驱动（Client Driver）。你作为设备驱动开发者，主要打交道的是客户端驱动。

适配器驱动由芯片厂商或社区提供，它负责操作具体的I2C控制器硬件，比如NXP i.MX6ULL芯片上的I2C1控制器，对应的驱动可能是drivers/i2c/busses/i2c-imx.c。它会在系统初始化时注册一个struct i2c_adapter。

客户端驱动则是你为具体的I2C从设备（如一个AT24C02 EEPROM芯片）编写的驱动。它通过I2C核心层提供的API，向适配器发起读写请求。

关键数据结构：i2c_client 与 i2c_driver

这是客户端驱动的两个核心。内核为每个探测到的I2C设备创建一个i2c_client，它包含了设备地址（addr）、所属适配器（adapter）等信息。你的驱动则提供一个i2c_driver结构体，其中定义了probe、remove等回调函数以及设备匹配表。当i2c_client与i2c_driver匹配成功，probe函数就会被调用。

我们从零开始为一个假设的“XYZ123”温度传感器编写驱动。该传感器I2C地址为0x48，有一个8位的温度寄存器（地址0x00）。

首先，定义驱动并实现匹配表。匹配可以通过设备树（Device Tree）兼容字符串，或者传统的i2c_device_id表进行。我推荐使用设备树，这是现代嵌入式Linux的标准。

/* 驱动匹配表 */
static const struct of_device_id xyz123_of_match[] = {
    { .compatible = "xyz,xyz123-temp", },
    {},
};
MODULE_DEVICE_TABLE(of, xyz123_of_match);

static const struct i2c_device_id xyz123_id_table[] = {
    { "xyz123-temp", 0 },
    {},
};
MODULE_DEVICE_TABLE(i2c, xyz123_id_table);

/* 定义 i2c_driver */
static struct i2c_driver xyz123_driver = {
    .driver = {
        .name = "xyz123_temp",
        .of_match_table = of_match_ptr(xyz123_of_match),
    },
    .probe = xyz123_probe,
    .remove = xyz123_remove,
    .id_table = xyz123_id_table,
};

接下来是probe函数。这里你需要初始化设备，并创建向用户空间暴露功能的接口。一个简单的做法是注册为字符设备。

static int xyz123_probe(struct i2c_client *client, const struct i2c_device_id *id)
{
    struct device *dev = &client->dev;
    int ret;
    u8 reg_val;

    // 1. 检查设备通信是否正常：尝试读取一个已知寄存器
    ret = i2c_smbus_read_byte_data(client, 0x00);
    if (ret < 0) {
        dev_err(dev, "Failed to read device ID, check address 0x%02x\n", client->addr);
        return ret;
    }
    dev_info(dev, "XYZ123 detected at 0x%02x, chip id: 0x%02x\n", client->addr, ret);

    // 2. 初始化硬件（例如，设置工作模式）
    ret = i2c_smbus_write_byte_data(client, 0x01, 0x80); // 假设0x01是配置寄存器
    if (ret) {
        dev_err(dev, "Failed to init config register\n");
        return ret;
    }

    // 3. 分配并初始化你的私有设备结构体（通常包含 i2c_client、cdev 等）
    // 4. 注册字符设备、创建 sysfs 节点或 IIO 设备（参见第18章）
    // ...

    return 0;
}

常见误区

  • 地址混淆：I2C设备地址通常是7位（0x00-0x7F）。内核API（如i2c_smbus_*）要求传入的就是这个7位地址。如果你从芯片手册看到“写地址：0x90，读地址：0x91”，这通常是将7位地址左移一位后加上了R/W位，你需要右移一位得到0x48作为驱动中使用的地址。
  • 未检查返回值：i2c_smbus_*系列函数失败时返回负数。实际开发中，大部分团队会忽略第一次通信失败的检查，导致probe成功了但设备实际无法工作。
  • 并发访问：I2C适配器本身由内核同步，但你的驱动如果通过字符设备提供read/write接口，必须考虑多个用户进程同时调用时的并发控制，需要使用第3章学的互斥锁或信号量。

9.3 I2C数据读写API详解

内核提供了两套主要的I2C读写API：SMBus兼容函数和更通用的i2c_transfer。对于大多数寄存器型设备，SMBus函数就足够了，它们处理了起止信号、ACK等底层细节。

常用 I2C/SMBus API 对比与选用
函数描述适用场景示例（地址0x48，寄存器0x00）
i2c_smbus_read_byte_data(client, reg)读取一个8位寄存器读取配置、状态寄存器读温度值
i2c_smbus_write_byte_data(client, reg, val)向一个8位寄存器写入一个字节设置工作模式、使能写配置寄存器
i2c_smbus_read_word_data(client, reg)读取一个16位寄存器（注意字节序）读取16位ADC结果读压力传感器数据
i2c_smbus_read_i2c_block_data(client, reg, len, buf)从寄存器开始读取多个字节读取EEPROM、FIFO数据读取多字节序列号
i2c_transfer(adapter, msg, num)最通用的I2C传输，可组合读/写复杂协议（如SHT3x温湿度传感器）、需要重复START信号非标准时序设备

我在一个使用NXP PCA9534 GPIO扩展芯片的项目中踩过坑。它的输入寄存器读取需要先写寄存器指针，再发起读操作。用i2c_smbus_read_byte_data不行，因为它内部是一个write-then-read的复合操作，但中间会发出STOP信号，而PCA9534要求无STOP的重复START。最终必须用i2c_transfer构造两个i2c_msg来解决。

/* 使用 i2c_transfer 读取 PCA9534 的输入端口寄存器 (0x00) */
static int pca9534_read_inputs(struct i2c_client *client, u8 *val)
{
    struct i2c_adapter *adap = client->adapter;
    struct i2c_msg msg[2];
    u8 reg = 0x00; // 输入寄存器地址
    int ret;

    msg[0].addr = client->addr;
    msg[0].flags = 0; // 写
    msg[0].len = 1;
    msg[0].buf = ®

    msg[1].addr = client->addr;
    msg[1].flags = I2C_M_RD; // 读
    msg[1].len = 1;
    msg[1].buf = val;

    ret = i2c_transfer(adap, msg, 2);
    if (ret != 2) {
        dev_err(&client->dev, "i2c_transfer error: %d\n", ret);
        return (ret < 0) ? ret : -EIO;
    }
    return 0;
}

编者提示：设备树（DTS）是声明I2C设备的标准方式。你需要在第7章《设备树DTS语法解析》知识的基础上，在I2C控制器节点下添加子节点来描述你的设备，例如&i2c1 { temp_sensor: xyz123-temp@48 { compatible = “xyz,xyz123-temp”; reg = <0x48>; };。驱动通过.compatible字符串与之匹配。

9.4 用户空间工具与调试

驱动编写过程中，不要急于集成到内核模块测试。先用用户空间的i2c-tools验证硬件连接和基本通信是否正常，这能帮你快速定位是硬件问题还是驱动问题。

探测设备

  • i2cdetect -y 1 (扫描I2C总线1上的设备)
  • 输出显示48表示地址0x48有设备响应
  • 如果显示UU，表示该地址已被内核驱动占用

读写测试

  • i2cget -y 1 0x48 0x00 (从设备0x48读寄存器0x00)
  • i2cset -y 1 0x48 0x01 0x80 (向设备0x48的寄存器0x01写0x80)
  • 注意：这些工具使用SMBus协议

驱动加载后

  • 检查/sys/bus/i2c/devices/下是否有对应设备目录
  • 查看内核日志dmesg | tail，搜索你的驱动打印信息
  • 如果驱动创建了字符设备，用hexdump或自定义小程序测试

如果你在probe时看到错误信息“i2c i2c-1: sendbytes: error -110, addr 0x48”，这大概率是超时错误（-ETIMEDOUT）。排查顺序：1) 确认设备地址是否正确；2) 用示波器或逻辑分析仪检查SCL/SDA波形，看是否有应答ACK；3) 检查上拉电阻（通常4.7kΩ），总线电容过大会导致上升沿太慢；4) 确认设备供电是否正常。

动手试一试

1. 在你的开发板（如树莓派或i.MX6ULL）上，使用i2cdetect扫描所有I2C总线，记录下已连接的设备地址。
2. 找一个I2C接口的EEPROM（如AT24C02，地址通常为0x50），编写一个简单的内核模块。在probe函数中，向EEPROM的偏移地址0x10处写入字符串“HelloI2C”，然后在remove函数或另一个读函数中将其读回并打印到内核日志。
3. （进阶）修改驱动，通过sysfs创建一个属性文件（例如/sys/bus/i2c/devices/1-0050/eeprom_test），使得用户可以通过echo “test” > eeprom_test写入，通过cat eeprom_test读取验证。

本章小结

  • Linux I2C驱动采用适配器驱动（控制硬件控制器）和客户端驱动（控制具体外设）分离的清晰架构。
  • 编写客户端驱动的核心是定义i2c_driver，实现probe和remove函数，并通过设备树或ID表与i2c_client匹配。
  • 优先使用i2c_smbus_*系列API进行寄存器读写，对于复杂时序（需重复START）则使用更底层的i2c_transfer。
  • 在开发驱动前，务必使用i2c-tools验证硬件连接与基本通信，并学会通过内核日志和错误码（如-110、-6）快速定位问题。
  • I2C设备信息通过设备树（DTS）描述，驱动通过compatible属性进行匹配，这是现代嵌入式Linux的标准做法。
