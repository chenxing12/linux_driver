核心概念struct net_deviceNAPI

如果你尝试用 ifconfig 或 ip 命令查看网卡，看到的 “eth0”、“wlan0” 就是网络设备驱动在内核中的具象化。与字符设备不同，网络设备驱动不直接提供 “/dev/eth0” 这样的设备文件，而是通过一套名为 “socket” 的抽象接口与用户空间通信。你可以把它想象成邮局系统：驱动是分拣中心（处理数据包），struct net_device 是邮局的门牌和操作手册（定义设备能力和方法），而上层协议（TCP/IP）则是邮递路线规则。

核心结构体：struct net_device

这是网络设备驱动的“身份证”和“功能说明书”。它定义了设备的所有属性（MAC地址、MTU、支持的功能）和操作方法（打开、关闭、发送数据包）。驱动开发者的主要工作就是填充这个结构体，并将其注册到内核网络子系统中。

网络设备驱动的核心数据结构

理解网络驱动，首先要看懂几个关键的数据结构。它们之间的关系构成了驱动的基本骨架。

网络协议栈
(TCP/IP)
↓
网络设备接口层
(net_device)
↓
硬件抽象层
(驱动私有数据)
↗↖
数据包队列
(sk_buff)
⇄
硬件收发器
(DMA/中断)

上图中，struct net_device 是承上启下的核心。我们来看一个最简化的驱动初始化代码片段，它创建并注册一个网络设备：

// 1. 分配 net_device 结构体
struct net_device *my_net_dev;
my_net_dev = alloc_etherdev(sizeof(struct my_private_data));
if (!my_net_dev) {
    return -ENOMEM;
}

// 2. 设置设备操作函数集 (这是关键！)
my_net_dev->netdev_ops = &my_netdev_ops;

// 3. 设置 MAC 地址 (例如从 EEPROM 读取)
unsigned char mac_addr[ETH_ALEN] = {0x00, 0x11, 0x22, 0x33, 0x44, 0x55};
ether_addr_copy(my_net_dev->dev_addr, mac_addr);

// 4. 注册设备到内核
int err = register_netdev(my_net_dev);
if (err) {
    printk(KERN_ERR "Failed to register net device\n");
    free_netdev(my_net_dev);
    return err;
}

关键点：netdev_ops 操作集

这是驱动功能的实现集合，类似于字符设备驱动中的 file_operations。你必须实现其中的几个核心函数，否则设备无法工作。下表列出了最关键的几个操作：

函数指针作用是否必须典型实现内容
ndo_open打开设备是申请中断、DMA通道，启动硬件
ndo_stop关闭设备是释放中断、DMA通道，停止硬件
ndo_start_xmit发送数据包是将sk_buff中的数据交给硬件发送队列
ndo_get_stats64获取统计信息推荐返回发送/接收包数、错误计数等
ndo_set_mac_address设置MAC地址如果硬件支持将新MAC地址写入硬件寄存器
ndo_do_ioctl设备专用控制可选处理 SIOCETHTOOL 等命令

我在为一个基于 Marvell 88E1111 PHY芯片的千兆网卡编写驱动时，就曾因为没正确实现 ndo_get_stats64，导致 ifconfig 显示的 RX/TX 数据一直为0，排查了半天才发现是这个统计函数返回了空指针。

数据包的生命周期：从发送到接收

数据在内核中以 struct sk_buff（socket buffer）的形式存在。理解它的流向是掌握网络驱动的关键。

应用层调用 send()
→
协议栈构造 sk_buff
→
调用 ndo_start_xmit()
→
驱动将数据放入硬件队列
→
硬件通过DMA发送

接收流程则更为复杂，因为涉及到中断和轮询的权衡。早期驱动在每次收到数据包时都触发一个硬件中断，这在千兆、万兆高速网络下会导致系统被中断淹没，性能急剧下降。为此，Linux引入了 NAPI（New API） 机制。

核心机制：NAPI（混合中断与轮询）

生活类比：就像快递柜，以前每到一个包裹就给你打电话（中断），现在攒够一批再通知你一次（混合中断），你自己去柜子前把所有包裹一次性取走（轮询）。
技术定义：NAPI是一种混合中断和轮询的数据包接收机制。首次数据包到达触发中断，关闭中断后，驱动进入轮询模式，一次性处理完所有队列中的数据包，再重新打开中断。这大大降低了高负载下的中断频率。

实现NAPI驱动的接收函数通常如下所示：

// 中断处理函数（上半部）
static irqreturn_t my_net_interrupt(int irq, void *dev_id) {
    struct net_device *dev = dev_id;
    struct my_private_data *priv = netdev_priv(dev);

    // 1. 读取中断状态寄存器，确认是接收中断
    if (!(read_reg(REG_ISR) & INT_RX_DONE)) {
        return IRQ_NONE;
    }

    // 2. 关闭接收中断，防止新中断干扰
    disable_rx_interrupt();

    // 3. 将设备的napi结构体加入轮询队列，并触发软中断
    if (napi_schedule_prep(&priv->napi)) {
        __napi_schedule(&priv->napi);
    }
    return IRQ_HANDLED;
}

// NAPI轮询函数（下半部）
static int my_net_poll(struct napi_struct *napi, int budget) {
    struct my_private_data *priv = container_of(napi, struct my_private_data, napi);
    struct net_device *dev = priv->dev;
    int work_done = 0;

    // 循环处理接收队列，直到处理完budget个包或队列为空
    while (work_done < budget) {
        struct sk_buff *skb = receive_packet_from_hardware(priv);
        if (!skb) {
            break; // 没有更多数据包
        }
        skb->dev = dev;
        netif_receive_skb(skb); // 送往上协议栈
        work_done++;
    }

    // 如果处理的数据包少于budget，说明本轮处理完毕，退出NAPI状态，重新开启中断
    if (work_done < budget) {
        napi_complete_done(napi, work_done);
        enable_rx_interrupt();
    }
    return work_done;
}

这里的 budget 是内核每次轮询允许处理的最大数据包数，通常由系统根据负载动态调整。实际开发中，一个常见的坑是忘记在 ndo_open 中初始化NAPI结构体（netif_napi_add）和在 ndo_stop 中清理它（napi_disable），这会导致内核崩溃或数据包丢失。

常见误区

  • sk_buff 生命周期管理不当：在 ndo_start_xmit 中，成功将skb交给硬件后，必须调用 dev_kfree_skb_any() 或其变体来释放；发送失败则返回 NETDEV_TX_BUSY 并不要释放skb，内核会稍后重试。
  • 中断共享处理遗漏：如果网卡IRQ是共享的（在PCI设备中常见），中断处理函数必须检查硬件中断状态寄存器，确认是本设备产生的中断后返回 IRQ_HANDLED，否则返回 IRQ_NONE。
  • DMA缓存一致性忽略：用于DMA传输的内存缓冲区必须使用 dma_alloc_coherent() 分配，或者使用 dma_map_single() 映射普通内存。在ARM等平台上，不这样做会导致数据损坏或系统死机。

虚拟网络设备：调试与学习的利器

在真机上调试网卡驱动风险高、周期长。内核提供的 tun/tap 和 dummy 虚拟网络设备是完美的替代品。它们完全由软件实现，帮你隔离硬件复杂性。

tun/tap 设备

  • 作用：模拟点对点或以太网设备。
  • 数据流向：内核网络栈 ↔ 驱动 ↔ 用户空间程序。
  • 典型应用：VPN（如OpenVPN）、虚拟机网络桥接、驱动协议栈学习。

dummy 设备

  • 作用：一个纯虚拟的、没有任何底层硬件的以太网接口。
  • 数据流向：内核网络栈 ↔ 驱动（数据被直接丢弃或回环）。
  • 典型应用：为没有物理网卡的机器提供IP地址、路由测试、防火墙规则测试。

MACVLAN 设备

  • 作用：在一个物理网卡上创建多个虚拟网卡，每个都有独立的MAC地址。
  • 数据流向：基于物理网卡进行复用和过滤。
  • 典型应用：容器网络（如Docker）、服务器虚拟化。

你可以用几行命令创建一个 dummy 设备并测试：

# 加载 dummy 内核模块（如果尚未加载）
$ sudo modprobe dummy
# 创建一个名为 mydummy0 的虚拟网卡
$ sudo ip link add mydummy0 type dummy
# 为其配置IP地址并启用
$ sudo ip addr add 192.168.100.1/24 dev mydummy0
$ sudo ip link set dev mydummy0 up
# 现在你可以 ping 192.168.100.1 了
$ ping 192.168.100.1

编者提示：网络驱动中大量使用了中断和DMA。如果你对中断处理流程（顶半部/底半部）或DMA内存分配不熟悉，强烈建议回顾 第5章《中断处理程序设计》 和 第4章《内存管理与分配》 中的相关章节，那里有更基础的原理讲解和代码示例。

对于真实硬件，驱动还需要处理设备电源管理（如WoL唤醒）、 ethtool 工具支持以查询链接状态和速度、以及可能的多队列（RSS）以支持多核处理器。例如，一个现代Intel I350千兆网卡驱动就需要支持MSI-X中断和多队列。

动手试一试

  1. 观察现有驱动：在你的Linux系统上，找到任意一个网络驱动的源码（如 drivers/net/ethernet/realtek/r8169.c）。搜索 “netdev_ops”，找到它的操作函数集，并对照本章表格，看看它实现了哪些函数。
  2. 创建虚拟设备：按照上文步骤，在你自己电脑的Linux虚拟机或开发板上，创建一个 dummy 接口，并为其配置IPv6地址（例如 2001:db8::1/64），然后用 ping6 命令测试连通性。
  3. 代码填空：以下是一个简化版网络驱动发送函数的骨架，请补全关键步骤（提示：需要检查硬件描述符状态、映射DMA、启动发送、更新统计信息）。
    static netdev_tx_t my_start_xmit(struct sk_buff *skb, struct net_device *dev) {
    struct my_private_data *priv = netdev_priv(dev);
    dma_addr_t dma_addr;

    // 1. 检查硬件发送队列是否已满
    if (is_tx_queue_full(priv)) {
        netif_stop_queue(dev); // 停止上层发包
        return NETDEV_TX_BUSY;
    }

    // 2. 为skb的数据区映射DMA地址
    dma_addr = dma_map_single(&dev->dev, skb->data, skb->len, DMA_TO_DEVICE);
    if (dma_mapping_error(&dev->dev, dma_addr)) {
        dev_kfree_skb_any(skb);
        priv->stats.tx_errors++;
        return NETDEV_TX_OK;
    }

    // 3. 将DMA地址和包长度填入硬件发送描述符
    fill_tx_descriptor(priv, dma_addr, skb->len);

    // 4. 通知硬件有新的数据包待发送
    start_tx_engine(priv);

    // 5. 释放skb资源？______ (a) 立即释放 (b) 等发送完成中断中释放
    // 正确答案是 (b)。通常将skb存入私有结构，在发送完成中断中释放。
    priv->tx_skb[priv->tx_tail] = skb;

    // 6. 更新队列尾指针和统计信息
    priv->tx_tail = NEXT_TX(priv->tx_tail);
    priv->stats.tx_packets++;
    priv->stats.tx_bytes += skb->len;

    return NETDEV_TX_OK;
}

检验你的理解

  1. 判断题：网络设备驱动通过 “/dev/eth0” 这样的设备文件与用户程序通信。 ( )
  2. 选择题：在NAPI机制中，当轮询函数处理的数据包数量（work_done）小于预算（budget）时，驱动应该怎么做？
    A. 继续轮询直到达到budget。 B. 调用 napi_complete_done() 并重新开启硬件中断。 C. 直接返回，什么也不做。
  3. 简答题：为什么在高流量场景下，纯中断模式的网络驱动性能会很差？NAPI是如何改善这一点的？

（答案：1. 错；2. B；3. 高流量下每秒产生数万甚至数十万中断，CPU大部分时间用于处理中断上下文切换，效率低下。NAPI在首次中断后关闭中断，进入轮询模式批量处理数据包，大幅减少中断次数，提升吞吐量。）

本章小结

  • 核心是 net_device：网络设备驱动的核心是填充并注册 struct net_device 结构体，特别是其 netdev_ops 操作函数集。
  • 数据包以 sk_buff 形式流转：发送时驱动从 ndo_start_xmit 获取skb；接收时驱动构造skb并通过 netif_receive_skb() 提交给上层。
  • NAPI是高性能接收的标准：采用“中断触发+轮询处理”的混合模式，有效解决高负载下的中断风暴问题，是现代网络驱动的必备实现。
  • 虚拟设备是重要工具：tun/tap、dummy 等虚拟网络设备无需真实硬件，是驱动学习、协议调试和网络功能测试的利器。
  • 注意资源管理细节：正确管理skb生命周期、处理中断共享、保证DMA缓存一致性是驱动稳定运行的关键，也是新手最容易出错的地方。
