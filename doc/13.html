核心概念块设备请求队列

想象一下图书馆，字符设备像是一本只能一页一页顺序阅读的小说，而块设备则像一个可以随意跳转到任何一页的百科全书。在Linux内核中，块设备是指以固定大小的“块”（通常是512字节或4KB）为单位进行随机访问的存储设备，比如硬盘、SSD、U盘。与字符设备驱动的read/write直接对应系统调用不同，块设备驱动的核心是高效地处理来自文件系统的批量I/O请求。

块设备 vs. 字符设备

理解两者的区别是块设备驱动的入门钥匙。字符设备（如键盘、串口）是面向字节流的，读写通常是顺序的。块设备则面向数据块，支持随机访问，并且内核为其提供了复杂的缓存（Page Cache）和I/O调度机制来提升性能。你在项目中如果混淆了两者的注册接口，驱动根本无法正常工作。

块设备驱动的核心数据结构

开发一个最简单的块设备驱动，你需要和三个关键数据结构打交道：struct gendisk（通用磁盘）、struct request_queue（请求队列）和struct block_device_operations（块设备操作集）。它们的关系构成了驱动的基本骨架。

上表中最关键的差异点：block_device_operations 中没有 read 和 write 函数。所有读写请求都由请求队列 (request_queue) 统一管理，这是块设备驱动性能优化的核心。

⚠️ 常见误区

  • 在 block_device_operations 中寻找 read/write： 这是新手最常犯的错。块设备的读写通过请求队列机制异步处理，驱动需要实现的是请求处理函数，而不是直接的 read/write。
  • 过早调用 add_disk： 必须在 gendisk 的所有信息（如容量、队列、操作集）都设置完毕后才能调用 add_disk。一旦调用，内核就可能开始向你的驱动发送I/O请求。
  • 忽略请求队列的锁： blk_init_queue 需要一个自旋锁参数，请求处理函数在该锁的保护下运行。如果驱动其他部分也需要访问共享数据，必须使用同一把锁。

从零实现一个RAMDISK驱动

让我们通过一个在内存中模拟的块设备（RAMDISK）来串联所有概念。这个驱动将分配一段内存（比如16MB），将其伪装成一个硬盘。

第一步：定义设备与初始化队列

首先定义设备结构体，包含gendisk指针、请求队列指针和内存缓冲区。在模块初始化函数中，我们需要完成三件事：1) 申请内存缓冲区；2) 初始化请求队列；3) 分配并设置gendisk。

// 1. 定义设备结构体
struct my_ramdisk_dev {
    unsigned char *data;          // 内存缓冲区
    struct gendisk *gd;           // gendisk结构体
    struct request_queue *queue; // 请求队列
    spinlock_t lock;             // 自旋锁
};

// 2. 模块初始化函数片段
static int __init my_ramdisk_init(void)
{
    struct my_ramdisk_dev *dev;
    int err = 0;

    // 分配设备结构体
    dev = kzalloc(sizeof(struct my_ramdisk_dev), GFP_KERNEL);
    // 分配16MB内存作为“磁盘”空间
    dev->data = vzalloc(16*1024*1024);

    // 初始化自旋锁，用于请求队列
    spin_lock_init(&dev->lock);
    // 初始化请求队列，绑定处理函数 my_request_fn
    dev->queue = blk_init_queue(my_request_fn, &dev->lock);

    // 分配一个gendisk，支持1个设备（即无分区）
    dev->gd = alloc_disk(1);
    // 设置gendisk的主要属性
    dev->gd->major = MY_MAJOR;        // 主设备号
    dev->gd->first_minor = 0;
    dev->gd->fops = &my_block_ops;    // 块设备操作集
    dev->gd->queue = dev->queue;      // 关联请求队列
    dev->gd->private_data = dev;      // 私有数据指针
    sprintf(dev->gd->disk_name, "myram%d", 0);
    set_capacity(dev->gd, 16*2048); // 容量=16MB/512字节=32768个扇区

    // 最后，将磁盘添加到系统
    add_disk(dev->gd);
    return 0;
}

代码中set_capacity设置的容量单位是扇区（Sector），传统硬盘每个扇区512字节。所以16MB的容量就是 `16*1024*1024 / 512 = 32768` 个扇区。我在实际项目中曾忘记做这个转换，结果系统识别出的磁盘容量只有预期的1/512，排查了很久。

请求处理函数：驱动的“心脏”

请求处理函数是驱动真正处理读写的地方。内核的I/O调度器会将多个相邻的读写请求合并（Merge）和排序（Sort），形成一个struct request链表。你的函数需要遍历这个链表，处理每一个请求。

→
使用 blk_fetch_request 从队列获取请求
→
判断 req->cmd_flags (读/写)
→
计算内存偏移，执行 memcpy
→
使用 __blk_end_request_cur 结束请求
static void my_request_fn(struct request_queue *q)
{
    struct request *req;
    struct my_ramdisk_dev *dev = q->queuedata; // 从队列获取私有数据
    int err = 0;

    // 循环处理队列中的所有请求
    while ((req = blk_fetch_request(q)) != NULL) {
        sector_t start_sector = blk_rq_pos(req); // 起始扇区号
        unsigned int nr_sectors = blk_rq_cur_sectors(req); // 当前请求扇区数
        unsigned char *buffer = bio_data(req->bio); // 数据缓冲区
        unsigned char *dev_mem = dev->data + (start_sector << 9); // 设备内存偏移 (扇区号*512)

        // 判断是读请求还是写请求
        if (rq_data_dir(req) == READ) {
            memcpy(buffer, dev_mem, nr_sectors << 9); // 从“设备”读到内存
        } else { // WRITE
            memcpy(dev_mem, buffer, nr_sectors << 9); // 从内存写到“设备”
        }

        // 标记当前请求处理完成，无错误。这是关键步骤！
        if (!__blk_end_request_cur(req, err))
            break; // 如果请求的所有数据段都处理完，req会变为NULL
    }
}

这里有个隐藏的坑：blk_rq_cur_sectors返回的是当前bio（BIO是I/O请求的中间结构）的扇区数，一个复杂的请求可能包含多个bio。我们这里简化处理，对于RAMDISK足够了。但在真实硬盘驱动中，你需要用blk_for_each_segment和blk_rq_map_sg来遍历处理所有分散/聚集（Scatter/Gather）列表。

编者提示： 请求处理函数中涉及大量的内核I/O数据结构，如struct request、struct bio。如果你想深入理解I/O请求在内存中的组织方式，可以结合第4章《内存管理与分配》中关于高端内存和DMA映射的知识，这对编写高性能块设备驱动至关重要。

简单RAMDISK/虚拟设备

  • 使用 blk_init_queue
  • 请求处理函数中直接memcpy
  • 适用于教学、快速原型

基于真实硬件的驱动 (如SD卡)

  • 使用 blk_queue_make_request (绕过I/O调度)
  • 需处理DMA和硬件中断
  • 需实现电源管理

高性能NVMe/SSD驱动

  • 使用多队列块层(blk_mq)
  • 实现轮询(Polling)模式减少延迟
  • 涉及复杂的队列管理和中断绑定

对于大多数嵌入式场景（比如为特定板载Flash芯片写驱动），你很可能从“基于真实硬件的驱动”模式开始。这时，blk_init_queue可能不再是最佳选择，因为它的单队列锁可能成为性能瓶颈。Linux 4.x之后的内核推荐使用多队列接口（blk_mq），但学习曲线更陡。

动手试一试

1. 编译并加载RAMDISK驱动： 将本章的代码片段整合成一个完整的内核模块（需补充模块退出函数、主设备号申请等）。使用insmod加载后，用lsblk或cat /proc/partitions查看是否出现了名为myram0的块设备。
2. 格式化与挂载： 尝试用sudo mkfs.ext4 /dev/myram0在该设备上创建文件系统，然后用mount命令将其挂载到一个目录（如/mnt/test），在里面创建几个文件，测试读写是否正常。
3. 观察I/O调度： 在驱动加载后，查看你的设备的请求队列设置：cat /sys/block/myram0/queue/scheduler。理解当前使用的I/O调度算法（如mq-deadline或none）。

检验你的理解

  1. 为什么块设备驱动操作集（block_device_operations）中没有read和write函数？读写请求是如何被处理的？
  2. 函数add_disk应该在驱动初始化的哪个阶段调用？如果在设置gendisk容量之前调用，会发生什么？
  3. 在请求处理函数中，__blk_end_request_cur(req, 0)这行代码的作用是什么？如果忘记调用它，会导致什么后果？

本章小结

  • 块设备的核心是随机访问和缓存： 它与字符设备的根本区别在于数据组织和访问方式，内核为其提供了请求队列和I/O调度来优化性能。
  • 驱动框架围绕三个结构体构建： gendisk代表设备本体，request_queue管理I/O请求，block_device_operations处理设备控制操作（非读写）。
  • 请求处理函数是驱动引擎： 你需要实现一个函数来处理从队列中取出的request，正确区分读/写方向，完成数据搬运，并最终使用__blk_end_request_cur等函数结束请求。
  • 从简单到复杂的选择： 对于内存模拟设备，blk_init_queue加简单memcpy的模式就足够了；对于真实硬件，需要考虑DMA、中断和更现代的blk_mq框架。
