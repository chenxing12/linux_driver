核心概念kobjectsysfs

在开发一个简单的字符设备驱动时，你可能只关心file_operations和cdev。但当你面对一个拥有多个子设备、需要动态创建属性文件、并能响应热插拔的复杂硬件时，你会需要一个更强大的组织框架。这就是Linux设备模型（Device Model）要解决的问题。

生活类比：设备模型就像公司的组织架构

想象一家科技公司（内核）。它有多个事业部（总线，如PCIe、I2C），每个事业部下有多个项目组（设备驱动），项目组管理着具体的工程师（物理设备）。设备模型就是定义这套层级关系、汇报链路和统一管理规则的“公司章程”。而kobject是这套模型中最基础的“员工”对象，每个设备、驱动、总线在内核中都是一个kobject，它们通过引用计数管理自己的“在职状态”。

设备模型的核心三要素：kobject, ktype, kset

设备模型不是单一的结构，而是由三个紧密协作的核心对象构成。理解它们的关系是掌握设备模型的关键。

对象职责类比关键API示例
kobject基础对象，提供引用计数、父指针、在sysfs中生成一个目录。一个“员工”实体。kobject_init(), kobject_add()
ktype定义kobject的“类型”，包含释放函数、默认属性等。同类型的kobject共享一个ktype。员工的“职位描述”（如“软件工程师”岗位说明书）。struct kobj_type
ksetkobject的集合，本身也是一个kobject。用于将多个kobject组织在一起，在sysfs中表现为一个子目录。一个“部门”，里面有很多员工。kset_create_and_add()

一个常见的误解是直接操作kobject。在实际开发中，我们几乎总是使用更上层的封装，如device、driver、bus，它们内部都包含了一个kobject。但当你需要创建自定义的内核对象并暴露到用户空间时，就需要直接和它们打交道。

常见误区

  • 混淆kobject和ktype： kobject是实例，ktype是类。就像你不能用“工程师岗位说明书”（ktype）去打卡上班，必须有一个具体的“张三”（kobject）。
  • 忘记设置父对象： 调用kobject_add()前必须设置kobj->parent，否则kobject无法正确挂接到sysfs树中，导致找不到目录。
  • 手动管理引用计数： 应使用kobject_get()/kobject_put()，而非直接操作内部的kref。我在一个早期项目中直接操作kref，导致对象在还有引用时被意外释放，引发了内核oops。

Sysfs：用户空间窥探设备模型的窗口

设备模型在内核中构建了一棵对象树，而sysfs（通常挂载在/sys）就是这棵树在用户空间的镜像。每一个kobject对应sysfs中的一个目录，kobject的属性（attribute）则对应目录下的文件。

// 一个简单的kobject属性定义
static ssize_t myattr_show(struct kobject *kobj, struct attribute *attr,
                          char *buf)
{
    return sprintf(buf, "%d\n", 42);
}
static ssize_t myattr_store(struct kobject *kobj, struct attribute *attr,
                           const char *buf, size_t count)
{
    int value;
    sscanf(buf, "%d", &value);
    // 处理value...
    return count;
}
static struct sysfs_ops my_sysfs_ops = {
    .show = myattr_show,
    .store = myattr_store,
};
static struct kobj_type my_ktype = {
    .sysfs_ops = &my_sysfs_ops,
    // .release 和 .default_attrs 也必须设置
};

当你执行cat /sys/class/myled/brightness时，内核会找到对应的kobject，调用其ktype中myattr_show函数来生成文件内容。这个过程是动态的，无需像procfs那样手动创建和移除文件节点。

关键点：kobject的生命周期

kobject的生命周期由引用计数（kref）严格管理。初始化后计数为1。每次kobject_get()增加计数，kobject_put()减少计数。当计数减为0时，会调用其ktype中注册的release函数来释放对象占用的所有资源。这个机制确保了内核对象不会在使用中被释放，也不会在无人引用后造成内存泄漏。

编者提示： 关于sysfs文件系统的更详细操作和编程接口，例如如何创建更复杂的属性文件（二进制属性、符号链接等），我们将在后续专门讲解VFS和文件系统的章节中进行深入探讨。

static int __init virtual_temp_init(void) {
    struct kset *my_kset;
    int ret;

    // 1. 在 /sys/kernel/ 下创建一个名为 “vdevices” 的kset
    my_kset = kset_create_and_add("vdevices", NULL, kernel_kobj);
    if (!my_kset) return -ENOMEM;

    // 2. 分配并初始化我们的温度传感器kobject
    temp_kobj = kzalloc(sizeof(*temp_kobj), GFP_KERNEL);
    kobject_init(temp_kobj, &virtual_ktype); // virtual_ktype需提前定义好
    temp_kobj->kset = my_kset; // 指定所属集合

    // 3. 将kobject添加到sysfs，此时会在 /sys/kernel/vdevices/ 下生成目录
    ret = kobject_add(temp_kobj, NULL, "temp_sensor0");
    if (ret) { kobject_put(temp_kobj); kset_unregister(my_kset); return ret; }

    // 4. 在kobject目录下创建属性文件
    ret = sysfs_create_file(temp_kobj, &temp_attribute.attr);
    if (ret) { kobject_del(temp_kobj); kobject_put(temp_kobj); kset_unregister(my_kset); }

    return ret;
}

模块加载后，你会在/sys/kernel/vdevices/temp_sensor0/目录下看到一个temperature文件。通过cat和echo命令可以读写这个虚拟的温度值。模块退出时，必须在module_exit函数中调用kobject_del()和kobject_put()来清理。

内核事件通知 (uevent)

  • kobject状态变化（如添加、移除）可生成uevent。
  • 用户空间的udev/mdev会监听这些事件，动态创建设备节点。
  • 通过kobject_uevent(&kobj, KOBJ_ADD)主动发送事件。

设备层次结构

  • 通过kobj->parent指针构建父子关系。
  • 在sysfs中表现为嵌套的目录结构。
  • 父对象被删除时，所有子对象也会被递归清理。

默认属性组

  • 在ktype中定义default_attrs数组。
  • kobject添加到sysfs时，会自动创建这些属性文件。
  • 适用于所有同类型kobject的通用属性。

检验你的理解

  1. 如果一个内核模块创建了一个kobject并成功添加到sysfs，但在模块退出时只调用了kfree()而忘记了kobject_put()，可能会导致什么后果？
  2. 在sysfs中，一个目录对应的是kobject、ktype还是kset？一个普通的文件（如`temperature`）又对应什么？
  3. 判断对错：一个kobject可以同时属于多个不同的kset。

参考答案（先思考再看）：
1. 内存泄漏和sysfs残留。kobject的引用计数未归零，其release函数不会被调用，导致其占用的内存及其关联资源（如属性）无法释放。同时，sysfs中对应的目录可能依然存在，访问时会导致内核异常。
2. 目录对应kobject（或kset，因为kset也是kobject）。普通文件对应该kobject的一个属性（attribute）。
3. 错。一个kobject只能通过其`kset`成员指向一个kset。这是由数据结构决定的。

本章小结

  • 设备模型是Linux内核用于统一管理设备、驱动、总线的对象层次框架，其基础构建块是kobject。
  • kobject、ktype、kset三者协同工作：kobject是实例，ktype定义同类对象的行为，kset用于对象分组集合。
  • Sysfs是设备模型在用户空间的直观体现，kobject对应目录，属性（attribute）对应文件，提供了动态的内核对象查看与控制接口。
  • kobject的生命周期由引用计数严格管理，必须使用kobject_get()/put() API，并在计数归零时通过release函数妥善清理资源。
  • 直接使用kobject API适用于创建自定义的内核对象并导出到sysfs，而标准设备驱动通常使用封装度更高的device等接口。
