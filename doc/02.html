第2章 内核模块编译与加载

入门 👤 刚接触内核开发，想跑通第一个驱动模块的程序员 🔧 为硬件编写驱动，或为内核添加新功能模块
本章你将学到
  • 编写一个最简单的“Hello World”内核模块
  • 使用Kbuild系统编译模块，生成.ko文件
  • 使用insmod/rmmod/modprobe等工具加载和卸载模块

核心概念内核模块Makefile/Kbuild

内核模块就像乐高积木，你可以随时把一块新积木（新功能）拼到主体结构（内核）上，也可以随时拆下来，而无需重启整个系统。在技术上，内核模块是一段可以动态加载到内核空间运行的目标代码，它扩展了内核功能，通常用于设备驱动、文件系统或网络协议。

2.1 你的第一个内核模块：Hello World

我们从最经典的代码开始。创建一个名为 hello.c 的文件，输入以下内容：

// hello.c - 最简单的内核模块
#include <linux/init.h>
#include <linux/module.h>

// 模块加载时执行的函数
static int __init hello_init(void)
{
    printk(KERN_INFO "Hello, Kernel World!\n");
    return 0;
}

// 模块卸载时执行的函数
static void __exit hello_exit(void)
{
    printk(KERN_INFO "Goodbye, Kernel World.\n");
}

// 指定模块的入口和出口函数
module_init(hello_init);
module_exit(hello_exit);

// 模块的元信息
MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("A simple hello world module");

代码中有几个关键点：module_init 和 module_exit 是宏，它们告诉内核哪个函数是加载入口和卸载出口。printk 是内核的打印函数，相当于用户空间的 printf，KERN_INFO 是日志级别。所有内核模块必须包含 linux/module.h 头文件。

2.2 编译模块：理解Kbuild与Makefile

内核模块不能直接用gcc编译，必须使用内核的构建系统Kbuild。你需要一个正确的Makefile。这个Makefile的核心是引用内核的构建脚本。

你的 hello.c
+
Makefile
→
Kbuild 系统
↓
hello.ko
+
Module.symvers 等

在你的 hello.c 同目录下，创建如下 Makefile（注意M是大写）：

# 指向你正在运行的内核的构建目录
KDIR := /lib/modules/$(shell uname -r)/build

# 当前目录
PWD := $(shell pwd)

# 目标模块名（注意：生成的文件会是 hello.ko）
obj-m := hello.o

# 默认的构建目标
default:
    $(MAKE) -C $(KDIR) M=$(PWD) modules

# 清理目标
clean:
    $(MAKE) -C $(KDIR) M=$(PWD) clean

现在，在终端执行 make。如果一切顺利，你会看到编译输出，并生成一个 hello.ko 文件。这就是你的内核模块。

⚠️ 常见误区

  • 内核头文件缺失：如果报错“linux/module.h: No such file or directory”，说明你没有安装当前内核的开发头文件。在Ubuntu上运行 sudo apt install linux-headers-$(uname -r)。
  • Makefile格式错误：Makefile中的命令必须以Tab键开头，而不是空格。直接复制粘贴可能导致空格问题，请仔细检查。
  • 内核版本不匹配：KDIR 路径必须指向你正在运行的内核的源码/构建目录。编译出的模块只能加载到对应版本或相近版本的内核中。

2.3 模块加载与卸载工具详解

有了 .ko 文件，就可以动态加载了。加载模块需要root权限。下表对比了常用的模块管理工具：

工具命令示例核心功能与区别典型应用场景
insmodsudo insmod hello.ko最基础的加载工具。只加载指定.ko文件，不解决依赖。测试单个简单模块，或依赖关系明确的场景。
rmmodsudo rmmod hello卸载模块。使用模块名（不是文件名）。卸载已加载的模块。
modprobesudo modprobe hello智能加载工具。会从标准模块目录查找模块，并自动加载其依赖。生产环境加载模块，或模块有复杂依赖时。
lsmodlsmod | grep hello列出当前内核已加载的所有模块及其大小、引用计数。查看模块是否成功加载。
modinfomodinfo hello.ko显示模块的元信息，如作者、许可证、描述、参数等。在加载前了解模块详情。

现在，让我们加载刚编译的模块。打开一个终端，依次执行：

1. 加载模块
sudo insmod hello.ko
→
2. 查看内核日志
dmesg | tail -2
→
3. 确认模块列表
lsmod | grep hello
→
4. 卸载模块
sudo rmmod hello
→
5. 再次查看日志
dmesg | tail -2

你应该在 dmesg 的输出中看到我们写在 printk 里的“Hello”和“Goodbye”信息。如果没看到，可能是日志级别问题，尝试用 dmesg -wH 或 journalctl -k 查看。

关键点：printk与内核日志

printk 的输出不会直接打印到终端，而是进入内核的环形缓冲区。你需要用 dmesg 命令查看。它的输出级别（如KERN_INFO）决定了这条信息是否会显示在系统控制台。在实际驱动开发中，这是最重要的调试手段之一。

2.4 模块参数：让模块可配置

很多时候，我们希望在加载模块时传入一些参数，比如设置调试级别、设备地址等。这可以通过模块参数实现。

修改你的 hello.c，增加以下代码（放在模块元信息之前）：

// 声明模块参数
static char *whom = "World";
static int howmany = 1;
module_param(whom, charp, S_IRUGO); // charp 表示字符指针
module_param(howmany, int, S_IRUGO); // int 表示整型

// 在初始化函数中使用参数
static int __init hello_init(void)
{
    int i;
    for (i = 0; i < howmany; i++)
        printk(KERN_INFO "Hello, %s!\n", whom);
    return 0;
}

重新编译并加载模块，这次可以带上参数：sudo insmod hello.ko whom=”Linux” howmany=3。查看 dmesg，你会看到打印了三次“Hello, Linux!”。使用 modinfo hello.ko 也能看到声明的参数列表。

编者提示：模块参数的类型和权限标志（如S_IRUGO）非常重要。关于权限标志的详细含义（谁可以读写sysfs中的参数），以及更复杂的参数类型（数组、回调函数），将在第26章《设备模型与kobject》中深入探讨。

2.5 模块版本与符号导出

内核有严格的版本校验机制，称为“模块版本校验”（CONFIG_MODVERSIONS）。其目的是防止将针对不同内核API编译的模块加载到当前内核，导致内核崩溃。

场景：模块依赖内核函数

  • 你的模块调用了内核函数 kmalloc
  • 内核升级后，kmalloc 的函数原型可能改变
  • 如果不校验，旧模块调用错误的函数，系统必崩

场景：模块间相互调用

  • 模块A导出一个函数 func_a
  • 模块B依赖并调用 func_a
  • 模块A更新后，func_a 参数变化，模块B需重新编译

当你编译模块时，Kbuild会生成一个 Module.symvers 文件，里面记录了所有使用和导出的内核符号及其CRC校验值。加载模块时，内核会比对校验值。我在项目中踩过这个坑：将一台机器上编译的模块拿到另一台内核版本稍异的机器上加载，直接报错“Invalid module format”，就是版本校验不通过。

要让其他模块使用你的函数，你需要“导出”符号：

// 在模块中定义并导出一个函数
void my_exported_func(void)
{
    printk(KERN_INFO "Function from Module A called.\n");
}
EXPORT_SYMBOL(my_exported_func); // 或者使用 EXPORT_SYMBOL_GPL 仅对GPL模块可见

实际开发中，大部分驱动项目会包含多个相互协作的 .ko 文件，符号导出是它们通信的基础。

🔧 实战技巧：快速排错

  • 加载失败：Invalid module format：首先用 uname -r 确认运行内核版本，再用 modinfo hello.ko | grep vermagic 查看模块编译的内核版本。两者必须一致。解决方法是用当前内核头文件重新编译。
  • 加载失败：Unknown symbol in module：模块依赖的某个内核函数或变量找不到。检查是否拼写错误，或者该内核配置选项未开启（如某些函数只在启用DEBUG_FS时存在）。使用 sudo cat /proc/kallsyms | grep 函数名 可以查看该符号是否在内核中。
  • 卸载失败：Module is in use：模块的引用计数不为0（可能有进程正打开该驱动设备）。检查是否有用户态程序未关闭。这是驱动设计需要考虑的，我们会在后续字符设备驱动章节详细讨论。

动手试一试

  1. 基础任务：按照本章步骤，成功编译并加载 hello.ko 模块，在 dmesg 中看到打印信息。
  2. 进阶任务：为模块添加一个整型参数 debug_level（0-3）。在初始化函数中，根据 debug_level 的值打印不同详细程度的信息（例如，0=只打印错误，1=打印基本信息，2=打印调试信息）。尝试在加载时传入 debug_level=2。
  3. 探索任务：创建两个模块 mod_a.ko 和 mod_b.ko。在 mod_a 中导出一个简单的函数（比如打印一句话），在 mod_b 的初始化函数中调用它。注意加载顺序：先加载 mod_a，再加载 mod_b。观察结果，然后尝试颠倒顺序，看看会发生什么错误。

检验你的理解

  1. 内核模块的入口函数和出口函数是通过哪两个宏指定的？如果模块初始化失败（比如资源申请失败），入口函数应返回什么值？
  2. 使用 insmod 和 modprobe 加载模块的主要区别是什么？在什么情况下你必须使用 modprobe？
  3. 编译模块时，Makefile中的 obj-m := hello.o 这一行，如果模块源文件是 my_module.c，这行应该怎么写？如果模块由 main.c 和 helper.c 两个源文件链接而成，又该怎么写？
  4. 模块参数 module_param(array_param, int, 0644); 这行声明有什么问题？（提示：参数类型和权限）

本章小结

  • 内核模块是动态加载的内核代码，通过 module_init 和 module_exit 指定生命周期函数，使用 printk 输出日志到内核缓冲区。
  • 编译依赖Kbuild系统，关键是一个指向内核构建目录的正确Makefile，生成 .ko 目标文件。
  • 加载与卸载工具各司其职：insmod/rmmod 用于简单操作，modprobe 能自动处理依赖，lsmod 和 modinfo 用于查看状态和信息。
  • 模块参数增强了灵活性，允许在加载时配置模块行为，通过 module_param 宏声明。
  • 模块版本校验和符号导出 是保证内核稳定性和模块间协作的基础机制，编译环境与运行环境的内核版本必须匹配。
