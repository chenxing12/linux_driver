本章你将学到

  • 将自定义驱动与用户空间工具（如iproute2）集成
核心概念net_deviceNAPI

想象一下，你有一个特殊的硬件盒子，它需要拦截所有经过的网络数据，检查一下内容，或者给数据包盖个“内部专用”的戳，再决定是放行还是丢弃。这个盒子就是我们要实现的定制网络设备。在Linux内核中，它本质上是一个虚拟的网络接口，我们可以完全控制其收发包的行为逻辑。

核心结构体：net_device

这就像网络设备的“身份证”和“能力说明书”。内核通过这个结构体来管理一个网络接口的所有信息，从MAC地址、最大传输单元（MTU）到最重要的——发送和接收数据包的回调函数指针。我们驱动的主要工作就是填充这个结构体，并注册到内核网络子系统中。

一个功能完整的定制网络驱动，其内部各模块的协作关系可以用下面的架构图清晰地展示：

这个架构的核心是struct net_device。对于虚拟设备，我们使用alloc_netdev来分配它。一个关键决策是选择中断模式还是NAPI模式。对于高性能或模拟真实硬件的虚拟设备，我强烈推荐使用NAPI（New API），它能有效减少中断开销，提升吞吐量。实际项目中，当包速率超过10Kpps时，NAPI的优势就非常明显了。

// 分配并初步设置一个虚拟网络设备
struct net_device *my_dev;
my_dev = alloc_netdev(0, "mynet%d", NET_NAME_UNKNOWN, my_setup);
if (!my_dev) {
    return -ENOMEM;
}
// 启用NAPI
netif_napi_add(my_dev, &my_priv->napi, my_poll, 64); // 64为权重

数据包的生命周期：sk_buff

内核用struct sk_buff（socket buffer）表示一个数据包。驱动开发者必须熟悉它。关键字段：data（指向包数据）、len（数据长度）、head/end（缓冲区边界）。发送时，协议栈将构造好的skb交给驱动的ndo_start_xmit；接收时，驱动需要分配或克隆一个skb，填充数据后，通过netif_receive_skb或NAPI的napi_gro_receive提交给协议栈。

核心操作函数集实现

驱动能力通过struct net_device_ops暴露。下表列出了定制驱动最必须实现的几个操作及其典型任务：

操作函数回调成员核心任务返回值说明
打开设备ndo_open申请资源（如DMA缓冲区），启动硬件或模拟队列，调用netif_start_queue。成功返回0，失败返回负的错误码。
关闭设备ndo_stop停止数据流(netif_stop_queue)，释放资源，确保无进行中的DMA。总是返回0。
发送数据包ndo_start_xmit最关键的函数。处理上层下发的skb，可进行过滤、修改、统计。最后必须释放skb(dev_kfree_skb)。NETDEV_TX_OK表示成功，NETDEV_TX_BUSY（已很少用）表示队列满。
改变MTUndo_change_mtu检查新MTU值是否在合理范围（如68-1500），更新dev->mtu。成功返回0。
获取统计信息ndo_get_stats64填充struct rtnl_link_stats64，更新收发包数、字节数、错误计数等。返回指向统计结构的指针。

这里有个隐藏的坑：在ndo_start_xmit中，如果你只是“窥视”数据包而不立即发送，需要小心skb被协议栈重用的情况。我踩过这个坑，错误的修改导致了其他网络连接的随机错误。安全的做法是，如果需要异步处理或修改，使用skb_copy复制一份，再处理副本。

⚠️ 常见误区

  • 在中断上下文调用可能睡眠的函数：ndo_start_xmit运行在中断上下文（软中断），绝对不能调用kmalloc(GFP_KERNEL)、mutex_lock等可能引起睡眠的函数。应使用GFP_ATOMIC标志。
  • 忘记释放skb：无论发送成功还是失败，在ndo_start_xmit函数退出前，都必须通过dev_kfree_skb()或类似函数释放传入的skb，否则会导致内存泄漏。
  • 统计信息不同步：在SMP系统上，更新stats->tx_bytes等统计变量时，需要使用原子操作（如atomic_long_add）或使用内核提供的dev_sw_netstats_tx_add等辅助函数，避免计数错误。

数据包处理实战：实现一个简单过滤器

让我们实现一个具体功能：过滤所有目标端口为23（Telnet）的TCP数据包。这需要在ndo_start_xmit中实现。

skb进入 ndo_start_xmit
→
检查skb协议类型 == ETH_P_IP?
→
解析IP头，检查协议 == IPPROTO_TCP?
→
计算TCP头偏移，获取目标端口
→
端口 == 23？
是：更新统计，释放skb
→
返回 NETDEV_TX_OK
否：调用真实的发送函数（或原路转发）
→
释放skb，返回 NETDEV_TX_OK
static netdev_tx_t my_start_xmit(struct sk_buff *skb, struct net_device *dev) {
    struct my_priv *priv = netdev_priv(dev);
    struct iphdr *iph;
    struct tcphdr *th;
    unsigned int len = skb->len;

    // 1. 更新基础统计（原子操作）
    u64_stats_update_begin(&priv->tx_syncp);
    priv->stats.tx_packets++;
    priv->stats.tx_bytes += len;
    u64_stats_update_end(&priv->tx_syncp);

    // 2. 过滤逻辑：检查是否为IPv4/TCP包
    if (skb->protocol == htons(ETH_P_IP)) {
        iph = ip_hdr(skb);
        if (iph->protocol == IPPROTO_TCP) {
            th = tcp_hdr(skb);
            if (th->dest == htons(23)) { // 目标端口23
                priv->stats.tx_dropped++; // 更新丢弃计数
                dev_kfree_skb(skb);
                return NETDEV_TX_OK; // 过滤丢弃
            }
        }
    }

    // 3. 此处可调用真实硬件发送函数，对于纯虚拟设备，可以模拟发送成功
    // 例如，将skb放入一个内部队列，由另一个线程处理，或直接环回。
    // 这里我们简单模拟发送成功。
    dev_kfree_skb(skb);
    return NETDEV_TX_OK;
}

编者提示：关于sk_buff的详细操作、内存布局以及更安全的协议头访问函数（如skb_transport_header），在第4章《内存管理与分配》和第12章《网络设备驱动框架》中有深入剖析。在编写生产级代码时，务必使用内核提供的封装函数来访问各层协议头，以确保兼容性和安全性。

接收路径与NAPI轮询

对于虚拟设备，数据来源可能是另一个内核模块、用户空间程序，或者我们自己构造的测试数据。接收数据的关键是触发NAPI调度。

数据源：内核模块

  • 其他驱动（如USB网卡）将skb送入你的驱动。
  • 调用 napi_schedule(&my_priv->napi) 触发轮询。

数据源：用户空间

  • 通过 write() 系统调用写入原始数据。
  • 在驱动中构造skb，调用 netif_rx() 或触发NAPI。

数据源：内部生成

  • 用于测试或构造特定协议包（如ARP响应）。
  • 分配skb，填充数据，提交给协议栈。

NAPI的轮询函数my_poll模板如下。它的任务是在预算（budget）内，从设备的接收队列中取出尽可能多的数据包，并通过napi_gro_receive送交协议栈。

static int my_poll(struct napi_struct *napi, int budget) {
    struct my_priv *priv = container_of(napi, struct my_priv, napi);
    struct sk_buff *skb;
    int work_done = 0;

    while (work_done < budget) {
        skb = my_get_rx_skb(priv); // 从私有队列获取一个skb
        if (!skb) {
            break;
        }
        skb->dev = priv->dev;
        skb->protocol = eth_type_trans(skb, priv->dev);
        napi_gro_receive(napi, skb);
        work_done++;
    }

    if (work_done < budget) {
        napi_complete_done(napi, work_done); // 处理完毕，禁用NAPI
    }
    return work_done;
}

性能与调试

使用ethtool -S mynet0可以查看我们自定义的驱动统计信息。为了支持这个，我们需要实现ndo_get_ethtool_stats和ndo_get_strings回调。调试时，/proc/net/dev和dropwatch工具是定位丢包问题的利器。如果发现tx_dropped计数异常增长，首先检查你的过滤或发送逻辑是否正确释放了skb。

动手试一试

1. 构建一个最简单的环回设备：基于本章框架，创建一个名为mylo的虚拟网卡。修改ndo_start_xmit函数，使其不进行任何过滤，直接将接收到的skb通过netif_rx或NAPI路径“环回”给协议栈，实现自己发给自己收。用ping 192.168.1.100（假设你为该接口配置了这个IP）测试连通性。
2. 添加统计信息：在私有数据结构中增加unsigned long tx_filtered_count计数器。修改端口过滤代码，每当丢弃一个Telnet包时，此计数器加1。并实现ndo_get_ethtool_stats，让这个自定义计数器可以通过ethtool -S mynet0显示出来。
3. 集成iproute2：使用ip link add name mynet type dummy（如果支持）或直接使用insmod加载你的驱动模块。然后使用ip link set mynet up和ip addr add 10.0.0.1/24 dev mynet命令配置接口。

本章小结

  • 定制网络设备的核心是填充并注册struct net_device，并实现其操作集net_device_ops，特别是ndo_start_xmit发送函数。
  • 数据包处理基石是sk_buff结构体。驱动在发送路径消费skb，在接收路径构造并提交skb。必须严格遵守内存管理和上下文规则。
  • 高性能接收推荐NAPI模型，通过napi_schedule触发轮询，在poll函数中批量处理接收队列的数据包。
  • 驱动应提供完善的统计信息，并可通过/proc、sysfs或ethtool接口访问，这是调试和监控的基础。
  • 虚拟网络设备功能强大且灵活，可用于实现软件交换机、流量过滤器、隧道协议、网络模拟器等高级功能。
