核心概念ProcfsSysfs

内核驱动开发中，经常需要向用户空间暴露一些信息：可能是驱动内部的统计计数、调试开关，或是硬件寄存器的当前值。直接使用printk输出不够灵活，而创建完整的字符设备又显得笨重。这时，/proc和/sys文件系统就成了最轻量、最直接的桥梁。

生活类比： 想象一下汽车的仪表盘（Proc）和中控台的设置菜单（Sys）。仪表盘主要显示车速、油量等运行状态（多为只读）；而设置菜单允许你调整空调温度、驾驶模式等参数（可读写）。在Linux内核中，/proc（Procfs）就像一个内核的“仪表盘”，用于报告系统和进程的运行时信息；/sys（Sysfs）则像一个“设备参数控制台”，专门用于展示和调整设备、驱动、总线的属性和配置。

Procfs vs Sysfs：一张表看清本质区别

很多新手会把两者混用。我在早期项目里就犯过这个错误，把本该放在/sys的设备配置项放到了/proc下，导致系统管理员找不到标准的管理入口。它们的定位有根本不同。

对比维度Proc文件系统 (Procfs)Sys文件系统 (Sysfs)
主要目的报告进程和内核运行时状态（如/proc/cpuinfo, /proc/meminfo）暴露设备、驱动、总线的层次化属性和配置（如/sys/class/, /sys/bus/）
数据性质多为只读的统计、状态、调试信息大量可读写的属性，用于控制设备行为
组织方式早期较为扁平、随意，文件命名和格式不统一严格遵循设备模型的层次化目录结构（kobject）
内核接口struct proc_dir_entry, proc_create设备属性组 (struct attribute_group), device_create_file
典型位置/proc/your_module/sys/class/your_class/your_device
官方态度不鼓励为新驱动创建Proc节点，仅用于进程信息或历史兼容推荐的、标准的设备信息导出方式

简单来说：想报告一个驱动处理了多少个数据包？用Procfs。想让用户能动态调整驱动的采样率？用Sysfs。 从内核2.6开始，Sysfs已经成为设备驱动与用户空间交互的“正规军”。

Procfs实战：创建内核状态报告文件

尽管官方不推荐为新驱动使用，但Procfs在输出调试信息、临时统计值时依然简单快捷。其核心是proc_create函数和对应的文件操作函数。

关键步骤：创建Proc文件

  1. 定义文件操作函数： 实现read（和可选的write）回调，将内核数据格式化输出到用户缓冲区。
  2. 创建文件节点： 在模块初始化时调用proc_create(“文件名”, 权限, 父目录, &文件操作)。
  3. 清理： 在模块退出时用remove_proc_entry删除节点。

下面是一个完整的例子，创建一个可读写的Proc文件，用于报告和设置一个简单的计数器。

// 定义驱动内部变量
static int my_counter = 0;
static struct proc_dir_entry *my_proc_file;

// Proc文件的读回调函数
static ssize_t my_proc_read(struct file *file, char __user *buf,
                          size_t count, loff_t *ppos)
{
    char data[32];
    int len;

    if (*ppos > 0) // 如果已经读过，返回0表示结束
        return 0;

    len = snprintf(data, sizeof(data), "Current counter: %d\n", my_counter);
    if (copy_to_user(buf, data, len))
        return -EFAULT;

    *ppos = len;
    return len;
}

// Proc文件的写回调函数
static ssize_t my_proc_write(struct file *file, const char __user *buf,
                           size_t count, loff_t *ppos)
{
    char data[32];
    unsigned long val;

    if (count >= sizeof(data))
        return -EINVAL;

    if (copy_from_user(data, buf, count))
        return -EFAULT;
    data[count] = '\0';

    if (kstrtoul(data, 10, &val)) // 将字符串转换为长整型
        return -EINVAL;

    my_counter = val;
    return count;
}

// 定义文件操作结构体
static const struct file_operations my_proc_fops = {
    .owner = THIS_MODULE,
    .read = my_proc_read,
    .write = my_proc_write,
};

// 模块初始化函数
static int __init my_module_init(void)
{
    // 在/proc根目录下创建文件“my_counter”，权限0644（用户可读写）
    my_proc_file = proc_create("my_counter", 0644, NULL, &my_proc_fops);
    if (!my_proc_file) {
        pr_err("Failed to create proc entry\n");
        return -ENOMEM;
    }
    pr_info("Proc entry created\n");
    return 0;
}

// 模块退出函数
static void __exit my_module_exit(void)
{
    remove_proc_entry("my_counter", NULL);
    pr_info("Proc entry removed\n");
}

⚠️ Procfs常见误区

  • 忘记处理`*ppos`： 在read函数中，必须正确更新*ppos。如果文件内容一次性读完，需要在第二次调用时返回0，否则会导致用户空间程序（如cat）陷入死循环。
  • 缓冲区溢出： 使用snprintf而非sprintf来格式化字符串，确保不会超出内核栈上的临时缓冲区大小。
  • 权限位设置不当： proc_create的权限参数（如0644）是八进制数，不要写成十进制644，否则实际权限会出错。

Sysfs实战：为设备驱动暴露标准属性

Sysfs与内核设备模型深度绑定。每个在Sysfs中出现的目录或文件，背后都对应着一个kobject。对于设备驱动而言，我们通常不需要直接创建kobject，而是利用现有的设备对象（struct device）来创建属性文件。

编者提示： Sysfs的底层基石是kobject和内核设备模型。如果你对struct device、struct kobject感到陌生，建议在学习本章后，结合第26章《设备模型与kobject》深入理解，这将帮助你构建更清晰的内核对象层次概念。

假设我们有一个虚拟的“健康监测”驱动，需要暴露两个属性：只读的current_value和可读写的sampling_rate。

// 驱动内部变量
static int sampling_rate_hz = 100; // 默认100Hz
static int simulated_value = 512;

// 属性1：显示当前采样值 (只读)
static ssize_t current_value_show(struct device *dev,
                                 struct device_attribute *attr, char *buf)
{
    return sprintf(buf, "%d\n", simulated_value);
}
static DEVICE_ATTR_RO(current_value); // 宏定义只读属性

// 属性2：显示和设置采样率 (可读写)
static ssize_t sampling_rate_show(struct device *dev,
                                  struct device_attribute *attr, char *buf)
{
    return sprintf(buf, "%d\n", sampling_rate_hz);
}

static ssize_t sampling_rate_store(struct device *dev,
                                   struct device_attribute *attr,
                                   const char *buf, size_t count)
{
    int val;
    if (kstrtoint(buf, 10, &val))
        return -EINVAL;
    if (val < 1 || val > 1000) // 限制范围1-1000Hz
        return -EINVAL;

    sampling_rate_hz = val;
    // 这里通常需要将新采样率配置到硬件寄存器
    pr_info("Sampling rate changed to %d Hz\n", val);
    return count;
}
static DEVICE_ATTR_RW(sampling_rate); // 宏定义读写属性

// 将属性组织成数组，方便批量创建和删除
static struct attribute *health_attrs[] = {
    &dev_attr_current_value.attr,
    &dev_attr_sampling_rate.attr,
    NULL,
};
static const struct attribute_group health_attr_group = {
    .attrs = health_attrs,
};

// 在probe函数中（假设dev是已分配好的struct device指针）
static int health_probe(struct platform_device *pdev)
{
    struct device *dev = &pdev->dev;
    int ret;

    // 一次性创建属性组中的所有文件
    ret = sysfs_create_group(&dev->kobj, &health_attr_group);
    if (ret) {
        dev_err(dev, "Failed to create sysfs group\n");
        return ret;
    }
    return 0;
}

// 在remove函数中删除属性组
static int health_remove(struct platform_device *pdev)
{
    struct device *dev = &pdev->dev;
    sysfs_remove_group(&dev->kobj, &health_attr_group);
    return 0;
}

编译加载驱动后，你会在/sys/devices/platform/你的设备名/目录下看到current_value和sampling_rate两个文件。使用cat和echo命令即可交互。

调试与监控

  • 场景： 快速查看驱动内部统计（如中断次数、缓冲区使用率）。
  • 推荐： 使用Procfs创建临时只读文件。方法简单，无需绑定设备模型。

设备参数配置

  • 场景： 让用户动态调整驱动工作模式、阈值、采样率等。
  • 推荐： 使用Sysfs创建标准属性文件。这是设备驱动的标准做法，易于集成到系统管理工具。

复杂状态导出

  • 场景： 需要导出具有复杂结构或二进制数据（如寄存器dump）。
  • 推荐： 考虑创建专用的字符设备（/dev/xxx）或使用debugfs。Procfs和Sysfs更适合文本型数据。

性能与安全提示

show和store函数会在用户每次读写文件时被调用，且执行在进程上下文中。避免在其中进行耗时操作（如大的内存分配、长时间休眠）或直接进行硬件IO。对于复杂的硬件操作，应该将参数存储在驱动变量中，然后由内核线程或中断处理程序去实际执行。同时，在store函数中必须严格验证用户输入的范围和格式，防止内核崩溃或安全漏洞。

动手试一试

1. 基础练习： 基于上面的Procfs示例代码，创建一个内核模块，在/proc下生成文件sysinfo。当用户读取时，输出当前内核版本（使用utsname()->release）和模块加载后的jiffies值（get_jiffies_64()）。

2. 进阶练习： 编写一个简单的平台驱动（可参考第6章框架），为其在Sysfs中创建两个属性：debug_level（可读写，范围0-3）和hw_version（只读，返回一个固定的字符串如“v1.2”）。加载驱动后，使用echo 2 > /sys/.../debug_level进行设置，并用cat验证。

3. 排错挑战： 故意在Proc文件的read函数中不更新*ppos，观察使用cat命令时会发生什么现象。在store函数中故意不验证输入，直接赋值给一个8位宽的硬件寄存器变量，然后传入一个300，观察内核日志（dmesg）是否有相关错误。

本章小结

  • Procfs主要用于报告内核和进程的运行时状态与统计信息，接口简单但已不推荐为新驱动创建节点，适合临时调试。
  • Sysfs是内核设备模型的对外窗口，用于以层次化目录结构暴露设备、驱动、总线的属性，是设备驱动与用户空间交互的推荐标准方式。
  • 实现Proc文件需要定义file_operations的read/write回调，并使用proc_create/remove_proc_entry进行生命周期管理。
  • 实现Sysfs属性需要为struct device_attribute定义show和store方法，并通过sysfs_create_group将其绑定到设备的kobject上。
  • 在回调函数中必须注意缓冲区安全、正确更新文件位置指针（Proc）、严格验证用户输入（Sysfs），并避免执行耗时操作。
