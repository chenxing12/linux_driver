核心概念platform_deviceplatform_driver

想象一下，你买了一套精装修的房子，开发商已经预埋好了水管、电线、网线接口（这些是平台设备），你只需要购买对应的热水器、路由器（这些是平台驱动）插上去就能用，而不需要自己凿墙布线。在Linux内核中，平台设备驱动模型就是用来管理那些集成在CPU芯片内部或板级固定的“基础设施”外设，比如集成在ARM SoC上的GPIO控制器、I2C总线控制器、定时器等。

平台设备 (Platform Device)

生活类比：就像主板上焊死的一个USB控制器芯片，位置和功能固定，无法热插拔。
技术定义：代表系统中物理上不可移除或位置固定的设备，通常通过struct platform_device结构体描述，包含设备名称、ID、资源（内存地址、中断号）等信息。

平台驱动 (Platform Driver)

生活类比：就像这个USB控制器的驱动程序，知道如何操作这个固定位置的芯片。
技术定义：用于驱动平台设备的驱动程序，通过struct platform_driver结构体描述，包含驱动名称、探测(probe)、移除(remove)等回调函数。

这个模型的核心是“匹配然后探测”。内核启动时，会注册所有已知的平台设备。当你加载一个平台驱动模块时，内核会遍历设备列表，根据名称或兼容性字符串进行匹配。一旦匹配成功，就调用驱动的probe()函数来初始化设备。

平台驱动模型的核心数据结构

理解平台驱动，首先要看懂两个核心结构体。下面的表格清晰地展示了它们的关键成员和用途。

结构体关键成员作用描述类比

在实际开发中，大部分团队会选择使用设备树（Device Tree）来静态声明平台设备，这比在内核代码中硬编码platform_device更灵活。但为了理解原理，我们先看一个传统的、通过C代码注册平台设备的流程。

动手写一个简单的平台驱动

我们以一个虚拟的“LED控制器”为例，假设它映射在物理地址0x10000000，长度为0x1000，使用中断号101。

首先，我们需要定义平台设备及其资源。在早期的内核或某些特定板级文件中，你会看到这样的代码：

/* 1. 定义设备资源：内存区域和中断 */
static struct resource fake_led_resources[] = {
    {
        .start = 0x10000000,
        .end   = 0x10000FFF,
        .flags = IORESOURCE_MEM, // 内存资源
    },
    {
        .start = 101,
        .end   = 101,
        .flags = IORESOURCE_IRQ, // 中断资源
    },
};

/* 2. 定义平台设备 */
static struct platform_device fake_led_device = {
    .name = "my_led_ctrl", // 关键匹配名
    .id   = -1, // 只有一个设备，用-1
    .num_resources = ARRAY_SIZE(fake_led_resources),
    .resource = fake_led_resources,
    // .dev.platform_data 可以传递私有数据
};

/* 3. 在板级初始化函数中注册设备 */
platform_device_register(&fake_led_device);

接下来是驱动部分。驱动模块需要实现platform_driver，并在probe函数中获取设备资源。

#include <linux/platform_device.h>
#include <linux/io.h>

static int fake_led_probe(struct platform_device *pdev)
{
    struct resource *mem_res;
    struct resource *irq_res;
    void __iomem *reg_base;
    int irq_num;

    // 获取内存资源
    mem_res = platform_get_resource(pdev, IORESOURCE_MEM, 0);
    if (!mem_res) {
        dev_err(&pdev->dev, "Failed to get MEM resource\n");
        return -ENXIO;
    }

    // 将物理地址映射到内核虚拟地址空间
    reg_base = devm_ioremap_resource(&pdev->dev, mem_res);
    if (IS_ERR(reg_base))
        return PTR_ERR(reg_base);

    // 获取中断资源
    irq_res = platform_get_resource(pdev, IORESOURCE_IRQ, 0);
    if (!irq_res) {
        dev_err(&pdev->dev, "Failed to get IRQ resource\n");
        return -ENXIO;
    }
    irq_num = irq_res->start;

    dev_info(&pdev->dev, "Probe success! Reg base=%px, IRQ=%d\n", reg_base, irq_num);
    // 后续：申请中断、初始化硬件、注册字符设备等...
    return 0;
}

static int fake_led_remove(struct platform_device *pdev)
{
    dev_info(&pdev->dev, "Device removed\n");
    return 0;
}

/* 定义平台驱动结构体 */
static struct platform_driver fake_led_driver = {
    .driver = {
        .name = "my_led_ctrl", // 必须与 platform_device.name 一致
        .owner = THIS_MODULE,
    },
    .probe  = fake_led_probe,
    .remove = fake_led_remove,
};

/* 模块加载和卸载函数 */
module_platform_driver(fake_led_driver); // 这个宏简化了注册和注销

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");

关键点：platform_get_resource 与 devm_ioremap_resource

在probe函数中，platform_get_resource()是获取设备预定义资源（内存、中断）的标准方法。而devm_ioremap_resource()是更安全的内存映射函数，它自动管理映射生命周期（当设备或驱动卸载时自动取消映射），我在项目中踩过忘记iounmap导致内存泄漏的坑，强烈推荐使用devm_系列托管函数。

⚠️ 常见误区

  • 匹配失败：最常见的问题是platform_device.name和platform_driver.driver.name不匹配，或者设备根本没有被注册到内核。加载驱动后probe没被调用，首先检查/sys/bus/platform/devices/下有没有你的设备。
  • 资源获取失败：platform_get_resource返回NULL。检查设备定义中的资源数组索引是否正确，第一个资源的索引是0。我曾把IORESOURCE_MEM和IORESOURCE_IRQ的顺序搞反，调试了半天。
  • 地址映射错误：使用ioremap后忘记iounmap，或者映射地址范围超出实际物理地址。对于内存映射，优先使用devm_ioremap_resource()，它内部会进行资源有效性检查。

平台数据：传递自定义参数

有时候，驱动需要一些硬件配置参数，比如LED的默认亮度、GPIO的极性等。这些不适合放在标准资源里，可以通过platform_data传递。

定义私有数据结构体
→
填充数据并赋值给 platform_device.dev.platform_data
→
在驱动的probe函数中通过 pdev->dev.platform_data 获取
/* 设备端：定义并传递数据 */
struct my_led_platform_data {
    int default_brightness;
    bool active_low;
};

static struct my_led_platform_data led_data = {
    .default_brightness = 50,
    .active_low = true,
};

static struct platform_device fake_led_device = {
    .name = "my_led_ctrl",
    .id = -1,
    .dev = {
        .platform_data = &led_data, // 传递私有数据
    },
};

/* 驱动端：接收并使用数据 */
static int fake_led_probe(struct platform_device *pdev)
{
    struct my_led_platform_data *pdata = dev_get_platdata(&pdev->dev);
    // 或者直接使用 pdev->dev.platform_data (需要类型转换)

    if (pdata) {
        printk("Default brightness: %d, Active low: %d\n",
               pdata->default_brightness, pdata->active_low);
    }
    return 0;
}

编者提示：平台数据(platform_data)是一种传统的、基于C代码的配置方式。尤其是ARM体系架构，设备树(Device Tree)已经全面取代了它，成为描述硬件配置的标准方式。设备树可以更清晰、更灵活地描述硬件资源和属性。我们将在下一章《设备树DTS语法解析》中深入探讨。

现代内核：从平台数据到设备树

虽然本章重点讲解传统的平台设备模型，但你必须知道现代的标准做法。设备树通过.dts文件描述硬件，驱动通过of_match_table进行匹配，并使用Device Tree API（如of_property_read_u32()）读取属性。

传统方式 (Legacy)

  • 硬件信息硬编码在内核C文件
  • 使用 platform_device/platform_data
  • 修改硬件需重新编译内核
  • 适用于简单、固定的x86 PC架构

设备树方式 (Modern)

  • 硬件信息写在独立的.dts文件
  • 驱动通过 of_match_table 匹配
  • 使用 of_ 系列API获取资源
  • ARM/MIPS/嵌入式系统事实标准

一个现代的平台驱动，其驱动结构体通常会同时支持两种匹配方式，但以设备树为主：

static const struct of_device_id my_led_of_match[] = {
    { .compatible = "vendor,my-led-ctrl" }, // 与设备树节点匹配
    {},
};
MODULE_DEVICE_TABLE(of, my_led_of_match);

static struct platform_driver fake_led_driver = {
    .driver = {
        .name = "my_led_ctrl",
        .of_match_table = of_match_ptr(my_led_of_match), // 设备树匹配表
    },
    .probe = fake_led_probe,
    .remove = fake_led_remove,
};

在probe函数中，你可以通过pdev->dev.of_node获取设备树节点，然后解析属性。实际开发中，对于新的ARM平台（如NXP i.MX系列、TI AM335x、Rockchip RK系列），你必须使用设备树。

动手试一试

1. 模拟实验：在QEMU或你的开发板上（如树莓派），创建一个最简单的平台驱动模块。不访问真实硬件，只在probe函数中打印一条信息（如“My Platform Driver Probed!”）。通过insmod加载，观察内核日志(dmesg)，确认probe是否被调用。
2. 代码阅读：在内核源码中（如drivers/leds/leds-gpio.c）找一个真实的平台驱动例子，跟踪它的platform_driver定义、probe函数实现，看它如何获取资源（内存、中断、GPIO号）。
3. 资源获取：修改你的驱动，假设你的设备需要两个内存区域（例如寄存器区和数据缓冲区），在probe中练习使用platform_get_resource分别获取它们，并打印出起始地址和长度。

本章小结

  • 平台设备模型用于管理CPU集成或板级固定的外设，核心是platform_device（描述设备）和platform_driver（驱动设备）的匹配与探测机制。
  • 匹配与探测：驱动通过名称与设备匹配，成功后内核自动调用驱动的probe()函数进行初始化，这是驱动执行的起点。
  • 资源管理：使用platform_get_resource()从设备获取预定义的内存、中断等资源；使用devm_ioremap_resource()等托管函数进行资源映射，可以自动释放，避免泄漏。
  • 数据传递：可通过platform_data传递设备私有数据，但现代嵌入式开发中，这已被更灵活、标准的设备树(Device Tree)所取代。
  • 开发模式：传统x86架构可能仍用C代码硬编码设备，而ARM等嵌入式架构已全面转向设备树描述硬件，驱动需通过of_match_table匹配并解析设备树节点。
