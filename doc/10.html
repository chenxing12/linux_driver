调试一个基于SPI的OLED屏幕时，屏幕显示乱码。我检查了电源、复位信号，最后用逻辑分析仪抓取波形，发现是SPI的时钟极性（CPOL）设置错了。这个经历让我深刻体会到，理解SPI的四种工作模式是驱动开发的第一步，它比I2C更灵活，但也更容易出错。

SPI协议：四线制全双工通信

你可以把SPI想象成一条工厂流水线。主设备（Master）是控制中心，它通过时钟线（SCLK）发出统一的“滴答”声，指挥所有从设备（Slave）同步工作。数据在两条独立的流水线（MOSI和MISO）上同时传输，就像原料送入和成品送出同时进行。

技术定义上，SPI（Serial Peripheral Interface）是一种高速、全双工、同步的串行通信总线。它使用四根线：SCLK（时钟）、MOSI（主设备输出/从设备输入）、MISO（主设备输入/从设备输出）和CS（片选）。核心特点是通信速率高（可达几十MHz），没有复杂的地址和应答机制，但每个从设备需要独占一根片选线。

SPI vs I2C：关键差异

实际开发中，选择SPI还是I2C取决于外设需求和硬件资源。SPI速率快，适合大数据量传输（如Nor Flash、TFT屏）；I2C节省引脚，适合连接多个低速设备（如温湿度传感器）。我通常用SPI接W25Q128JV Flash（时钟可达104MHz），用I2C接BMP280气压计。

SPI的四种工作模式

SPI模式由时钟极性（CPOL）和时钟相位（CPHA）两个参数组合而成，决定了数据在时钟的哪个边沿被采样。这是驱动配置中最关键也最容易出错的地方。

模式CPOLCPHA空闲时SCLK电平数据采样边沿常见设备
Mode 000低电平SCLK上升沿大部分SPI Flash（如W25Q系列）
Mode 101低电平SCLK下降沿某些ADC芯片
Mode 210高电平SCLK下降沿不常见
Mode 311高电平SCLK上升沿某些RFID读卡芯片

如果你在读取SPI Flash时得到全0xFF或全0x00，十有八九是模式设错了。务必查阅芯片数据手册的“SPI Timing Diagram”章节确认。例如，Microchip的AT25系列Flash通常使用Mode 0或Mode 3。

常见误区

  • 混淆CPOL和CPHA：CPOL定义时钟空闲状态，CPHA定义采样时刻。记住口诀：“CPOL看空闲，CPHA看采样”。
  • 忽略位序（LSB First）：大部分SPI设备是MSB（最高位）先传，但有些（如某些音频Codec）是LSB先传。内核通过spi_device的mode字段的SPI_LSB_FIRST位控制。
  • 片选信号配置不当：片选默认低电平有效。如果硬件设计是反逻辑（高电平选中），需要在设备树中配置spi-cs-high属性。

对于设备驱动开发者，我们主要与三个核心结构体打交道：spi_master（由芯片厂商提供）、spi_device（代表一个SPI从设备）和spi_driver（我们编写的驱动）。

核心结构体：spi_device

这个结构体由内核根据设备树（DTS）信息创建，包含了该SPI设备的“身份信息”和“通信规则”。我们驱动中通过spi->max_speed_hz获取最大时钟频率，通过spi->mode获取工作模式。

编者提示：设备树（DTS）是描述SPI硬件连接的标准方式。如果你对DTS语法还不熟悉，建议回顾第7章《设备树DTS语法解析》，那里详细讲解了如何定义SPI节点和属性。

编写一个SPI设备驱动

我们以一个虚拟的“SPI温度传感器”为例，展示从设备树配置到驱动实现的完整流程。假设该传感器读取温度需要先发送一个命令字节0xAA，然后读取2个字节的温度数据。

首先，在设备树（如arch/arm/boot/dts/imx6ull.dtsi）中描述硬件连接：

/* 1. 启用SPI控制器 */
&ecspi1 {
    status = "okay";
    pinctrl-names = "default";
    pinctrl-0 = <&pinctrl_ecspi1>;
    cs-gpios = <&gpio4 9 GPIO_ACTIVE_LOW>; // 使用GPIO4_9作为片选

    /* 2. 添加SPI温度传感器设备节点 */
    temp_sensor: temp-sensor@0 {
        compatible = "vendor,spi-temp"; // 驱动匹配的关键字
        reg = <0>; // 片选编号0
        spi-max-frequency = <10000000>; // 最大10MHz时钟
        spi-cpol; // 设置CPOL=1
        spi-cpha; // 设置CPHA=1，即Mode 3
    };
};

接着，编写对应的SPI设备驱动。核心是定义spi_driver，并实现probe函数。

// 必要的头文件
#include <linux/spi/spi.h>
#include <linux/module.h>

static int spi_temp_read(struct spi_device *spi)
{
    int ret;
    u8 cmd = 0xAA;
    u8 rx_buf[2] = {0};
    struct spi_transfer t[] = {
        {
            .tx_buf = &cmd,
            .len = 1,
        },
        {
            .rx_buf = rx_buf,
            .len = 2,
        },
    };
    struct spi_message m;

    spi_message_init(&m);
    spi_message_add_tail(&t[0], &m); // 添加发送阶段
    spi_message_add_tail(&t[1], &m); // 添加接收阶段
    ret = spi_sync(spi, &m); // 同步传输

    if (ret == 0) {
        // 假设数据高字节在前，单位为0.1摄氏度
        int temp = (rx_buf[0] << 8) | rx_buf[1];
        printk(KERN_INFO "Temperature: %d.%d C\n", temp/10, temp%10);
    }
    return ret;
}

static int spi_temp_probe(struct spi_device *spi)
{
    printk(KERN_INFO "SPI Temp Sensor detected, max speed: %d Hz\n",
           spi->max_speed_hz);
    // 这里可以初始化设备，创建字符设备或sysfs节点等
    return spi_temp_read(spi); // 测试读取一次
}

static const struct of_device_id spi_temp_of_match[] = {
    { .compatible = "vendor,spi-temp", }, // 与设备树中的compatible匹配
    {},
};
MODULE_DEVICE_TABLE(of, spi_temp_of_match);

static struct spi_driver spi_temp_driver = {
    .driver = {
        .name = "spi-temp-sensor",
        .of_match_table = spi_temp_of_match,
    },
    .probe = spi_temp_probe,
};

module_spi_driver(spi_temp_driver); // 简化注册和注销

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("SPI Temperature Sensor Driver");

代码中的spi_message和spi_transfer是组织一次复杂SPI通信的利器。一个spi_message可以包含多个spi_transfer，它们会背靠背连续执行，期间片选保持有效。这完美处理了“先发命令，再读数据”这种常见操作。

SPI Flash驱动

  • 使用spi_mem框架（Linux 4.17+）
  • 需实现->exec_op()操作集
  • 典型芯片：W25Q128JV（16MB）

SPI LCD屏幕驱动

  • 通常基于FrameBuffer或DRM框架
  • 传输大量像素数据，需启用DMA
  • 注意初始化序列（MIPI DCS命令）

SPI传感器驱动

  • 可集成到IIO子系统（见第18章）
  • 周期性读取，可能用内核定时器
  • 典型芯片：LIS3DH加速度计

SPI音频Codec

  • 集成到ALSA框架（见第31章）
  • 配置寄存器多，注意位序（LSB）
  • 典型芯片：WM8731

调试技巧与性能优化

当SPI通信失败时，不要盲目修改代码。按以下步骤排查：

  1. 检查硬件连接：用万用表测量电源、地线，确认SCLK、MOSI、MISO、CS四根信号线已连通。
  2. 确认设备树配置：使用cat /proc/device-tree/soc/spi@xxx/status查看控制器状态是否为“okay”。使用ls /sys/bus/spi/devices/查看设备是否成功注册。
  3. 启用内核调试信息：编译内核时打开CONFIG_SPI_DEBUG，驱动中添加#define DEBUG，可以打印详细的传输日志。
  4. 使用逻辑分析仪：这是最直接的手段。抓取SCLK、MOSI、MISO波形，检查时钟频率、模式、数据内容是否与预期一致。

对于高速SPI设备（如Flash），性能优化至关重要：

  • 启用DMA：当传输数据长度超过spi_master->dma_alignment（通常为4字节）时，内核会自动尝试使用DMA。确保控制器支持并在设备树中配置了DMA通道。
  • 调整消息队列：对于实时性要求高的场景，可以调整spi_master->queue_size。
  • 使用spi_async：异步传输不阻塞当前进程，适合在中断上下文或需要高吞吐的场景中使用。

本章小结

  • SPI是一种四线制高速全双工总线，核心是SCLK、MOSI、MISO、CS四根信号线，其工作模式由CPOL和CPHA组合决定，共四种（Mode 0-3）。
  • Linux SPI驱动框架分为控制器驱动（spi_master）和设备驱动（spi_driver）。设备驱动开发者主要通过spi_device、spi_transfer和spi_message三个结构体与SPI核心层交互。
  • 编写SPI设备驱动的标准流程：在设备树中正确描述设备节点（compatible、reg、频率、模式）→ 实现spi_driver，并在probe函数中使用spi_sync等API进行数据传输。
  • 调试SPI驱动应遵循从硬件连接到软件配置的排查顺序，逻辑分析仪是验证通信时序的终极工具；对于高速设备，考虑启用DMA传输以提升性能。
