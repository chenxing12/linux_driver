核心概念USB设备描述符URB

想象一下你家的总电闸和各个电器。总电闸（USB主机控制器）统一管理电力（数据流），而每个电器（USB设备）都有自己独特的插头和用电需求。Linux内核的USB子系统就是这样一个“智能管家”，它负责识别插上的“电器”，并为它们分配合适的“插座”（驱动）。本章，我们就来学习如何为一种新“电器”编写说明书，也就是USB设备驱动。

USB驱动的三层架构

与I2C、SPI等总线驱动类似，USB驱动也采用分层架构，但更为复杂。整个USB子系统由三大部分构成，它们协同工作，将硬件操作抽象成驱动开发者可以调用的标准接口。

USB设备驱动
↓
USB核心层
↓
主机控制器驱动 (HCD)
↓
UHCI / OHCI / EHCI / xHCI
→
物理USB总线

最底层是主机控制器驱动（HCD），它直接操作USB主机控制器硬件芯片，如Intel平台的xHCI。中间是USB核心层，它提供了一套完整的API和框架，管理所有USB设备和驱动，处理通用的USB协议。我们开发者工作在顶层，即USB设备驱动，我们只需关心自己的设备如何通过核心层提供的接口进行通信。

💡 核心概念：USB设备描述符

生活类比：就像每个人的身份证记录了姓名、性别、出生地等唯一信息，USB设备一插入，就必须向主机“出示”自己的“身份证”——设备描述符。技术定义：设备描述符是一系列标准化的数据结构，包含了设备的厂商ID（idVendor）、产品ID（idProduct）、设备类（bDeviceClass）、配置信息等，是内核用来识别和匹配驱动的主要依据。

关键数据结构与驱动框架

编写一个USB设备驱动，核心是填充一个struct usb_driver结构体，并实现几个关键的回调函数。下面这个表格清晰地展示了驱动的主要组成部分及其作用。

数据结构/函数作用是否必须
struct usb_driver驱动的主结构体，包含驱动名、ID表、探测/断开函数指针是
.probe(struct usb_interface *intf, const struct usb_device_id *id)当内核发现匹配的设备时调用，进行设备初始化和资源分配是
.disconnect(struct usb_interface *intf)设备拔出或驱动卸载时调用，进行资源释放是
struct usb_device_id 数组驱动支持的设备ID列表，用于匹配是
.id_table指向上述ID表的指针是
.suspend / .resume电源管理回调，处理设备休眠与唤醒否（建议实现）

一个最简驱动框架的代码骨架如下。我在第一个USB键盘驱动项目中，就是因为忘了在MODULE_DEVICE_TABLE宏里声明支持的设备，导致驱动永远无法自动加载。

// 必要的头文件
#include <linux/module.h>
#include <linux/usb.h>

// 1. 定义设备ID表：支持特定厂商和产品的设备
static const struct usb_device_id my_usb_id_table[] = {
    { USB_DEVICE(0x1234, 0x5678) }, // 厂商ID 0x1234, 产品ID 0x5678
    { } /* 终止项 */
};
MODULE_DEVICE_TABLE(usb, my_usb_id_table);

// 2. 实现probe函数
static int my_usb_probe(struct usb_interface *intf, const struct usb_device_id *id) {
    struct usb_device *udev = interface_to_usbdev(intf);
    dev_info(&intf->dev, "设备已连接 (VID:PID=%04x:%04x)\n",
             le16_to_cpu(udev->descriptor.idVendor),
             le16_to_cpu(udev->descriptor.idProduct));
    // 在这里分配私有数据结构、注册字符设备等
    return 0; // 返回0表示成功
}

// 3. 实现disconnect函数
static void my_usb_disconnect(struct usb_interface *intf) {
    dev_info(&intf->dev, "设备已断开\n");
    // 在这里释放probe中分配的资源
}

// 4. 定义usb_driver结构体
static struct usb_driver my_usb_driver = {
    .name = "my_usb_drv",
    .id_table = my_usb_id_table,
    .probe = my_usb_probe,
    .disconnect = my_usb_disconnect,
};

// 5. 模块的初始化与退出函数
static int __init my_usb_init(void) {
    return usb_register(&my_usb_driver);
}
static void __exit my_usb_exit(void) {
    usb_deregister(&my_usb_driver);
}

module_init(my_usb_init);
module_exit(my_usb_exit);
MODULE_LICENSE("GPL");

⚠️ 常见误区

  • 混淆struct usb_device和struct usb_interface： probe和disconnect的参数是usb_interface，它代表设备的一个功能接口。一个物理USB设备（如带麦克风的摄像头）可能包含多个接口。通过interface_to_usbdev()宏可以从接口获取到对应的usb_device。
  • ID表格式错误： 数组最后必须有一个全零的终止项{ }，否则内核在遍历时会越界崩溃。
  • 忘记处理字节序： USB描述符中的多字节数据（如idVendor）是小端格式，而CPU可能是大端。必须使用le16_to_cpu()等函数进行转换后再使用，否则读出的ID值是错的。

数据传输的核心：URB

与字符设备直接使用read/write不同，USB通信的基本单位是URB（USB Request Block）。你可以把URB理解为一个封装好的“数据包裹”，里面包含了收件人地址（端点地址）、包裹内容（数据缓冲区）、操作类型（读、写、控制）以及一个“回执”回调函数。

💡 核心概念：URB

生活类比： 就像你叫一个快递（URB），需要填写收件人（端点地址）、包裹（数据）、要求（读或写），并留下电话（回调函数）以便收到取件或送达通知。技术定义： URB是Linux USB驱动中用于发起一次异步USB传输请求的数据结构。它管理着传输的所有细节，包括目标端点、数据缓冲区、传输类型和完成后的回调函数。

使用URB进行批量传输（Bulk Transfer）的典型流程如下，这是USB大容量存储设备（如U盘）最常用的方式：

1. 创建并初始化URB
→
2. 提交URB到USB核心
→
3. 等待或异步处理完成
→
4. 检查状态并释放URB

下面是一个同步批量读写的代码示例。实际开发中，对于高速设备，我们更常用异步方式以避免阻塞。

// 同步批量读取示例
static int bulk_read_sync(struct usb_device *udev, unsigned char endpoint,
                         void *data, int len, int timeout) {
    int actual_length;
    // usb_bulk_msg 是核心层提供的同步API，内部封装了URB的创建、提交和等待
    int retval = usb_bulk_msg(udev,
                              usb_rcvbulkpipe(udev, endpoint), // 构造输入管道
                              data, len,
                              &actual_length, // 实际读取的字节数
                              timeout); // 超时（毫秒）
    if (retval)
        dev_err(&udev->dev, "批量读取错误: %d\n", retval);
    return retval;
}

🔑 关键点：端点和管道

  • 端点（Endpoint）：USB设备上的一个数据收发“门户”，每个端点有唯一的地址和方向（IN-设备到主机，OUT-主机到设备）。
  • 管道（Pipe）：逻辑概念，是主机控制器到一个设备端点的连接。API中的usb_rcvbulkpipe()、usb_sndbulkpipe()就是用来根据设备地址和端点号构造管道的。

编者提示： URB的提交和完成回调是典型的异步操作。如果你对内核中如何处理异步事件感到陌生，建议回顾第27章《内核定时器与延时》和第28章《内核工作队列机制》，那里详细讲解了内核的异步执行模型和下半部机制，其思想与URB的完成回调是相通的。

USB驱动的应用场景与选择

并非所有USB设备都需要我们从零编写驱动。Linux内核已经为大量标准设备类提供了通用驱动。我们的工作通常分为以下三类：

1. 使用内核已有驱动

  • 场景： 标准设备，如U盘（usb-storage）、USB键盘鼠标（usbhid）、USB转串口（ftdi_sio, pl2303）。
  • 做法： 确保设备ID在内核驱动的支持列表中，模块会自动加载。
  • 数据： 内核的usb-storage.ko支持超过2000种不同型号的U盘和移动硬盘。

2. 编写特定设备驱动

  • 场景： 非标准或专用设备，如特定厂家的数据采集卡、定制USB加密狗。
  • 做法： 使用本章框架，实现probe/disconnect和数据传输逻辑。
  • 数据： 需要从设备手册获取准确的VID/PID和端点信息。

3. 实现USB Gadget驱动

  • 场景： 让你的开发板（如树莓派）模拟成一个USB设备（如U盘、网卡）。
  • 做法： 使用Linux的USB Gadget框架，这是另一个方向（设备端驱动）。
  • 提示： 这涉及不同的API（如gadget API），不在本章讨论范围。

对于大多数嵌入式产品，我们遇到的是第二种场景。我曾为一个使用Cypress FX2LP（CY7C68013A）芯片的USB2.0高速数据采集卡编写驱动，其批量传输端点最大包大小为512字节，在驱动中正确配置这个参数对性能至关重要。

动手试一试

1. 查找设备信息： 将一个USB设备（如一个不常见的USB小灯或玩具）插入Linux系统，执行lsusb -v命令。在输出中找到该设备的idVendor和idProduct，并观察它的接口描述符和端点描述符。
2. 编译并加载框架驱动： 将本章的“最简驱动框架”代码保存为my_usb.c，将VID/PID替换成你刚才查到的值。编写对应的Makefile，编译成模块。插入你的设备，使用sudo insmod my_usb.ko加载驱动，观察dmesg是否有你的probe函数打印的信息。拔出设备，观察disconnect信息。

检验你的理解

  1. USB设备驱动的.probe函数会在什么时候被内核调用？它的主要职责是什么？
  2. 如果有一个USB温度传感器，它通过中断端点（Interrupt Endpoint）每隔1秒上报一次数据。在驱动中，你应该使用哪个核心层函数来发起一次中断传输请求？
  3. 为什么在probe函数中读取udev->descriptor.idVendor时，需要使用le16_to_cpu()函数进行转换？

本章小结

  • USB驱动采用三层架构：设备驱动 → USB核心层 → 主机控制器驱动（HCD）。开发者聚焦于最上层的设备驱动开发。
  • 编写USB设备驱动的核心是定义usb_driver结构体，实现probe和disconnect回调，并通过usb_device_id表声明支持的设备。
  • USB数据传输的基本单位是URB（USB Request Block）。对于简单的同步传输，可以使用usb_bulk_msg()等封装函数；复杂或高速场景需直接操作URB进行异步传输。
  • 务必注意USB描述符数据的字节序问题（小端），使用le16_to_cpu()等函数进行转换。
  • 在动手前，先用lsusb工具确认设备的VID/PID和端点信息，这是驱动匹配和正确通信的基础。
