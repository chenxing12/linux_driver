核心概念Runtime PM系统休眠

凌晨三点，你被手机警报吵醒：产线测试报告显示，基于i.MX8MM处理器的智能门锁样机，在待机一周后电量耗尽，远低于设计目标。你打开串口日志，发现设备从未进入深度休眠。这个问题指向一个关键领域：内核电源管理。

Linux电源管理框架负责协调CPU、总线和外设，在系统空闲或用户操作时动态调整功耗。它不是一个单一的驱动，而是一套贯穿内核子系统的协作机制。实际项目中，电源管理没做好，产品就可能变成“电老虎”。

PM框架的两种模式与架构

内核PM主要处理两类场景：一是系统全局的休眠与唤醒（如挂起到内存S3），二是单个设备在运行时的电源状态管理。你可以把前者想象成让整栋大楼熄灯关门（系统休眠），后者则是根据房间是否有人自动开关灯和空调（设备运行时管理）。

这个架构图展示了PM框架的分层协作。用户空间的休眠指令触发核心流程，核心层依次调用总线、平台和具体设备的回调函数。我在调试RK3566平台的休眠问题时发现，一个设备的回调函数卡住，整个系统唤醒就会延迟数百毫秒。

关键点：系统休眠状态

Linux定义了多种系统全局电源状态，对应ACPI标准的S0-S5。驱动开发者最常打交道的是S3（挂起到内存）和S4（挂起到磁盘）。对于嵌入式Linux，我们通常使用mem（对应S3）和disk（对应S4）来触发。例如，在终端执行 sudo echo mem > /sys/power/state 即可尝试挂起到内存。

为设备实现Runtime PM

Runtime PM允许内核在设备空闲时自动将其挂起，有I/O请求时再恢复。这是降低设备动态功耗的关键。实现它主要分为三个步骤：初始化、提供回调函数、管理引用计数。

1. 初始化
pm_runtime_enable()
→
2. 设置空闲超时
pm_runtime_set_autosuspend_delay()
→
3. 启用自动挂起
pm_runtime_use_autosuspend()
→
4. 驱动I/O操作
pm_runtime_get_sync() / put()

下面是一个I2C传感器驱动实现Runtime PM的骨架代码：

/* 1. 定义电源管理操作集 */
static const struct dev_pm_ops sensor_pm_ops = {
    .runtime_suspend = sensor_runtime_suspend,
    .runtime_resume = sensor_runtime_resume,
    /* 系统休眠相关的回调后面会讲 */
};

/* 2. 在驱动probe函数中初始化PM */
static int sensor_probe(struct i2c_client *client) {
    struct device *dev = &client->dev;
    ...
    pm_runtime_enable(dev);
    // 设置空闲2秒后自动挂起
    pm_runtime_set_autosuspend_delay(dev, 2000);
    pm_runtime_use_autosuspend(dev);
    // 初始状态设为活跃（非挂起）
    pm_runtime_set_active(dev);
    ...
}

/* 3. 实现挂起/恢复回调 */
static int sensor_runtime_suspend(struct device *dev) {
    struct i2c_client *client = to_i2c_client(dev);
    // 关闭传感器电源，或切到最低功耗模式
    regmap_write(sensor->regmap, REG_PWR_CTL, PWR_STANDBY);
    // 可选：关闭I2C总线时钟（需与总线驱动协调）
    clk_disable_unprepare(sensor->clk);
    return 0;
}

/* 4. 在数据读写函数中管理引用计数 */
static int sensor_read_data(struct sensor_dev *s) {
    int ret;
    ret = pm_runtime_get_sync(&s->client->dev);
    if (ret < 0) {
        // 唤醒设备失败
        return ret;
    }
    // 此时设备已唤醒，可以安全进行I2C读写
    ret = i2c_smbus_read_byte_data(s->client, REG_DATA);
    // 操作完成，释放引用，可能触发自动挂起计时
    pm_runtime_mark_last_busy(&s->client->dev);
    pm_runtime_put_autosuspend(&s->client->dev);
    return ret;
}

pm_runtime_get_sync会增加设备的使用计数并确保设备唤醒，pm_runtime_put_autosuspend则减少计数并启动一个定时器，超时后内核会调用你的runtime_suspend。这里有个隐藏的坑：pm_runtime_get_sync在设备挂起失败时会返回错误码，你的驱动必须处理这个错误，而不是假设设备总是可用的。

常见误区

  • 忘记调用 `pm_runtime_mark_last_busy`：这会导致自动挂起计时器从设备初始化时开始计算，而不是从最后一次操作后，可能使设备过早进入休眠。
  • 在中断处理程序中调用PM函数：pm_runtime_get/put可能休眠，绝对不能在原子上下文中使用。如果需要，应使用其异步版本或在工作队列中处理。
  • 电源状态与时钟管理不同步：在runtime_suspend中关闭了时钟，但在runtime_resume中忘记重新使能，导致设备“睡死”。

实现系统休眠支持

当系统进入S3/S4状态时，内核会调用一套更复杂的回调链。你需要扩展之前的dev_pm_ops，实现suspend、resume及其变体。它们的调用顺序有严格规定。

回调函数调用时机典型操作可否休眠
.suspend休眠过程早期，设备仍可访问保存易失性寄存器配置可以
.suspend_late大部分子系统已休眠将设备置为最低功耗状态可以
.suspend_noirq中断已禁用，紧接在最后关闭设备时钟/电源域不可以
.resume_noirq唤醒最早阶段，中断仍禁用恢复时钟/电源域不可以
.resume_early早期唤醒阶段将设备切回可操作状态可以
.resume唤醒后期，系统基本恢复恢复寄存器配置可以

一个关键原则：noirq（“黑暗”）阶段的回调运行在中断禁用的上下文中，绝对不能进行可能休眠的操作，如互斥锁、I2C读写、内存分配等。我在为一个PCIe网卡驱动实现休眠时，曾在suspend_noirq中调用了msleep，导致内核死锁，调试了整整一天。

概念：唤醒源（Wakeup Source）

生活类比：就像你设置了闹钟（唤醒源）叫醒熟睡的自己（系统）。技术定义：唤醒源是一个能够将系统从休眠状态（如S3）触发回运行状态（S0）的硬件事件或信号。常见的有GPIO中断、RTC闹钟、网络数据包（WOL）等。

在驱动中，你需要明确告知内核你的设备是否具备唤醒能力，并在休眠前启用它。例如，一个触摸屏驱动在系统休眠后，需要能通过触摸中断唤醒系统。

启用唤醒源的代码示例如下：

static int touch_suspend(struct device *dev) {
    ...
    // 1. 告知内核本设备可唤醒系统，并分配唤醒源
    if (!device_can_wakeup(dev)) {
        device_init_wakeup(dev, true);
    }
    // 2. 配置硬件产生唤醒中断（如使能触摸中断）
    enable_irq_wake(touch->irq);
    // 3. 在休眠前启用该唤醒源
    return pm_system_wakeup_enabled(touch->ws, true);
}

static int touch_resume(struct device *dev) {
    ...
    // 唤醒后，禁用唤醒源并恢复硬件中断设置
    pm_system_wakeup_enabled(touch->ws, false);
    disable_irq_wake(touch->irq);
    ...
}

编者提示：配置唤醒源经常涉及GPIO和中断的底层操作。如果你对enable_irq_wake的底层机制或GPIO子系统不熟悉，建议回顾第5章《中断处理程序设计》和第20章《GPIO与Pinctrl子系统》的相关内容。

调试PM问题

PM问题难以复现，依赖有效的调试手段。内核提供了丰富的sysfs接口和跟踪点。

查看设备PM状态

  • /sys/devices/.../power/runtime_status: 显示 “active”, “suspended”, “error”等。
  • /sys/devices/.../power/control: 可手动写入 “on”, “auto”。
  • /sys/kernel/debug/pm_runtime/: 内核需开启CONFIG_PM_ADVANCED_DEBUG。

分析休眠唤醒失败

  • dmesg | grep -E \"(PM:|suspend|resume)\" 过滤相关日志。
  • 在/sys/power/pm_print_times写入1，打印各阶段耗时。
  • 使用ftrace跟踪pm_runtime*和suspend*函数调用流。

测量功耗

  • 硬件：使用精密电源表（如Keysight N6705）监测电流曲线。
  • 软件：通过/sys/class/power_supply/读取电池信息（如有）。
  • 目标：对比驱动实现PM前后，设备静态电流（如从15mA降至200μA）。

一个典型的调试命令流：首先 echo 1 > /sys/power/pm_print_times，然后触发休眠 echo mem > /sys/power/state，唤醒后查看dmesg，你会看到类似下面的耗时信息，帮助你定位瓶颈：

[  125.367] PM: suspend entry (deep)
[  125.371] PM: suspend devices took 0.004 seconds
[  125.872] PM: late suspend devices took 0.500 seconds # 这里耗时异常长！
[  125.873] PM: noirq suspend devices took 0.001 seconds

如果发现某个阶段耗时异常，比如上面的`late suspend`花了0.5秒，你就可以集中检查在该阶段被调用的驱动回调函数。

动手试一试

1. 为虚拟设备添加Runtime PM：基于一个简单的字符设备驱动框架（例如一个只读的虚拟传感器），为其实现Runtime PM支持。在/sys/class/下找到你的设备节点，观察runtime_status的变化，并通过连续读取设备文件触发其唤醒与挂起。

2. 模拟系统休眠：在开发板（如树莓派或i.MX6ULL）上，编写一个简单的平台驱动，实现suspend和resume回调，并在其中打印日志。然后执行echo mem > /sys/power/state让系统休眠，再通过按键唤醒，查看你的驱动日志是否被正确打印。

3. 调试唤醒源：找一个支持GPIO按键唤醒的开发板，编写一个按键驱动，将其配置为唤醒源。让系统进入休眠，然后按下该按键，检查系统是否能被成功唤醒，并在驱动日志中记录唤醒事件。

  3. 如果一个设备驱动实现了Runtime PM，但没有实现系统休眠回调（如.suspend），当系统整体进入S3状态时，这个设备会发生什么？

本章小结

  • Linux电源管理框架分为运行时PM（管理单个设备空闲状态）和系统级PM（管理全局休眠唤醒）两套协作机制。
  • 实现Runtime PM的核心是初始化、提供runtime_suspend/resume回调，并在驱动I/O路径中使用pm_runtime_get_sync/put_autosuspend正确管理设备引用计数。
  • 实现系统休眠支持需要填充dev_pm_ops中的一系列回调（如suspend, resume_noirq），并严格遵守其调用上下文（特别是noirq阶段不可休眠）。
  • 唤醒源是设备将系统从休眠状态唤醒的能力，需要在驱动中通过device_init_wakeup和enable_irq_wake等API显式配置。
  • 调试PM问题主要依靠sysfs状态查询、内核日志时间戳分析以及ftrace跟踪，目标是定位耗时的回调或失败的状态转换。
