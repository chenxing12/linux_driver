核心概念V4L2视频缓冲区

一个典型的摄像头驱动问题：用户空间调用ioctl(VIDIOC_DQBUF)获取视频帧，但驱动总是返回EAGAIN错误。这通常意味着缓冲区队列是空的，根本原因往往是驱动没有正确实现缓冲区的入队（vb2_queue）状态机。我在调试一个OV5640摄像头驱动时，就因为vb2_buffer的state字段设置错误，卡在这个问题上整整两天。

V4L2驱动框架与核心数据结构

V4L2（Video for Linux 2）是Linux内核为视频设备（输入、输出、编解码）定义的一套标准驱动接口。你可以把它想象成一个高度标准化的“视频设备插座”，无论什么品牌、什么接口（USB、MIPI-CSI、PCIe）的摄像头，只要插上这个“插座”（实现V4L2接口），上层的应用（如GStreamer、OpenCV）就能以统一的方式获取视频流。

驱动开发者的主要工作，就是填充struct video_device和struct v4l2_file_operations，并实现一系列ioctl操作的回调。核心数据结构的关系如下表所示：

数据结构作用关键成员/回调示例
struct v4l2_device代表一个V4L2设备实例，是顶级容器。name, v4l2_dev
struct video_device向用户空间暴露的设备文件接口。v4l2_dev, fops, ioctl_ops, release
struct v4l2_file_operations文件操作集（open, release, poll等）。.poll = v4l2_poll, .unlocked_ioctl = video_ioctl2
struct v4l2_ioctl_ops所有V4L2 ioctl命令的具体实现。vidioc_querycap, vidioc_s_fmt_vid_cap, vidioc_reqbufs
struct vb2_queue管理视频缓冲区的队列，核心中的核心。ops(queue_setup, buf_prepare等), mem_ops, type

关键点：video_device 与 vb2_queue 的绑定

驱动初始化时，必须将vb2_queue与video_device关联：video_device->queue = &my_vb2_queue。V4L2核心层会通过这个指针自动处理许多流I/O相关的ioctl。如果忘记绑定，VIDIOC_REQBUFS等命令将无法工作。

实现视频流捕获：从初始化到数据流

视频流捕获（Capture）是V4L2驱动最核心的功能。整个过程遵循一个清晰的状态机，我推荐使用内核的videobuf2框架来管理缓冲区，它能处理90%的复杂逻辑。

其中，第6、7步是驱动需要实现的硬件交互部分。以MIPI CSI接口的摄像头为例，当一帧图像通过DMA传输完成，硬件会产生一个中断。在中断处理函数中，你需要：

// 1. 获取当前正在使用的缓冲区
struct vb2_buffer *vb = vb2_get_buffer(vq, frame_id);
// 2. 设置实际填充的数据长度（对于压缩格式如MJPEG至关重要）
vb2_set_plane_payload(vb, 0, actual_frame_size);
// 3. 标记缓冲区数据就绪，并通知核心层
vb2_buffer_done(vb, VB2_BUF_STATE_DONE);

⚠️ 常见误区

  • 缓冲区状态混乱： 在buf_prepare回调中错误地设置vb->state = VB2_BUF_STATE_ACTIVE。正确的做法是，只有在START_STREAMING操作或硬件真正开始向该缓冲区写入时，才将其状态设为ACTIVE。
  • 忘记设置payload： 对于变长编码格式（MJPEG, H264），中断处理中必须调用vb2_set_plane_payload告知实际数据大小，否则用户空间读到的帧长度是错误的。
  • DMA缓冲区同步： 使用DMA时，在将缓冲区交给硬件前（buf_queue回调），需调用dma_sync_single_for_device()；在中断取回数据后（交给用户空间前），需调用dma_sync_single_for_cpu()，确保缓存一致性。

处理视频格式与控制

一个完整的驱动必须报告其支持的视频格式（Pixel Format）和分辨率。这通过实现vidioc_enum_fmt_vid_cap和vidioc_try_fmt_vid_cap/vidioc_s_fmt_vid_cap来完成。实际项目中，硬件支持的格式是固定的。

常见视频格式

  • YUYV (YUV 4:2:2): 未压缩，最通用，如V4L2_PIX_FMT_YUYV
  • MJPEG: 运动JPEG压缩，节省带宽，V4L2_PIX_FMT_MJPEG
  • H264: 硬件编码器输出，V4L2_PIX_FMT_H264
  • RGB565: 用于某些显示屏，V4L2_PIX_FMT_RGB565

必须实现的控制ioctl

  • VIDIOC_QUERYCAP: 报告设备能力（是Capture设备？支持Streaming？）。
  • VIDIOC_ENUM_FMT / VIDIOC_G_FMT / VIDIOC_S_FMT: 格式枚举、获取、设置。
  • VIDIOC_REQBUFS / QBUF / DQBUF / STREAMON/OFF: 流I/O核心操作。
  • VIDIOC_QUERYCTRL / G_CTRL / S_CTRL: 曝光、增益、白平衡等相机控制。

对于相机控制（如曝光时间、增益），你需要定义并注册一个struct v4l2_ctrl数组。V4L2控制框架会自动在/dev/v4l-subdev*或通过主设备的ioctl暴露这些参数。例如，为OV5640设置曝光范围：

static const struct v4l2_ctrl_config ov5640_ctrls[] = {
    {
        .ops = &ov5640_ctrl_ops,
        .id = V4L2_CID_EXPOSURE_ABSOLUTE,
        .name = "Exposure",
        .type = V4L2_CTRL_TYPE_INTEGER,
        .min = 1,
        .max = 65535, // OV5640的实际寄存器范围
        .step = 1,
        .def = 1000,
    },
    // ... 更多控制项
};

编者提示： 视频驱动中频繁涉及中断处理和DMA内存操作。如果你对内核中断处理流程或DMA API的使用细节感到模糊，强烈建议回顾第5章 中断处理程序设计和第21章 DMA引擎驱动中的实战示例，这对理解数据流的底层传输机制至关重要。

实战：一个简易摄像头驱动骨架

下面是一个基于videobuf2和平台设备驱动的简易摄像头驱动骨架，省略了具体的硬件寄存器操作，但展示了框架搭建的关键部分。

// 1. 定义驱动私有结构体，包含所有核心对象
struct my_camera_dev {
    struct v4l2_device v4l2_dev;
    struct video_device vdev;
    struct vb2_queue queue;
    struct mutex lock; // 保护并发访问
    void __iomem *reg_base;
    int irq;
};

// 2. 实现 vb2_queue 的操作集
static const struct vb2_ops my_vb2_ops = {
    .queue_setup = my_queue_setup, // 缓冲区数量/大小协商
    .buf_prepare = my_buf_prepare, // 缓冲区准备（地址映射）
    .buf_queue = my_buf_queue,     // 缓冲区入队，可在此启动DMA
    .start_streaming = my_start_streaming,
    .stop_streaming = my_stop_streaming,
    .wait_prepare = vb2_ops_wait_prepare,
    .wait_finish = vb2_ops_wait_finish,
};

// 3. 在 probe 函数中进行初始化
static int my_camera_probe(struct platform_device *pdev)
{
    struct my_camera_dev *dev;
    // ... 分配内存、映射寄存器、申请中断

    // 初始化 v4l2_device
    v4l2_device_register(&pdev->dev, &dev->v4l2_dev);

    // 初始化 vb2_queue
    dev->queue.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    dev->queue.io_modes = VB2_MMAP | VB2_DMABUF; // 支持的模式
    dev->queue.ops = &my_vb2_ops;
    dev->queue.mem_ops = &vb2_dma_contig_memops; // 使用DMA连续内存
    dev->queue.drv_priv = dev; // 私有数据指针
    dev->queue.lock = &dev->lock;
    vb2_queue_init(&dev->queue);

    // 初始化 video_device
    dev->vdev = video_device_alloc();
    dev->vdev->v4l2_dev = &dev->v4l2_dev;
    dev->vdev->queue = &dev->queue; // 关键绑定！
    dev->vdev->fops = &my_fops;
    dev->vdev->ioctl_ops = &my_ioctl_ops;
    strscpy(dev->vdev->name, "my-camera", sizeof(dev->vdev->name));
    video_set_drvdata(dev->vdev, dev);
    video_register_device(dev->vdev, VFL_TYPE_VIDEO, -1);
    return 0;
}

概念：MMAP vs. USERPTR

MMAP：应用通过mmap()系统调用直接将内核分配的DMA缓冲区映射到用户空间进行访问。这是性能最高、最推荐的方式，因为避免了数据拷贝。驱动需要实现vb2_mem_ops（通常用vb2_dma_contig_memops）。
USERPTR：应用提供自己分配的用户空间内存地址。驱动需要将这些地址转换为物理地址供DMA使用，过程复杂且容易出错，除非有特殊需求，否则不要使用。我在一个需要与特定图形库共享内存的项目中才被迫用过一次。

动手试一试

1. 环境搭建：使用QEMU模拟一个V4L2测试设备。在Linux内核源码中，启用CONFIG_V4L_TEST_DRIVERS和CONFIG_VIDEO_VIVID，编译并启动。你将获得一个虚拟视频设备/dev/video0，它完美实现了V4L2接口，是学习的最佳参考。
2. 代码跟踪：使用v4l2-ctl --all命令查询/dev/video0的能力和格式。然后，编写一个简单的C程序，调用VIDIOC_REQBUFS, VIDIOC_QBUF, VIDIOC_STREAMON，并尝试用VIDIOC_DQBUF获取一帧数据。
3. 驱动实验：基于上面的骨架代码，尝试在start_streaming回调中模拟产生数据。例如，启动一个内核定时器，每33毫秒（模拟30fps）调用一次vb2_buffer_done，并在缓冲区中填充一个简单的颜色渐变图案（如递增的灰度值）。观察用户空间程序是否能收到连续的“视频帧”。

检验你的理解

  1. 用户空间调用VIDIOC_DQBUF一直阻塞，可能是什么原因？从驱动角度列出两种可能性。
  2. 在vb2_queue的操作集中，buf_prepare和buf_queue回调分别应该在什么时机被调用？它们的主要职责是什么？
  3. 如果硬件一次中断表示一帧图像传输完成，驱动在中断处理函数中必须至少完成哪两个关键调用，才能让用户空间的VIDIOC_DQBUF成功返回？

本章小结

  • V4L2驱动框架以video_device、v4l2_ioctl_ops和vb2_queue为核心，videobuf2框架是管理视频缓冲区的标准且高效的选择。
  • 视频流捕获遵循“初始化->设格式->申请并映射缓冲区->启动流->硬件中断填充数据->通知应用”的固定流程，驱动需正确实现vb2_ops中的状态机回调。
  • 必须正确处理视频格式（Pixel Format）和相机控制（Control），通过实现对应的ioctl并注册v4l2_ctrl来暴露硬件能力。
  • MMAP是首选的缓冲区交换模式，驱动需关注DMA缓存一致性问题；调试时，虚拟驱动vivid和工具v4l2-ctl是得力助手。
