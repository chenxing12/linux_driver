核心概念watchdog_device喂狗(keepalive)

系统在野外运行，程序跑飞了怎么办？看门狗就是那个在角落里默默计时的“监工”，如果你不按时“喂它”（发送心跳信号），它就会认为系统已死，并触发硬件复位。在Linux中，内核通过一个统一的子系统来管理这些脾气暴躁的硬件“监工”。

核心概念：看门狗定时器 (Watchdog Timer)

生活类比： 就像你养了一只必须每隔15分钟就抚摸一下的猫，如果你忘了，它就会打翻你的水杯（系统复位）。技术定义： 看门狗是一个独立的硬件定时器，需要软件周期性地对其“喂狗”（重置计时），若超时未喂，则产生复位信号重启整个系统，用于从软件死锁或崩溃中恢复。

17.1 看门狗子系统架构

Linux的看门狗子系统抽象得很好，驱动开发者主要与struct watchdog_device和struct watchdog_ops打交道。整个数据流和控制流遵循标准的“驱动-核心-用户”模型。

这个架构确保了无论底层是i.MX6ULL的内置WDOG，还是通过I2C连接的独立看门狗芯片如MAX6370，对上层应用提供的接口都是统一的/dev/watchdog字符设备。

17.2 关键数据结构与驱动填充

编写驱动，本质上是填充几个关键的结构体。首先，你需要定义一个watchdog_device，它描述了看门狗设备本身。

struct watchdog_device {
    const struct watchdog_info *info;
    const struct watchdog_ops *ops;
    unsigned int timeout;
    unsigned int min_timeout;
    unsigned int max_timeout;
    struct device *parent;
    // ... 其他内部字段
};

其中，watchdog_info定义了设备的身份和能力标志。

字段类型说明与典型值
identitychar[32]设备标识字符串，如 "i.MX6UL Watchdog"
firmware_versionu32固件版本，通常为0
optionsu32能力标志位，至关重要

options字段的常用标志如下，它们直接影响用户空间能做什么：

  • WDIOF_SETTIMEOUT: 允许动态设置超时时间。
  • WDIOF_MAGICCLOSE: 支持“魔术关闭”（Magic Close），即要求关闭/dev/watchdog前必须先写入特定字符‘V’，否则驱动会认为关闭是误操作并继续让看门狗运行。这是一个重要的安全特性。
  • WDIOF_KEEPALIVEPING: 设备支持喂狗操作。

接下来是watchdog_ops，这是驱动真正的“肌肉”，包含了所有操作硬件的函数指针。你至少需要实现start, stop, ping, set_timeout。

static const struct watchdog_ops imx_wdt_ops = {
    .owner = THIS_MODULE,
    .start = imx_wdt_start,
    .stop = imx_wdt_stop,
    .ping = imx_wdt_ping, // 喂狗
    .set_timeout = imx_wdt_set_timeout,
    // .get_timeleft 可选，用于读取剩余时间
};

关键点：喂狗(ping)的实现

喂狗函数ping是驱动中最频繁调用的操作。它的实现因硬件而异：对于STM32的独立看门狗(IWDG)，是向键值寄存器IWDG_KR写入0xAAAA；对于NXP的芯片，可能是向某个控制寄存器的特定位写1。我在一个项目中曾因为喂狗时序不对，导致系统在负载高时意外复位，后来发现是操作寄存器前缺少必要的读-修改-写屏障。

17.3 驱动初始化与注册流程

驱动的初始化流程是标准化的。对于设备树（Device Tree）描述的现代SoC，你需要在.probe函数中完成所有设置。

platform_driver.probe
→
分配并填充 watchdog_device
→
设置超时范围 (min/max)
→
watchdog_init_timeout
→
watchdog_register_device

这里有个细节：watchdog_init_timeout函数会按顺序尝试确定超时值：1) 模块参数，2) 设备树属性（timeout-sec），3) 驱动默认值。这保证了配置的灵活性。

static int my_wdt_probe(struct platform_device *pdev) {
    struct my_wdt_drvdata *drvdata;
    struct watchdog_device *wdd;
    int ret;

    drvdata = devm_kzalloc(&pdev->dev, sizeof(*drvdata), GFP_KERNEL);
    wdd = &drvdata->wdd;

    // 1. 填充watchdog_device
    wdd->info = &my_watchdog_info;
    wdd->ops = &my_watchdog_ops;
    wdd->min_timeout = 1; // 最小1秒
    wdd->max_timeout = 128; // 最大128秒，取决于硬件分频器
    wdd->parent = &pdev->dev;

    // 2. 初始化超时（从设备树读取）
    watchdog_init_timeout(wdd, 0, &pdev->dev); // 0表示不使用模块参数

    // 3. 注册看门狗设备
    ret = devm_watchdog_register_device(&pdev->dev, wdd);
    if (ret)
        return ret;

    platform_set_drvdata(pdev, drvdata);
    dev_info(&pdev->dev, "My Watchdog registered (timeout=%d sec)\n", wdd->timeout);
    return 0;
}

⚠️ 常见误区与踩坑点

  • 混淆nowayout与magic close： nowayout是一个模块参数或设备树属性，若为1，则看门狗一旦启动就永远无法通过软件关闭，即使进程崩溃也必须复位。而WDIOF_MAGICCLOSE只是一种安全的关闭协议。很多新手以为设置了magic close就高枕无忧，实际上如果内核panic，任何软件关闭机制都会失效。
  • 喂狗时机在中断上下文： 切忌在硬件中断服务程序（ISR）中直接喂狗。如果是因为某个中断风暴导致系统卡死，在ISR中喂狗会让看门狗失效。喂狗应该由内核的工作队列或定时器触发，这些机制在系统软锁定时也会停止。
  • 未处理时钟源变化： 对于从系统时钟分频的看门狗，如果系统进入低功耗模式改变了时钟源，看门狗的计数频率可能变化，导致实际超时时间与设置不符。在驱动的suspend/resume回调中需要重新配置超时。

17.4 用户空间控制与喂狗策略

驱动注册后，会在/dev下生成设备节点，通常是/dev/watchdog或/dev/watchdog0。用户空间通过标准的文件操作来控制它。

基本喂狗

  • 向设备写入任意数据（如一个空格字符）。
  • 命令：echo V > /dev/watchdog
  • 这是最简单的“心跳”。

设置超时

  • 使用ioctl命令WDIOC_SETTIMEOUT。
  • 工具：watchdog -t 30 /dev/watchdog (设置30秒超时)。
  • 驱动需支持WDIOF_SETTIMEOUT。

安全关闭

  • 先写入字符‘V’，再关闭文件描述符。
  • 命令序列：echo V > /dev/watchdog; exec 3>&-
  • 防止误关闭导致系统失去保护。

在实际产品中，我们通常使用systemd的watchdog.service或独立的看门狗守护进程（如watchdogd）来负责喂狗。这些守护进程会监控系统关键服务，如果服务异常，它们会主动停止喂狗，触发系统复位，这比简单的定时喂狗更智能。

编者提示： 看门狗驱动中经常需要使用定时器或工作队列来实现周期性的喂狗任务。关于内核定时器的高精度模式（hrtimer）和工作队列（workqueue）的详细使用方法，你可以参考本教程的第27章 内核定时器与延时和第28章 内核工作队列机制。

17.5 设备树绑定

为你的看门狗驱动添加设备树支持是标准做法。这允许在板级.dts文件中灵活配置看门狗属性。

// 在板级.dts文件中的示例：
&watchdog0 {
    compatible = "vendor,my-watchdog";
    reg = <0x40000000 0x1000>;
    interrupts = <GIC_SPI 50 IRQ_TYPE_LEVEL_HIGH>;
    clocks = <&clk_apb>;
    timeout-sec = <30>; // 默认超时30秒
    nowayout; // 可选，表示禁止关闭
};

在驱动中，你需要使用of_property_read_u32()等函数来解析这些属性。对于nowayout，内核提供了辅助函数of_property_read_bool()。

动手试一试

假设你手头有一块基于NXP i.MX6ULL的开发板，其内置看门狗设备树节点已配置。

  1. 加载驱动并观察： 编译并加载内核自带的imx2_wdt.c驱动（位于drivers/watchdog/）。使用dmesg | grep watchdog查看注册信息，确认超时时间。
  2. 测试喂狗与复位： 使用命令watchdog -t 10 /dev/watchdog启动看门狗并设置10秒超时。等待超过10秒，观察系统是否自动复位。
  3. 探究Magic Close： 编写一个简单的C程序：打开/dev/watchdog，启动看门狗，然后直接调用close()。观察系统行为。然后修改程序，在close()前先写入字符‘V’，再次观察。

检验你的理解

  1. 看门狗驱动中的nowayout属性和WDIOF_MAGICCLOSE选项标志有何本质区别？
  2. 如果一个看门狗硬件的超时时间只能通过硬件分频器设置为几个固定值（如1.6s, 6.4s, 25.6s），你如何在set_timeout回调函数中处理用户请求的任意超时值（如10秒）？
  3. 为什么说在用户空间使用一个简单的while sleep(1); echo > /dev/watchdog循环来喂狗不是一个健壮的方案？

本章小结

  • Linux看门狗子系统提供了从驱动到应用的统一框架，核心是watchdog_device和watchdog_ops结构体。
  • 编写驱动需至少实现start、stop、ping（喂狗）、set_timeout操作，并正确设置info->options能力标志。
  • 务必理解nowayout（不可关闭）与WDIOF_MAGICCLOSE（安全关闭协议）的区别，这是系统可靠性的关键。
  • 用户空间通过/dev/watchdog设备文件控制看门狗，生产环境应使用能监控系统状态的守护进程进行智能喂狗。
  • 通过设备树绑定（timeout-sec, nowayout）可以灵活配置看门狗参数，提高驱动可移植性。
