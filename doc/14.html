核心概念Framebufferfb_info

想象一下，你家的窗户就是一块显示屏，窗玻璃后面贴着一张巨大的、可以随时更换的画布，这张画布就是帧缓冲（Framebuffer）。在Linux内核中，帧缓冲是一个抽象层，它为应用程序提供了一块线性的内存区域，这块内存直接对应着屏幕上每一个像素的颜色值。应用程序只需向这块内存写入数据，内核或硬件就会自动将其扫描输出到物理显示器上。

帧缓冲（Framebuffer）

生活类比：就像电影院放映机的胶片，每一帧画面都预先绘制在胶片上，放映机只是匀速播放它。
技术定义：一块在内核空间或显存中分配的、与显示设备像素一一对应的连续内存区域，用于存储单帧图像的所有像素数据。

FB驱动的核心任务，就是初始化这块“画布”（分配内存、设置参数），并告诉内核“画布”在哪里、有多大、以及如何把“画布”上的内容“贴”到窗户（显示器）上。所有复杂的硬件操作，如时序控制、信号转换，都被封装在驱动内部，对上只提供一个简单、统一的内存读写接口。

FB驱动的核心数据结构：fb_info

内核通过一个名为 struct fb_info 的结构体来管理一个帧缓冲设备。你可以把它理解为一个显示设备的“身份证”加“操作手册”。驱动开发者的主要工作就是填充这个结构体，并将其注册到内核的FB子系统中。

用户空间 (App)
↓ (write/ioctl)
FB核心层 (/dev/fbX)
↓ (调用ops)
FB驱动层 (fb_info)
↓ (配置寄存器)
硬件 (LCD控制器)

上图的架构清晰地展示了数据流向：应用写入 "/dev/fb0"，FB核心层将调用你驱动中 fb_info 注册的操作函数，最终由驱动操控硬件完成显示。下面这个表格列出了 fb_info 中你必须关注的几个关键成员：

结构体成员类型作用描述示例值/说明
varstruct fb_var_screeninfo可变屏幕参数（应用可修改）分辨率（xres, yres）、色深（bits_per_pixel）
fixstruct fb_fix_screeninfo固定屏幕参数（驱动设定）缓冲区长（smem_len）、内存起始地址（smem_start）
fbopsstruct fb_ops *驱动操作函数集合指针包含 fb_check_var, fb_set_par 等
screen_basevoid *帧缓冲内存的虚拟地址（内核空间）通过 dma_alloc_wc 或 kmalloc 分配
devicestruct device *关联的底层设备结构体通常指向平台设备 struct platform_device

关键点：色深（bits_per_pixel）与内存计算

色深决定了每个像素用多少位（bit）来表示颜色，它直接决定了帧缓冲需要多大内存。一个常见的坑是分配的内存大小不足。计算公式是：
内存大小 (bytes) = xres * yres * (bits_per_pixel / 8)
例如，对于一款800x480的RGB565屏幕（16位色深），所需内存为 800 * 480 * 2 = 768,000 字节。实际开发中，我通常会多分配一点（比如1MB）来对齐DMA边界或防止溢出。

动手写一个最简单的FB驱动

我们从一个最简化的模型开始，忽略硬件初始化，只关注如何向内核“声明”一块内存作为帧缓冲。这能帮你快速理解FB驱动的注册流程。

// 1. 包含必要的头文件
#include <linux/fb.h>
#include <linux/module.h>
#include <linux/dma-mapping.h>

static struct fb_info *my_fb_info;

// 2. 定义最简化的fb_ops（至少需要fb_fillrect）
static struct fb_ops my_fb_ops = {
    .owner = THIS_MODULE,
    .fb_fillrect = cfb_fillrect, // 使用内核提供的通用软件实现
    // 其他如fb_copyarea, fb_imageblit也指向cfb_*系列函数
};

// 3. 探测函数：分配并设置fb_info
static int my_fb_probe(struct platform_device *pdev)
{
    int ret = 0;
    void *fb_virt_addr;

    // 3.1 分配fb_info结构体
    my_fb_info = framebuffer_alloc(0, &pdev->dev);
    if (!my_fb_info) {
        return -ENOMEM;
    }

    // 3.2 设置固定参数
    strcpy(my_fb_info->fix.id, "my_simple_fb");
    my_fb_info->fix.type = FB_TYPE_PACKED_PIXELS;
    my_fb_info->fix.visual = FB_VISUAL_TRUECOLOR; // 真彩色
    my_fb_info->fix.line_length = 800 * 2; // 800像素 * 2字节(16位)
    my_fb_info->fix.smem_len = 800 * 480 * 2; // 总内存大小

    // 3.3 分配实际的帧缓冲内存（使用DMA一致性映射）
    fb_virt_addr = dma_alloc_wc(&pdev->dev, my_fb_info->fix.smem_len,
                                &my_fb_info->fix.smem_start, GFP_KERNEL);
    if (!fb_virt_addr) {
        ret = -ENOMEM;
        goto err_fb_alloc;
    }
    my_fb_info->screen_base = fb_virt_addr;

    // 3.4 设置可变参数（这里配置为800x480 RGB565）
    my_fb_info->var.xres = my_fb_info->var.xres_virtual = 800;
    my_fb_info->var.yres = my_fb_info->var.yres_virtual = 480;
    my_fb_info->var.bits_per_pixel = 16;
    // 设置RGB分量偏移（RGB565格式：R[11:5], G[5:1], B[0:4]）
    my_fb_info->var.red.offset = 11; my_fb_info->var.red.length = 5;
    my_fb_info->var.green.offset = 5; my_fb_info->var.green.length = 6;
    my_fb_info->var.blue.offset = 0; my_fb_info->var.blue.length = 5;

    // 3.5 关联操作函数集和设备
    my_fb_info->fbops = &my_fb_ops;
    my_fb_info->device = &pdev->dev;

    // 3.6 注册帧缓冲设备到内核！
    ret = register_framebuffer(my_fb_info);
    if (ret < 0) {
        goto err_dma_alloc;
    }
    return 0;

err_dma_alloc:
    dma_free_wc(&pdev->dev, my_fb_info->fix.smem_len,
                my_fb_info->screen_base, my_fb_info->fix.smem_start);
err_fb_alloc:
    framebuffer_release(my_fb_info);
    return ret;
}

代码加载后，你会看到系统多了一个 "/dev/fb0" 设备节点。虽然它还不能点亮真实的屏幕（因为缺少硬件初始化），但你已经可以通过 write 或 mmap 向这块内存写入数据了。

常见误区

  • 混淆 fix.smem_start 和 screen_base：smem_start 是物理地址（或总线地址），用于DMA；screen_base 是内核虚拟地址，用于CPU访问。使用 dma_alloc_wc 可以一次性正确获取两者。
  • 忘记设置颜色分量偏移：在 fb_var_screeninfo 中，必须根据像素格式（如RGB565， ARGB8888）正确设置 red/green/blue/transp 的 offset 和 length，否则显示颜色会完全错乱。
  • 未实现必要的 fb_ops 函数：即使你只想用最简单的内存映射，也必须至少提供 fb_fillrect, fb_copyarea, fb_imageblit 这三个函数指针，通常指向 cfb_fillrect 等内核通用实现。

连接真实硬件：配置LCD控制器

要让像素数据真正在屏幕上亮起来，驱动必须初始化LCD控制器。这通常涉及配置一组硬件寄存器，包括时序参数、像素格式、时钟频率等。这部分与具体芯片强相关，但流程是通用的。

获取平台资源
(时钟、GPIO、寄存器基址)
→
使能像素时钟
(PCLK)
→
配置时序参数
(HBP, HFP, VBP, VFP)
→
设置输出格式
(RGB顺序、数据宽度)
→
将帧缓冲地址
写入DMA寄存器
→
启动LCD控制器
和背光

以一款常见的SSD2828 RGB转MIPI桥接芯片为例，在驱动中配置800x480屏幕的典型时序参数如下（单位：像素时钟周期）：

水平时序

  • HBP (后沿): 46
  • HFP (前沿): 210
  • HSYNC (同步脉宽): 10

垂直时序

  • VBP (后沿): 23
  • VFP (前沿): 22
  • VSYNC (同步脉宽): 10

时钟与格式

  • 像素时钟: ~33MHz
  • 数据宽度: 24-bit RGB
  • 同步极性: HSYNC低有效，VSYNC低有效

这些参数通常可以在屏幕的数据手册（Datasheet）的“时序特性（Timing Characteristics）”章节找到。配置错误会导致花屏、闪屏或完全无显示。我在调试一块5英寸IPS屏时，就曾因为把HBP和HFP填反，导致图像右侧出现一条垂直黑线。

编者提示：配置LCD控制器经常需要操作GPIO（如复位引脚、背光使能）和时钟。关于GPIO子系统的详细用法，请参考第20章《GPIO与Pinctrl子系统》；关于时钟API的调用，在第6章《平台设备驱动模型》中有深入讲解。

调试与信息查看

驱动加载后，如何确认它工作正常？除了直接看屏幕，内核和用户空间提供了多个调试接口。

首先，检查内核日志 dmesg，成功的注册会打印类似信息：
[ 1.234567] graphics fb0: my_simple_fb frame buffer device

其次，使用 fbset 工具可以查看和修改当前FB的参数：

$ fbset

mode "800x480-60"
    # D: 33.000 MHz, H: 33.000 kHz, V: 60.000 Hz
    geometry 800 480 800 480 16
    timings 30303 46 210 10 23 22 10
    rgba 5/11,6/5,5/0,0/0
endmode

如果 fbset 能正确读出你设置的分辨率、色深和时序，说明驱动的基础信息注册成功了。

动手试一试

1. 虚拟FB实验：基于本章的简化代码框架，编写一个内核模块。将分辨率改为 640x480，色深改为 32位（ARGB8888）。加载模块后，使用命令 dd if=/dev/urandom of=/dev/fb0 bs=1K count=300 向帧缓冲写入随机数据，观察系统控制台（如果支持）或通过后续的FB测试工具查看效果。
2. 信息提取：在你的开发板上（或使用QEMU模拟器），运行 cat /proc/fb 和 fbset -i 命令，记录下系统现有FB设备的信息，理解每个输出字段的含义。
3. 简单绘图：写一个最小的用户空间C程序，通过 mmap 映射 "/dev/fb0"，然后将帧缓冲的中间1/4区域填充为红色（RGB565格式下红色约为 0xF800）。

检验你的理解

本章小结

  • 帧缓冲（Framebuffer）是Linux中用于图形显示的核心抽象，它为上层提供了一块与屏幕像素对应的线性内存区域。
  • 编写FB驱动的核心是填充并注册 struct fb_info 结构体，关键步骤包括：分配该结构体、设置 fix/var 参数、分配帧缓冲内存、实现必要的 fb_ops 操作函数。
  • 驱动必须根据屏幕手册正确配置LCD控制器的时序参数（如HBP, HFP, VBP, VFP）和像素格式，才能使硬件正常扫描显示。
  • 帧缓冲内存大小的计算取决于分辨率、色深和虚拟分辨率，使用 dma_alloc_wc 等DMA接口分配可以同时获得物理地址和虚拟地址。
  • 通过 dmesg、/proc/fb 和 fbset 工具可以有效地调试和验证FB驱动的工作状态。
