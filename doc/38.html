核心概念KGDBCrash Dump

printk打印的日志在控制台滚动，系统却莫名其妙死锁了；内存使用量缓慢增长，最终触发OOM（Out-Of-Memory）杀手。这些场景下，仅靠日志输出如同盲人摸象。你需要更强大的“内窥镜”和“手术刀”来深入内核现场。

本章定位

本章不讨论printk、/proc、sysfs等基础调试手段（这些在第24、25章已覆盖）。我们聚焦于高级的、需要特定工具和配置的深度调试技术，目标是让你在驱动出现最棘手问题时，有明确的工具链和排查路径。

一、 源码级调试：KGDB

KGDB允许你像调试用户态程序一样，通过GDB连接正在运行的内核，设置断点、单步执行、查看变量。这需要两台机器（开发机与目标机）通过串口或网络连接。

首先，你需要为目标内核启用KGDB编译选项。我在调试一个基于i.MX6ULL的触摸屏驱动死锁问题时，就依赖KGDB才找到了那个错误的锁获取顺序。

// 内核配置 (make menuconfig)
Kernel hacking --->
    [*] KGDB: kernel debugger --->
        [*] KGDB: use kgdb over the serial console
        [*] KGDB: internal test suite // 可选，用于测试

配置并编译内核后，在目标机的内核启动参数中添加：kgdboc=ttyS0,115200 kgdbwait。这样内核启动早期就会暂停，等待GDB连接。在开发主机上，使用vmlinux（带调试符号的内核镜像）启动GDB：

$ gdb ./vmlinux
(gdb) set serial baud 115200
(gdb) target remote /dev/ttyUSB0 // 根据实际串口设备调整
(gdb) break my_driver_ioctl // 在你的驱动函数设断点
(gdb) continue

常见误区

  • 符号文件不匹配：GDB加载的vmlinux必须与目标机运行的内核完全一致（同一编译产出）。否则地址和符号对不上，调试毫无意义。
  • 串口配置错误：确保开发主机和目标机的串口波特率、数据位、停止位、流控设置完全一致。115200 8N1是最常用配置。
  • 中断影响：在断点处，整个系统（包括调度器）都会暂停。不要在生产环境使用KGDB，它会导致业务中断。

二、 事后分析：Kdump与Crash工具

当内核发生严重错误（如panic、oops）导致崩溃时，KGDB可能来不及连接。这时需要Kdump机制：在第一内核（生产内核）崩溃瞬间，由Kexec快速启动一个第二内核（捕获内核），将第一内核的内存完整转储（Dump）到文件。这就像飞机失事后的“黑匣子”，保存了崩溃瞬间的完整现场。

（预留内存）
→

→

→

配置Kdump的关键是预留内存。对于内存为1GB的树莓派4B，我通常在/etc/default/grub的启动参数中添加crashkernel=128M，为捕获内核预留128MB内存。安装相关工具后，触发一个崩溃，转储文件通常位于/var/crash/目录下。

拿到vmcore转储文件后，使用crash工具进行分析。你需要准备与崩溃内核对应的vmlinux文件。

$ crash /usr/lib/debug/lib/modules/$(uname -r)/vmlinux /var/crash/vmcore

crash> log // 查看崩溃时的内核日志
crash> bt // 查看崩溃进程的调用栈回溯
crash> ps // 查看崩溃瞬间所有进程状态
crash> kmem -i // 查看内核内存使用概况
crash> struct my_device -x // 查看你自定义结构体的内容
Crash工具常用命令解析
命令功能实战输出线索示例
bt回溯调用栈锁定崩溃函数及行号（如my_driver_write+0x45）
ps进程状态查看是否有进程处于`D`(不可中断睡眠)状态，判断死锁
kmem -sSlab缓存统计发现某个缓存（如`kmalloc-512`）对象数量异常多，提示内存泄漏
search -t内存搜索搜索一个已知的魔数或字符串，定位相关数据结构
irq -i中断统计查看某个IRQ是否被疯狂触发，导致软中断占用过高

编者提示：理解crash工具分析的内存信息，需要扎实的内核数据结构基础。如果你对`struct task_struct`、`vm_area_struct`等核心结构感到陌生，建议回顾第4章《内存管理与分配》和第26章《设备模型与kobject》。

三、 动态追踪：Ftrace与SystemTap

对于性能瓶颈、偶发性问题，你需要在不重启系统、不修改代码的情况下动态插入探针。这就像给运行中的汽车安装实时传感器。

Ftrace（内核内置）

  • 场景：函数调用耗时分析、中断延迟测量、调度器行为跟踪。
  • 要点：无需额外安装，通过/sys/kernel/debug/tracing接口控制。
  • 实战：排查一个SPI传输偶发超时，用`function_graph`跟踪器发现某处`spin_lock`持有时间过长。

SystemTap（脚本化探针）

  • 场景：复杂逻辑的统计、用户态与内核态关联分析、自定义聚合数据。
  • 要点：需要安装kernel-debuginfo包，编写.stp脚本。
  • 实战：统计某个ioctl命令在不同参数下的调用频率和耗时分布。

Ftrace使用示例：跟踪`my_device_open`函数及其子函数的调用情况。

# 切换到跟踪目录
$ cd /sys/kernel/debug/tracing
# 设置当前跟踪器为 function_graph
$ echo function_graph > current_tracer
# 设置要跟踪的函数
$ echo my_device_open > set_graph_function
# 开始跟踪
$ echo 1 > tracing_on
# ... 执行触发操作 ...
# 停止跟踪并查看结果
$ echo 0 > tracing_on
$ cat trace | head -100

SystemTap脚本示例：监控`my_driver_ioctl`的调用参数和返回值。实际开发中，我常用它来验证驱动在压力下的行为是否符合预期。

// monitor_ioctl.stp
probe kernel.function(“my_driver_ioctl”).call {
    printf(“ioctl called: cmd=%d, arg=0x%x, pid=%d\n”, $cmd, $arg, pid())
}
probe kernel.function(“my_driver_ioctl”).return {
    printf(“ioctl returned: ret=%d, time=%d ns\n”, $return, gettimeofday_ns() - @entry(gettimeofday_ns()))
}

运行脚本：stap -v monitor_ioctl.stp。你会看到实时的调用流。

工具选型速查

KGDB：适用于能稳定复现的复杂逻辑错误，需要源码级单步洞察。Kdump/Crash：适用于分析已发生的、致命的系统崩溃，进行事后 forensic（取证）。Ftrace/SystemTap：适用于分析性能、统计、以及难以复现的偶发问题，进行动态观测。

在实际项目中，我通常的排查顺序是：先看日志（printk），偶发问题用Ftrace抓取，复杂崩溃配置Kdump保留现场，最后对于需要深入特定代码路径的逻辑错误，才架设KGDB。

动手试一试

  1. 配置并触发一次Kdump：在一台虚拟机（如预留2GB内存）中，安装kdump-tools，配置crashkernel=256M。通过echo c > /proc/sysrq-trigger手动触发内核崩溃。检查/var/crash/目录下是否生成vmcore文件。
  2. 使用Ftrace跟踪一次系统调用：打开/sys/kernel/debug/tracing/trace，将current_tracer设置为function，清空trace，然后执行echo 1 > tracing_on。在另一个终端执行ls命令，然后停止跟踪。查看trace文件，理解内核函数调用的风暴。
  3. （可选）编写一个SystemTap脚本：统计vfs_read函数在5秒内的调用次数。提示：使用probe timer.s(5)和全局计数器。

检验你的理解

  • KGDB调试时，为什么开发主机上的GDB必须加载vmlinux文件，而不是zImage或bzImage？
  • 当系统因为某个驱动模块的内存泄漏而缓慢耗尽内存，最终被OOM Killer杀死进程时，使用Kdump还能捕获到有效的崩溃转储吗？为什么？
  • 使用Ftrace的`function_graph`跟踪器时，发现某个内核函数后面显示了“+”符号但并没有展开子函数调用，可能的原因是什么？

本章小结

  • KGDB提供源码级调试能力，通过GDB远程连接运行中的内核，适用于复杂逻辑错误的深入分析，但需要串口/网络连接和匹配的调试符号。
  • Kdump/Crash是内核崩溃的“黑匣子”分析工具，Kdump负责捕获转储，Crash工具用于事后分析转储文件中的内存、进程、堆栈等完整现场信息。
  • Ftrace和SystemTap实现动态追踪，无需重启即可在内核关键路径插入探针，分别适用于内置的灵活跟踪和脚本化的复杂数据收集，是分析性能与偶发问题的利器。
  • 工具选择取决于问题阶段：动态分析用Ftrace/SystemTap，事后分析用Crash，主动深入调试用KGDB，三者构成驱动调试的完整工具箱。
