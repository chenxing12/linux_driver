核心概念IIO设备通道(Channel)

凌晨三点，产线告警：一个基于STM32MP157的工控板，其用于监测电压的ADC读数突然全部归零。你检查硬件，供电正常；用示波器看模拟信号，波形完好。最终定位到驱动：工程师直接操作寄存器读取ADC，但在一次内核升级后，寄存器映射地址变了。如果当初使用了IIO框架，这个驱动将与具体硬件寄存器解耦，根本不会出这个问题。

IIO（Industrial I/O）子系统是Linux内核为各类模拟/数字传感器（ADC、DAC、陀螺仪、加速度计、光传感器等）建立的标准驱动框架。它统一了数据访问接口（sysfs和字符设备），并提供了丰富的内置缓冲区和事件处理机制，让驱动开发者能专注于硬件交互，而不是重复造轮子。

生活类比：超市的标准化货架

想象一个超市，每种商品（传感器数据）都被放在标准尺寸的货架（IIO通道）上，并贴上统一的条形码（属性名称）。顾客（用户程序）无论想买酱油还是食盐，都通过同样的扫码枪（sysfs接口或read系统调用）来获取信息，而不需要知道货物是如何从仓库搬出来的。IIO就是这个“标准化货架系统”，它定义了传感器数据如何被命名、组织和访问。

技术定义：IIO是一个内核子系统，它通过struct iio_dev抽象一个传感器设备，用struct iio_chan_spec定义数据通道，并在/sys/bus/iio/devices/下导出统一的属性文件，或在/dev/iio:deviceX提供字符设备接口进行数据读写。

IIO子系统架构全景

理解IIO，首先要看清它的全貌。与直接编写一个混杂（misc）设备驱动相比，IIO框架将驱动分层，每层各司其职。我在一个气压计驱动项目中，就是因为没理清这几层关系，把硬件触发逻辑写在了不该写的地方，导致系统延迟极高。

用户空间
↓ (sysfs/字符设备)
IIO核心层
⇄
IIO缓冲区
↓ (ops回调)
IIO驱动层
↓ (I2C/SPI/寄存器)
物理硬件
(ADC, Sensor)

IIO核心层：提供框架基石，管理设备的注册/注销，处理sysfs属性文件的创建与访问，维护缓冲区逻辑。你几乎不需要修改这部分代码。

IIO驱动层：这是你工作的主战场。你需要实现一个struct iio_info结构体，填充其中的回调函数（如read_raw），告诉核心层“如何读取数据”。同时，要定义设备具备哪些通道。

IIO缓冲区：一个可选但强大的组件，用于支持硬件触发和连续数据采集。当数据速率很高（如高速ADC）时，必须启用缓冲区。

核心数据结构与驱动编写步骤

编写一个IIO驱动，本质上是填充几个关键结构体并向内核注册。我们以一个假设的4通道ADC芯片“VREF-ADC100”（12位精度，I2C接口）为例。

第一步：定义通道（Channels）

通道是IIO中最重要的概念，它定义了传感器能测量什么。每个通道有类型（电压、温度、光强等）、索引、方向等信息。我们的ADC有4个单端电压输入通道。

// 定义ADC的4个电压输入通道
static const struct iio_chan_spec adc100_channels[] = {
    {
        .type = IIO_VOLTAGE, // 通道类型：电压
        .indexed = 1,
        .channel = 0,        // 通道0
        .info_mask_separate = BIT(IIO_CHAN_INFO_RAW) | BIT(IIO_CHAN_INFO_SCALE),
        .address = 0x00,     // 芯片内该通道的寄存器地址
    },
    {
        .type = IIO_VOLTAGE,
        .indexed = 1,
        .channel = 1,        // 通道1
        .info_mask_separate = BIT(IIO_CHAN_INFO_RAW) | BIT(IIO_CHAN_INFO_SCALE),
        .address = 0x01,
    },
    // ... 通道2和通道3
};

info_mask_separate指明了这个通道通过sysfs暴露哪些属性。这里BIT(IIO_CHAN_INFO_RAW)会生成in_voltage0_raw文件，BIT(IIO_CHAN_INFO_SCALE)会生成in_voltage0_scale文件。

第二步：实现iio_info回调函数

这是驱动与硬件对话的地方。read_raw是最常用的回调，当用户读取in_voltageX_raw文件时，内核最终会调用它。

static int adc100_read_raw(struct iio_dev *indio_dev,
                          struct iio_chan_spec const *chan,
                          int *val, int *val2, long mask)
{
    struct adc100_data *data = iio_priv(indio_dev);
    int ret;

    switch (mask) {
    case IIO_CHAN_INFO_RAW: // 处理读取原始值的请求
        // 通过I2C读取硬件寄存器，地址为chan->address
        ret = i2c_smbus_read_word_data(data->client, chan->address);
        if (ret < 0)
            return ret;
        *val = ret & 0x0FFF; // 我们的ADC是12位
        return IIO_VAL_INT;
    case IIO_CHAN_INFO_SCALE: // 处理读取比例因子的请求
        // 假设参考电压为3.3V，12位ADC，scale = 3300mV / 4096 LSB
        *val = 0;
        *val2 = 805664; // 单位是微伏 (3.3V/4096 ≈ 0.805664mV)
        return IIO_VAL_INT_PLUS_MICRO;
    default:
        return -EINVAL;
    }
}

static const struct iio_info adc100_info = {
    .read_raw = adc100_read_raw,
};

⚠️ 常见误区

  • 混淆val和val2的返回值类型：这是新手最易踩的坑。IIO_VAL_INT表示*val是有效值；IIO_VAL_INT_PLUS_MICRO表示*val是整数部分，*val2是微（10^-6）部分。如果比例因子是1.5，应设置*val=1，*val2=500000，返回IIO_VAL_INT_PLUS_MICRO。
  • 忘记处理字节序：通过I2C/SPI读取的多字节数据，要注意芯片的字节序（Endianness）。我曾调试一个温度传感器，读数总是错乱，最后发现是芯片返回的数据是低字节在前，而驱动按高字节在前解析了。
  • 在read_raw中睡眠或长时间占用锁：这个函数可能在中断上下文或持有锁的情况下被调用，必须保持高效，不能调用可能引起睡眠的函数（如msleep）。对于慢速传感器，应使用缓冲区+触发模式。

数据访问：Sysfs vs 字符设备

IIO为用户空间提供了两种数据访问方式，适用于不同场景。

Sysfs属性文件

  • 场景：配置、调试、低频数据读取。
  • 用法：直接cat或echo到/sys/bus/iio/devices/iio:deviceX/下的文件。
  • 示例：cat /sys/bus/iio/devices/iio:device0/in_voltage0_raw 读取原始值。

字符设备 (/dev/iio:deviceX)

  • 场景：高速、连续数据采集。
  • 用法：使用read()、ioctl()系统调用，配合libiio库。
  • 优势：支持阻塞/非阻塞IO、缓冲区、硬件触发。

对于大多数简单应用，Sysfs足够用。但如果你需要以100Hz以上的频率采集数据，或者希望由硬件事件（如GPIO中断）来触发采样，就必须启用IIO缓冲区并走字符设备接口。实际开发中，我建议先实现Sysfs接口，验证驱动基础功能，再根据需要添加缓冲区支持。

编者提示：IIO驱动底层通常依赖I2C或SPI总线与硬件通信。关于如何编写I2C/SPI客户端驱动、如何解析设备树节点获取配置，请回顾本教程的第9章 I2C总线驱动开发和第10章 SPI总线驱动开发。

实战：一个IIO驱动的完整流程

让我们把上面的代码片段串联起来，看看一个IIO驱动从初始化到被用户读取的完整生命周期。以下流程图忽略了错误处理以保持清晰。

2. 设置通道、模式、info等

关键步骤的代码骨架如下：

static int adc100_probe(struct i2c_client *client)
{
    struct iio_dev *indio_dev;
    struct adc100_data *data;

    // 1. 分配IIO设备结构，并分配私有数据结构空间
    indio_dev = iio_device_alloc(sizeof(*data));
    // 2. 关联i2c_client，初始化私有数据（如校准参数）
    data = iio_priv(indio_dev);
    data->client = client;

    // 3. 设置IIO设备基本信息
    indio_dev->name = "adc100";
    indio_dev->info = &adc100_info; // 前面定义的info结构体
    indio_dev->channels = adc100_channels;
    indio_dev->num_channels = ARRAY_SIZE(adc100_channels);
    indio_dev->modes = INDIO_DIRECT_MODE; // 直接模式，支持sysfs读取

    // 4. 向IIO子系统注册设备
    return iio_device_register(indio_dev);
}

注册成功后，你会在/sys/bus/iio/devices/下看到iio:deviceX目录，里面包含了定义的所有通道属性文件。

  1. 基于本章的代码框架，编写这个ADC的IIO驱动。在read_raw函数中实现原始值读取（注意10位数据可能位于读取结果的某几位）。
  2. 编译并加载驱动模块。使用i2c-tools中的i2cdetect命令确认设备被识别。
  3. 在/sys/bus/iio/devices/下找到你的设备，尝试用cat命令读取四个电压通道的原始值。
  4. （进阶）计算并实现正确的scale属性，使得用户读取in_voltageX_scale能得到以微伏为单位的比例因子。

本章小结

  • IIO子系统是Linux内核为传感器类设备提供的标准化框架，通过统一接口（sysfs/字符设备）简化了驱动开发和应用访问。
  • 编写IIO驱动的核心是定义iio_chan_spec通道数组并实现iio_info中的回调函数（特别是read_raw），将硬件操作封装在其中。
  • 数据访问有两种模式：Sysfs适用于低频配置和读取；字符设备配合缓冲区适用于高速、连续或硬件触发的数据采集。
  • 必须正确理解并返回IIO_VAL_*系列常量，这是驱动能正确提供数据的关键，也是新手最容易出错的地方。
  • 在实际项目中，应优先使用IIO框架而非从头编写字符设备驱动，以获得更好的可维护性、可移植性和内核社区支持。
