核心概念设备树节点属性

在ARM Linux 3.x版本之后，内核启动时不再需要大量的板级初始化代码。取而代之的，是一份名为设备树（Device Tree）的硬件描述文件。你可以把它想象成一份交给内核的“硬件清单”，内核根据这份清单来识别和管理CPU、内存、串口、I2C控制器等所有硬件资源。

设备树是什么？

生活类比：就像组装一台电脑前，你需要一份列明了主板型号、CPU、内存条、显卡、硬盘接口的配置单。设备树就是给Linux内核的这份“硬件配置单”。

技术定义：设备树是一种描述硬件资源的数据结构，以文本文件（.dts/.dtsi）形式存在，由DTC（Device Tree Compiler）编译成二进制格式（.dtb），在系统启动时由Bootloader传递给内核。

实际开发中，大部分驱动工程师的工作不再是写复杂的探测代码，而是正确编写或修改DTS文件，并在驱动中读取其中的属性。我在一个基于i.MX6ULL的项目中就曾因为一个引脚复用属性写错，导致整个SPI接口无法工作。

设备树的核心语法结构

一个典型的设备树源文件（.dts）由节点（node）和属性（property）构成，形成一棵树状结构。根节点是“/”，所有硬件都挂载在这棵树上。

// 一个最简单的设备树示例
/dts-v1/;
/ { // 根节点
    compatible = "my-company,my-board";
    model = "My Board Rev 1.0";
    #address-cells = <1>;
    #size-cells = <1>;

    cpus { // CPU子节点
        #address-cells = <1>;
        #size-cells = <0>;
        cpu@0 {
            compatible = "arm,cortex-a7";
            device_type = "cpu";
            reg = <0>;
        };
    };

    uart0: serial@30860000 { // 串口设备节点，带标签‘uart0’
        compatible = "fsl,imx6ul-uart", "fsl,imx6q-uart";
        reg = <0x30860000 0x1000>;
        interrupts = <26 IRQ_TYPE_LEVEL_HIGH>;
        status = "okay";
    };
};

属性是键值对，值支持多种数据类型。下面这个表格总结了最关键的几个属性及其用途，你会在几乎每个DTS文件中遇到它们。

关键点：compatible属性是灵魂

内核驱动通过compatible属性与设备树节点绑定。一个节点可以有多个compatible值，内核会按顺序尝试匹配。例如compatible = “fsl,imx6ul-uart”, “fsl,imx6q-uart”;，内核会优先匹配imx6ul-uart的驱动，如果没找到，再尝试匹配更通用的imx6q-uart驱动。

设备树的层次化与引用机制

设备树支持良好的代码复用。SoC厂商会提供一个基础的.dtsi（include文件），描述芯片内部的所有外设。板级开发者则编写具体的.dts文件，通过#include引用.dtsi，然后进行覆盖、修改和添加。

imx6ull.dtsi (SoC厂商提供)
↗
my-board.dts (板级开发)
↘
最终DTB (Bootloader加载)
↓
定义：
&uart1 { reg = ...; };
→
引用并启用：
&uart1 { status = “okay”; };

节点可以通过标签（label）和句柄（phandle）被引用。标签以冒号结尾（如uart1: serial@...），在其他地方用&uart1即可引用该节点。这是配置引脚复用（pinctrl）和时钟依赖的常用方式。

// 在pinctrl节点中定义一组引脚配置
pinctrl_uart1: uart1grp {
    fsl,pins = </* 具体的引脚复用配置 */>;
};

// 在uart1节点中引用这组配置
&uart1 {
    pinctrl-names = "default";
    pinctrl-0 = <&pinctrl_uart1>; // 使用‘&’引用标签
    status = "okay";
};

⚠️ 常见误区

  • 地址写错单位：reg属性中的地址通常是32位值（1个cell为4字节）。对于像64位地址的PCIe空间，需要设置#address-cells = <2>;。我曾见过有人把0x1000（4KB）误写成0x1000（4KB）但期望是1KB，实际长度是0x400。
  • compatible字符串拼写错误：多一个空格或少一个横杠，驱动都无法匹配。内核日志会打印“No compatible driver found for node ...”，这是排查设备树问题的第一个线索。
  • 忘记status = “okay”：即使节点定义正确，如果status不是“okay”，内核也不会初始化该设备。默认状态可能是“disabled”。

在驱动中如何访问设备树？

驱动代码通过一系列以of_（open firmware）开头的内核API来读取设备树节点信息。基本流程是：先匹配节点，再从节点中解析出需要的属性值。

1. 定义of_device_id匹配表
→
2. 在驱动结构体中注册该表
→
3. 在probe函数中解析节点属性
// 1. 定义匹配表
static const struct of_device_id my_uart_of_match[] = {
    { .compatible = "fsl,imx6ul-uart" },
    { .compatible = "fsl,imx6q-uart" },
    { /* 哨兵，标记结束 */ },
};
MODULE_DEVICE_TABLE(of, my_uart_of_match);

// 2. 在platform_driver中注册
static struct platform_driver my_uart_driver = {
    .probe = my_uart_probe,
    .driver = {
        .name = "my-uart",
        .of_match_table = my_uart_of_match, // 关键！
    },
};

// 3. 在probe函数中解析属性
static int my_uart_probe(struct platform_device *pdev)
{
    struct device_node *np = pdev->dev.of_node; // 获取设备树节点指针
    u32 reg_addr, reg_size;
    int irq_num;

    // 解析reg属性
    if (of_address_to_resource(np, 0, &res)) {
        dev_err(&pdev->dev, "failed to get memory resource\n");
        return -EINVAL;
    }
    // 或者直接解析数值
    of_property_read_u32_index(np, "reg", 0, ®_addr);
    of_property_read_u32_index(np, "reg", 1, ®_size);

    // 解析中断
    irq_num = platform_get_irq(pdev, 0);
    if (irq_num < 0) {
        return irq_num;
    }

    // 解析字符串属性，如状态
    const char *status;
    of_property_read_string(np, "status", &status);
    dev_info(&pdev->dev, "Device status: %s\n", status);
    // ... 其他初始化代码
}

编者提示：设备树驱动的加载依赖于第6章讲解的平台设备驱动模型。驱动通过of_match_table与设备树节点匹配，匹配成功后，内核会自动创建对应的platform_device并调用驱动的probe函数。

如果你在驱动中读取一个不存在的属性，相关API会返回一个错误码或默认值（例如of_property_read_u32返回非0值）。实际开发中，对于非必需的属性，应该提供合理的默认值。

场景：添加一个LED设备

  • 在DTS中创建leds子节点。
  • 使用gpios属性指定控制引脚。
  • compatible设为"gpio-leds"，内核LED子系统会自动识别。

场景：配置I2C总线上的传感器

  • 在&i2c1节点下添加子节点。
  • reg属性填写传感器的I2C从机地址（如0x68）。
  • 通过自定义属性（如poll-interval-ms）传递驱动参数。

场景：禁用某个默认外设

  • 在板级DTS文件中引用SoC DTSI中的节点。
  • 将其status属性覆盖为"disabled"。
  • 例如：&usdhc2 { status = “disabled”; };

设备树也支持一些简单的逻辑和计算，比如通过#include包含头文件（定义常量），使用#define宏，以及通过&引用其他节点的值。这让你能写出更清晰、可维护的硬件描述。

本章小结

  • 设备树（DTS）是描述硬件拓扑和配置的文本文件，用于替代硬编码的板级信息，是驱动与硬件之间的“合同”。
  • 设备树由节点和属性构成，compatible属性是驱动绑定的关键，reg、interrupts等属性描述了硬件资源。
  • 通过.dtsi包含和节点覆盖机制，可以实现SoC级定义与板级定制的分离，提高代码复用率。
  • 内核驱动通过of_系列API（如of_property_read_u32、of_address_to_resource）从设备树节点中解析出配置信息，完成设备初始化。
  • 常见的坑包括属性拼写错误、地址长度单位混淆以及忘记将status设为“okay”。
