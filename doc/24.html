核心概念日志级别动态调试

驱动代码在目标板上跑飞了，系统没有任何反应，串口一片寂静。这是我早期调试一个I2C触摸屏驱动时遇到的真实场景。没有有效的日志输出，定位问题就像在黑暗中摸索。本章将带你系统掌握内核的“眼睛”和“耳朵”——调试与日志系统。

内核日志系统：printk与日志级别

内核日志就像汽车的仪表盘，不同颜色的指示灯代表不同紧急程度的信息。printk是内核最基础的日志输出函数，它的核心在于日志级别。这个级别决定了消息是否会打印到控制台，以及最终出现在哪个日志文件里。

日志级别宏数值控制台显示规则典型使用场景
KERN_EMERG0总是打印系统即将崩溃
KERN_ALERT1总是打印需要立即行动的错误
KERN_CRIT2总是打印严重硬件/软件故障
KERN_ERR3总是打印驱动操作失败（如I2C读错误）
KERN_WARNING4总是打印非致命异常（如内存紧张）
KERN_NOTICE5默认打印正常但重要的事件（如设备探测成功）
KERN_INFO6默认不打印信息性消息（如寄存器值）
KERN_DEBUG7默认不打印详细的调试信息

控制台实际打印的级别阈值由/proc/sys/kernel/printk文件控制。一个典型的用法如下：

// 错误信息，总会看到
printk(KERN_ERR "my_driver: Failed to read register 0x%x\n", reg_addr);

// 调试信息，默认不显示
printk(KERN_DEBUG "my_driver: i2c transfer, addr=0x%x, len=%d\n", client->addr, len);

常见误区

  • 滥用printk导致性能下降：在高速中断处理函数或频繁调用的路径中使用printk（即使是KERN_DEBUG）会严重拖慢系统。printk内部有锁，开销不小。
  • 忘记添加驱动名前缀：裸奔的printk(“value is %d\n”, val);在dmesg输出中难以定位来源，务必加上“驱动名: ”前缀。
  • 混淆printk与printf格式：内核的printk不支持%lld打印64位整数，需要用%llu或%lld并配合long long类型强制转换。

设备专属日志：dev_xxx系列函数

对于有struct device或struct device *上下文的驱动，推荐使用dev_xxx系列函数。它们不仅自动包含设备标识信息，还能根据设备配置决定输出目的地（比如重定向到串口或网络）。

→

→
内核日志缓冲区
↓
dmesg / klog
↗
系统日志 (syslog)
↘
控制台

实际开发中，大部分团队会强制要求使用dev_xxx替代裸printk。它的输出包含了总线地址和设备名，对排查多设备系统的问题至关重要。

// 在 probe 函数中
struct i2c_client *client;
dev_info(&client->dev, "probed at address 0x%02x\n", client->addr);

// 在中断处理中（假设有 device 指针）
dev_dbg(data->dev, "interrupt triggered, status=0x%02x\n", status_reg);

关键点：动态调试（Dynamic Debug）

pr_debug()和dev_dbg()在编译时默认是“空操作”，除非你定义了DEBUG宏。但更强大的方法是动态调试。它允许你在系统运行时，通过/sys/kernel/debug/dynamic_debug/control文件动态开启或关闭特定文件、函数、行号的调试信息，无需重新编译或重启。

启用动态调试需要内核配置CONFIG_DYNAMIC_DEBUG=y。使用方式如下：

# 启用某个文件的所有 dev_dbg 打印
echo 'file drivers/i2c/busses/i2c-imx.c +p' > /sys/kernel/debug/dynamic_debug/control

# 启用特定函数的打印
echo 'func i2c_imx_xfer +p' > /sys/kernel/debug/dynamic_debug/control

# 禁用所有动态调试打印
echo '-p' > /sys/kernel/debug/dynamic_debug/control

编者提示：动态调试依赖于debugfs文件系统。关于如何挂载和使用debugfs，以及更多/sys下的调试接口，你可以在第25章《Proc与Sys文件系统》中找到详细讲解。

内核跟踪利器：ftrace与Kprobes

当日志无法定位间歇性死锁或性能瓶颈时，你需要更底层的跟踪工具。ftrace是内核内置的函数跟踪器，开销极低。我在排查一个SD卡驱动偶尔超时的问题时，就是通过ftrace发现中断被意外关闭了。

→
启用跟踪器（如function）
→
设置过滤（函数名、PID）
→
开始跟踪
→
执行测试用例
→
停止跟踪并查看报告

基本操作命令序列：

# 1. 挂载（如果尚未挂载）
mount -t debugfs none /sys/kernel/debug

# 2. 切换到ftrace目录，启用函数跟踪器
cd /sys/kernel/debug/tracing
echo function > current_tracer

# 3. 设置只跟踪你的驱动函数（避免海量数据）
echo my_driver_* > set_ftrace_filter
# 或者跟踪特定函数，如 spin_lock
echo spin_lock >> set_ftrace_filter

# 4. 开始跟踪
echo 1 > tracing_on
# ... 运行你的测试 ...
echo 0 > tracing_on

# 5. 查看结果
cat trace | less

Kprobes：在任意位置插入探测点

如果说ftrace是观察主干道车流，那么Kprobes就是在特定十字路口安装摄像头。它允许你在几乎任何内核指令地址（包括函数内部）插入断点，执行自定义处理程序（比如记录寄存器值）。这对于分析没有源码的第三方内核模块或极其隐秘的竞争条件非常有用。

内核源码的samples/kprobes/目录下有完整示例。一个简单的Kprobe模块骨架如下：

#include 

static struct kprobe kp = {
    .symbol_name = "do_fork", // 要探测的函数名
};

// 探测点被命中时的处理函数
static int handler_pre(struct kprobe *p, struct pt_regs *regs) {
    pr_info("do_fork called, pre-handler\n");
    return 0;
}

static int __init kprobe_init(void) {
    kp.pre_handler = handler_pre;
    int ret = register_kprobe(&kp);
    if (ret < 0) { ... }
    return 0;
}

快速打印调试

  • 工具: printk, dev_xxx
  • 场景: 验证代码路径、查看变量值
  • 开销: 中（printk有锁）

运行时动态调试

  • 工具: 动态调试 (dev_dbg)
  • 场景: 生产环境按需开启详细日志
  • 开销: 低（不开启时为零）

性能与流程分析

  • 工具: ftrace, trace-cmd
  • 场景: 分析延迟、函数调用关系、调度问题
  • 开销: 极低

深度指令级探测

  • 工具: Kprobes, SystemTap
  • 场景: 无源码调试、分析极端竞态
  • 开销: 高（单点）

动手试一试

  1. 编写一个简单的内核模块，在其init函数中使用KERN_EMERG到KERN_DEBUG不同级别打印信息。加载模块，观察dmesg输出。然后修改/proc/sys/kernel/printk的当前控制台日志级别（第二个值）为4，重新加载模块，观察哪些信息不再显示在控制台。
  2. 找一个已有驱动（如一个GPIO LED驱动），在关键函数中添加dev_dbg()语句。重新编译内核并启用CONFIG_DYNAMIC_DEBUG。启动系统后，使用echo 'file drivers/leds/leds-gpio.c +p' > /sys/kernel/debug/dynamic_debug/control命令动态开启调试信息，操作LED设备，观察dmesg输出。
  3. 使用ftrace跟踪schedule()函数的调用。按照本章流程，设置过滤器，在运行一个消耗CPU的程序（如dd if=/dev/zero of=/dev/null）前后开启和停止跟踪，分析trace文件，看schedule被调用了多少次。

检验你的理解

  1. 在中断处理函数顶部，为了快速记录一个错误码，你应该使用printk(KERN_INFO, ...)还是printk(KERN_ERR, ...)？为什么？
  2. 动态调试（Dynamic Debug）的主要优势是什么？它和定义DEBUG宏再编译有什么区别？
  3. 如果你的驱动在某个罕见条件下发生死锁，导致系统无响应，连printk都无法输出。除了加日志，你还可以采用本章提到的哪种技术来事后分析？简述其思路。

本章小结

  • 分级日志是基础：掌握printk的八个日志级别，优先使用dev_xxx系列函数输出设备相关日志，便于问题追踪。
  • 动态调试是生产环境利器：使用pr_debug()/dev_dbg()配合动态调试，可以在不重启的情况下灵活控制调试信息的输出，极大提升线上问题排查效率。
  • ftrace用于性能与流程分析：它是内核内置的低开销跟踪框架，适合分析函数调用关系、调度延迟和中断关闭情况，是定位复杂性能问题的首选工具。
  • Kprobes用于深度探测：当需要在内核任意位置（包括无源码处）插入探测点来收集数据时，Kprobes提供了强大的底层能力，常用于分析极端竞态或第三方模块行为。
  • 工具选择取决于场景：从快速打印到动态调试，再到系统跟踪和指令探测，根据问题的紧急程度、复现难度和对系统的影响，选择合适的调试工具组合。
