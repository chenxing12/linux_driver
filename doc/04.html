核心概念内核地址空间GFP标志

启动开发板，加载你的第一个驱动模块，系统立刻报错“Unable to handle kernel NULL pointer dereference”。十有八九，你踩到了内存管理的坑。内核内存不像用户态程序那样可以随意malloc/free，它有严格的规则和边界。

4.1 内核的“地盘”：内核地址空间

生活类比： 一个公司的办公区（内核空间）和员工的个人储物柜（用户空间）是严格分开的，办公区的工具不能带回家，家里的私人物品也不能随便放进办公区。技术定义： 在Linux中，虚拟地址空间被划分为内核空间（通常为高地址1GB或更多）和用户空间，驱动程序运行在内核空间，只能直接访问内核地址。

所有内核代码，包括你的驱动，都共享这同一个内核地址空间。这意味着你驱动里一个错误的指针写操作，可能破坏文件系统、网络协议栈，直接导致系统崩溃（Kernel Panic）。我在调试一个触摸屏驱动时，就因为一个越界写，破坏了邻近的USB核心数据结构，让整个USB子系统失灵。

4.2 内核内存分配器与API选择

内核提供了多种内存分配函数，就像工具箱里有不同尺寸的扳手。选错了，要么效率低下，要么根本拧不上。

分配函数内存来源特点与限制典型应用场景是否可睡眠
kmalloc物理连续的页速度快，地址连续，大小有限（通常≤4MB）DMA缓冲区、小型数据结构取决于GFP标志
vmalloc虚拟连续的非连续页可分配大内存，地址虚拟连续但物理不连续大型软件缓冲区、模块加载是（可能睡眠）
kzalloc物理连续的页等价于 kmalloc 并清零内存需要初始化为零的结构体取决于GFP标志
get_free_page整页按页分配，获取物理连续页需要页对齐的缓冲区取决于GFP标志

实际开发中，kmalloc 是驱动开发最常用的分配器，用于分配小于一页（通常是4KB）或几个页的小块内存。它的第一个参数是大小，第二个参数是GFP（Get Free Page）标志，这个标志决定了分配行为。

GFP标志：告诉内核你的“需求清单”

GFP标志控制分配器从哪里、以何种方式获取内存。用错了标志，在中断上下文会导致死锁，在压力下可能分配失败。

  • GFP_KERNEL：最常用。标准内核内存分配。可能睡眠、可能进行磁盘I/O来回收内存。只能在进程上下文（例如open、read等系统调用路径）中使用。
  • GFP_ATOMIC：原子分配，不会睡眠。用于中断上下文、自旋锁持有期间等不能睡眠的场景。成功率低于GFP_KERNEL。
  • GFP_DMA：为DMA分配可从特定区域（如16MB以下）寻址的内存。老式ISA设备需要。
  • __GFP_ZERO：与其他标志组合使用，分配后自动清零内存。

来看一段典型代码。假设我们在驱动的 probe 函数中为一个私有数据结构分配内存：

struct my_device_data {
    int status;
    char name[32];
    dma_addr_t dma_handle;
};

static int my_driver_probe(struct platform_device *pdev) {
    struct my_device_data *data;

    // 使用 GFP_KERNEL，因为 probe 在进程上下文执行
    data = kzalloc(sizeof(struct my_device_data), GFP_KERNEL);
    if (!data) {
        dev_err(&pdev->dev, "Failed to allocate device data\n");
        return -ENOMEM;
    }
    // ... 初始化 data
    platform_set_drvdata(pdev, data);
    return 0;
}

常见误区

  • 在中断处理函数中使用 GFP_KERNEL：这是致命错误。中断上下文不能睡眠，使用 GFP_KERNEL 可能导致内核死锁。必须使用 GFP_ATOMIC。
  • 忘记检查分配返回值：内核内存分配可能失败，尤其是使用 GFP_ATOMIC 或在内存紧张时。不检查 NULL 指针会导致后续的 Oops。
  • 混淆 kmalloc 和 vmalloc 的用途：为 DMA 分配缓冲区必须使用物理连续的 kmalloc 或专用 DMA API，vmalloc 返回的地址不能直接用于 DMA 传输。

4.3 内存的释放与生命周期管理

有借有还，再借不难。内核没有垃圾回收器，你申请的内存必须显式释放，否则就是内存泄漏。泄漏的内核内存直到重启都无法回收。

驱动加载/设备探测
→
kmalloc/vmalloc 申请内存
→
驱动运行期使用内存
→
驱动卸载/设备移除
→
kfree/vfree 释放内存

释放函数必须与分配函数严格配对：kmalloc -> kfree，vmalloc -> vfree。一个隐蔽的坑是：你只能释放由你自己分配的、且未被释放过的内存的起始指针。 释放一个中间指针，或者重复释放（Double Free），都会立刻破坏内核内存管理器的数据结构。

static void my_driver_remove(struct platform_device *pdev) {
    struct my_device_data *data = platform_get_drvdata(pdev);

    // 正确：释放 probe 中分配的内存
    kfree(data);
    // 错误：data = NULL; // 虽然好习惯，但这里 platform_set_drvdata(pdev, NULL) 更彻底
}

编者提示： 内存的生命周期管理常与设备的生命周期绑定。关于设备模型的详细创建、注册以及在 probe/remove 中的资源管理，我们将在第6章《平台设备驱动模型》和第26章《设备模型与kobject》中深入探讨。

4.4 为DMA分配内存

当你的设备需要通过DMA（直接内存访问）与内存交换数据时，分配要求更严格：缓冲区必须在物理上是连续的，并且设备可能对地址有对齐要求（例如32字节对齐）。

对于现代内核（尤其是使用设备树后），推荐使用统一的DMA API（如 dma_alloc_coherent），它处理了缓存一致性和地址映射的复杂性。一个典型的为网络芯片分配接收环缓冲区的例子：

/* 假设我们有一个PCIe设备 */
void *cpu_addr;
dma_addr_t dma_handle;

// 分配 4KB 对齐的、设备可访问的DMA缓冲区
cpu_addr = dma_alloc_coherent(&pdev->dev, 4096, &dma_handle, GFP_KERNEL);
if (!cpu_addr) {
    // 处理错误
}
// cpu_addr: CPU可访问的虚拟地址，用于读写数据
// dma_handle: 总线地址，需要写入设备的DMA寄存器
my_device_regs->dma_target_addr = dma_handle;

小对象，进程上下文

  • 函数： kzalloc(size, GFP_KERNEL)
  • 场景： 在 probe、open、read/write 中分配设备私有结构体、小缓冲区。

中断或原子上下文

  • 函数： kmalloc(size, GFP_ATOMIC)
  • 场景： 在中断处理函数（ISR）、定时器回调、tasklet中分配紧急数据。

大型软件缓冲区

  • 函数： vmalloc(size)
  • 场景： 驱动需要一个大数组做软件FIFO（例如视频帧缓冲），且不用于DMA。

设备DMA操作

  • 函数： dma_alloc_coherent
  • 场景： 为网络卡、存储控制器等分配设备可直接读写的DMA缓冲区。

最后，记住一个数字：PAGE_SIZE。它是内核内存管理的基本单位，在x86和ARM上通常是4096字节。当你申请的内存大小接近或超过一个页时，分配器的行为会变化，开销也会增大。

本章小结

  • 内核内存位于独立的内核地址空间，驱动错误的内存访问会导致系统级崩溃。
  • kmalloc 用于分配物理连续的小块内存，是最常用的分配器；vmalloc 用于分配虚拟连续的大块内存，但物理不连续，不能用于DMA。
  • GFP标志（GFP_KERNEL/GFP_ATOMIC）控制分配行为，必须根据执行上下文（能否睡眠）正确选择。
  • 分配的内存必须配对释放（kfree/vfree），且必须检查分配是否成功，否则会导致内存泄漏或空指针访问。
  • 为DMA操作分配内存应使用专用的DMA API（如 dma_alloc_coherent），以确保缓存一致性和物理连续性。
