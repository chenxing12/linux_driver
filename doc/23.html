核心概念热插拔uevent

当你把U盘插入电脑，系统瞬间就能识别并挂载它；拔掉时，文件管理器里的图标也立刻消失。这个看似简单的“即插即用”背后，是Linux内核一套精密的事件通知机制在运作。本章我们将深入内核，拆解驱动如何感知硬件变化，又如何将变化“广播”给整个系统。

热插拔：内核的动态感知系统

生活类比： 就像一个智能仓库的自动入库系统，当有新货物（设备）通过传送带（总线）进入时，系统会自动扫描条码、分配货架并更新库存清单。技术定义： 热插拔（Hotplug）指在系统不关机、不重启的情况下，动态地添加或移除硬件设备，内核需要实时感知这种状态变化并做出响应。

内核中的热插拔处理是一个由底层到上层的链条。总线驱动（如USB、PCIe）负责检测物理连接变化，设备模型核心（kobject）负责生成标准化的事件，最后通过netlink套接字将事件发送给用户空间的守护进程（如udev）。

uevent：内核与用户空间的通信信封

热插拔信息如何传递？答案就是uevent。它是内核空间向用户空间发送的事件通知，包含了设备动作（add/remove/change等）和设备属性（如设备路径、子系统、驱动名）。

硬件层
↓
总线驱动
↓
内核设备模型
↓
生成uevent
→
netlink socket
→
用户空间 (udev/mdev)

一个典型的uevent消息是一系列KEY=VALUE格式的环境变量。例如，插入一个USB鼠标后，内核可能会发出如下内容：

ACTION=add
DEVPATH=/devices/pci0000:00/0000:00:14.0/usb3/3-2/3-2:1.0
SUBSYSTEM=usb
DEVNAME=/dev/bus/usb/003/004
PRODUCT=46d/c52b/200
TYPE=0/0/0

关键点：谁负责生成uevent？

在驱动中，我们通常不直接调用底层API生成uevent。当你在设备模型中正确操作kobject时（例如调用 device_add() 或 kobject_uevent()），内核会自动帮你生成并发送uevent。我在一个SDIO Wi-Fi模块驱动项目中踩过坑，手动拼接uevent字符串导致udev规则无法匹配，后来改用标准API问题迎刃而解。

uevent的动作类型与属性

理解uevent的关键是看懂它的“动作”和“属性”。下表列出了最核心的字段：

字段名可能的值说明
ACTIONadd, remove, change, move, online, offline事件类型。“add”和“remove”最常见。
DEVPATH如 /devices/.../usb1/1-1设备在sysfs中的路径，是uevent的绝对标识。
SUBSYSTEMusb, input, block, net, platform设备所属的子系统。
MAJOR & MINOR数字设备的主次设备号，对字符/块设备至关重要。
SEQNUM递增的整数事件序列号，用于区分先后顺序。
DRIVER驱动模块名绑定到该设备的驱动名称。

实际开发中，大部分团队会依赖SUBSYSTEM和DEVPATH来编写udev规则，实现设备的自动配置（如修改权限、创建符号链接、加载固件）。

⚠️ 常见误区

  • 混淆kobject_uevent的发送目标： kobject_uevent(&struct kobject *kobj, enum kobject_action action) 默认发送给用户空间，而 kobject_uevent_env() 可以添加自定义环境变量。如果你只想在内部内核模块间通知，应该用其他的IPC机制。
  • 在错误时机发送uevent： 必须在设备在sysfs中创建成功（即device_add完成）之后，才能发送“add”事件。顺序反了，用户空间程序会找不到设备节点。
  • 忽略uevent的失败： kobject_uevent() 是有可能失败的（比如netlink缓冲区满）。对于关键设备，需要添加简单的错误处理或重试逻辑。

在驱动中触发uevent

驱动开发者通常在这两个场景需要主动触发uevent：1. 设备状态发生重要变化（如从睡眠中唤醒）；2. 为设备添加了新的、需要用户空间知晓的属性。

下面是一个在平台设备驱动中，当检测到设备“上线”时触发“change”事件的示例：

// 假设 dev 是一个已注册的 struct device 指针
static void my_device_come_online(struct my_private_data *priv)
{
    // ... 硬件初始化代码 ...
    
    // 触发一个uevent，通知用户空间设备状态变为“online”
    char *envp[] = { "MY_STATUS=online", NULL };
    kobject_uevent_env(&priv->dev->kobj, KOBJ_CHANGE, envp);
    
    pr_info("Device online event sent.\n");
}

编者提示： 理解本章的uevent机制，需要你对Linux设备模型中的kobject、kset和sysfs有基本认识。如果对这些概念感到陌生，建议先回顾第26章《设备模型与kobject》的内容，那里详细解释了这些核心数据结构是如何组织和工作的。

用户空间如何接收uevent？

内核发出的uevent，最终由用户空间的守护进程处理。最主流的是udev（systemd体系）和mdev（BusyBox嵌入式系统）。它们的工作流程如下：

内核生成uevent
→
netlink监听线程接收
→
解析ACTION/DEVPATH等
→
匹配 /etc/udev/rules.d/*.rules
→
执行规则（创建设备节点、设置权限等）

你可以直接使用工具监听原始的uevent流，这对调试驱动非常有帮助：

# 使用udevadm监控所有内核uevent
$ udevadm monitor --kernel --property --subsystem-match=usb
# 插入一个USB设备，你将看到实时的uevent输出

调试驱动

  • 使用 udevadm monitor 验证驱动发出的uevent是否正确。
  • 检查 dmesg 中是否有 kobject_uevent 相关的错误。

定制设备行为

  • 编写udev规则，为特定VID/PID的USB设备自动加载固件。
  • 当网络设备出现时，自动执行配置脚本。

嵌入式系统

  • 在资源受限系统使用mdev，基于uevent动态创建/dev下的节点。
  • 通过uevent触发轻量级的挂载或服务启动。

如果你在开发板上看到用户空间没有创建设备节点，一个快速的排错命令是 udevadm test $(DEVPATH)，它能模拟处理过程并显示详细的规则匹配和错误信息。

动手试一试

1. 监听热插拔事件： 打开两个终端。在终端A运行 udevadm monitor --kernel --property。在终端B插入一个U盘。观察终端A输出的完整uevent信息，记录下ACTION, DEVPATH, SUBSYSTEM, MAJOR, MINOR等字段。

2. 编写简易udev规则： 创建一个文件 /etc/udev/rules.d/99-my-test.rules，内容如下： 

ACTION=="add", SUBSYSTEM=="usb", ATTR{idVendor}=="1234", ATTR{idProduct}=="5678", SYMLINK+="my_cool_device"
这条规则会在添加一个特定VID/PID的USB设备时，在/dev下创建一个名为my_cool_device的符号链接（请将VID/PID替换成你实际设备的，或用通用的如 ATTR{idVendor}=="1d6b"（Linux Foundation）测试）。插入设备，检查 /dev/my_cool_device 链接是否生成。

3. 内核模块实验（可选，需内核开发环境）： 编写一个简单的内核模块，在初始化函数中创建一个kobject，并调用 kobject_uevent(&my_kobj, KOBJ_ADD)。加载模块，用 udevadm monitor 查看是否捕获到该事件。

检验你的理解

  1. 当物理上拔出一个USB网卡时，内核发出的uevent中，ACTION字段应该是什么？这个事件首先由内核的哪个部分检测并发起？
  2. 驱动代码中，函数 device_add(&my_dev) 被调用后，内核会自动发送一个uevent吗？如果会，它的ACTION是什么？
  3. 在用户空间，程序如何区分两个先后到达的、DEVPATH相同的uevent？
  4. （情景题）你为一块定制PCIe加速卡编写驱动。当卡上某个传感器数值超过阈值时，你需要在驱动中通知用户空间一个守护进程。除了uevent，还有哪些内核到用户空间的通信机制可以考虑？在什么情况下你会选择uevent？

本章小结

  • 热插拔是动态硬件管理的核心机制，由总线驱动检测，通过设备模型生成标准化uevent事件完成内核到用户空间的通告。
  • uevent是格式为“KEY=VALUE”的环境变量集合，必须包含ACTION、DEVPATH、SUBSYSTEM等核心字段，通过netlink套接字发送。
  • 驱动中应使用标准API（如kobject_uevent）触发事件，并确保在设备在sysfs中注册成功后再发送“add”事件，避免顺序错误。
  • 用户空间的udev/mdev是uevent的消费者，它们根据规则文件（/etc/udev/rules.d/）执行设备节点创建、权限设置等自动化操作。
  • 掌握udevadm monitor和udevadm test工具是调试驱动热插拔相关问题的关键手段。
