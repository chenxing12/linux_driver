核心概念input_devinput_handler

想象一下，你家里的所有电器遥控器（电视、空调、音响）都通过一个万能遥控器来操作。这个万能遥控器定义了统一的按键编码和通信协议，无论你买哪个品牌的电器，只要它支持这个协议，就能被遥控。Linux的输入子系统就是这个“万能遥控器”，它为键盘、鼠标、触摸屏、游戏手柄等所有输入设备提供了一个统一的上报事件接口，让应用层（如桌面环境、游戏）无需关心底层硬件的具体差异。

技术定义：Linux输入子系统（Input Subsystem）是内核中一个标准化的框架，用于抽象和管理所有输入设备。它将驱动开发分为设备驱动层、核心层和事件处理层，实现了设备与应用的解耦。

8.1 输入子系统三层架构

输入子系统采用典型的分层设计，每一层职责明确。理解这个架构是编写正确驱动的前提。我在调试一个电容触摸屏驱动时，就是因为没搞清楚事件应该上报到哪一层，导致应用层收不到触摸坐标。

应用层 (Userspace)
↑↓
事件处理层 (Input Handler)
↑↓
核心层 (Input Core)
↑↓
设备驱动层 (Input Driver)
→
硬件 (键盘/鼠标/触摸屏)

这三层通过核心层进行连接和匹配。设备驱动层负责与硬件通信，将原始的扫描码、坐标、压力值等转换为标准事件。核心层是“交通枢纽”，维护设备列表和处理程序列表，并进行匹配。事件处理层则将标准事件封装成用户空间可读的格式（如/dev/input/eventX）。

各层核心数据结构

  • 设备驱动层：核心是struct input_dev。它描述了一个输入设备的能力（能产生哪些事件）和状态。
  • 事件处理层：核心是struct input_handler。例如evdev_handler负责生成/dev/input/eventX节点，joydev_handler负责游戏手柄。
  • 核心层：通过input_register_device和input_register_handler函数完成注册与匹配。

8.2 如何描述一个输入设备：input_dev

编写输入设备驱动的第一步，就是创建并配置一个input_dev结构体。这相当于给你的硬件设备制作一张“身份证”，告诉内核：“我是一个键盘，我能产生A、B、回车等按键事件”。

struct input_dev 关键成员与设置函数
成员/函数作用典型代码示例
name设备名称，会在/proc/bus/input/devices中显示dev->name = “My GPIO Keys”;
phys设备在系统中的物理路径dev->phys = “gpio-keys/input0”;
id厂商、产品、版本等ID信息dev->id.bustype = BUS_HOST;
evbit设备支持的事件类型位图set_bit(EV_KEY, dev->evbit);
keybit当支持EV_KEY时，具体支持哪些键值set_bit(KEY_ENTER, dev->keybit);
absbit当支持EV_ABS时，具体支持哪些绝对坐标轴set_bit(ABS_X, dev->absbit);
input_set_capability()设置设备能力的辅助函数input_set_capability(dev, EV_KEY, KEY_POWER);
input_set_abs_params()设置绝对坐标轴参数（如分辨率、范围）input_set_abs_params(dev, ABS_X, 0, 1023, 0, 0);

实际开发中，大部分团队会使用input_set_capability和input_set_abs_params这些辅助函数，它们比直接操作位图更安全、可读性更好。我曾经接手过一个老驱动，直接对keybit进行移位操作，代码晦涩难懂，后来全部重构成了标准函数调用。

常见误区

  • 忘记设置事件类型：只设置了KEY_ENTER，但没设置EV_KEY，导致设备无法上报任何按键事件。内核会忽略没有事件类型支持的键值。
  • abs坐标参数设置错误：对于触摸屏，input_set_abs_params中的最小值和最大值必须与实际硬件ADC范围一致。如果最大值设小了，上报的坐标会被截断。
  • 混淆EV_REL和EV_ABS：鼠标移动应使用EV_REL上报相对位移，触摸屏应使用EV_ABS上报绝对坐标。用错类型会导致应用层无法正确解析。

8.3 编写一个GPIO按键驱动实战

让我们从一个最简单的硬件开始：一个连接到GPIO的按键。当按键按下时，GPIO变为低电平；松开时恢复高电平。我们的目标是将其注册为一个标准的输入设备，上报按键事件。

// 1. 包含必要的头文件
#include <linux/module.h>
#include <linux/input.h>
#include <linux/gpio/consumer.h>
#include <linux/interrupt.h>

struct gpio_key_data {
    struct input_dev *input;
    struct gpio_desc *gpiod;
    int irq;
    int keycode;
};

// 2. 中断处理函数：上报按键事件
static irqreturn_t gpio_key_isr(int irq, void *dev_id)
{
    struct gpio_key_data *data = dev_id;
    int state = gpiod_get_value(data->gpiod);

    // 上报键值，状态：1为松开，0为按下（假设低电平有效）
    input_report_key(data->input, data->keycode, !state);
    // 同步事件，告知事件处理层一个事件报告已完成
    input_sync(data->input);

    return IRQ_HANDLED;
}

// 3. 驱动初始化：创建并配置input_dev
static int gpio_key_probe(struct platform_device *pdev)
{
    struct gpio_key_data *data;
    struct input_dev *input;
    int ret;

    // 分配驱动私有数据结构
    data = devm_kzalloc(&pdev->dev, sizeof(*data), GFP_KERNEL);
    if (!data)
        return -ENOMEM;

    // 分配input_dev结构体
    input = devm_input_allocate_device(&pdev->dev);
    if (!input)
        return -ENOMEM;

    // 配置input_dev的基本信息
    input->name = "GPIO Power Key";
    input->phys = "gpio-keys/input0";
    input->id.bustype = BUS_HOST;

    // 设置设备能力：支持按键事件，具体键值为KEY_POWER
    input_set_capability(input, EV_KEY, KEY_POWER);

    // 获取GPIO和中断资源（这里简化，实际应从设备树获取）
    data->gpiod = gpiod_get(&pdev->dev, "power", GPIOD_IN);
    if (IS_ERR(data->gpiod))
        return PTR_ERR(data->gpiod);

    data->irq = gpiod_to_irq(data->gpiod);
    data->keycode = KEY_POWER;
    data->input = input;

    // 注册中断处理函数，双边沿触发
    ret = devm_request_irq(&pdev->dev, data->irq, gpio_key_isr,
                         IRQF_TRIGGER_RISING | IRQF_TRIGGER_FALLING,
                         "gpio-key-power", data);
    if (ret) {
        dev_err(&pdev->dev, "Failed to request IRQ: %d\n", ret);
        return ret;
    }

    // 4. 关键一步：向输入核心层注册设备
    ret = input_register_device(data->input);
    if (ret) {
        dev_err(&pdev->dev, "Failed to register input device: %d\n", ret);
        return ret;
    }

    platform_set_drvdata(pdev, data);
    dev_info(&pdev->dev, "GPIO power key registered\n");
    return 0;
}

这段代码展示了一个最小化的输入设备驱动框架。核心流程可以总结为以下四步：

分配并配置 input_dev
→
设置硬件能力 (EV_*)
→
在中断/轮询中上报事件
→
调用 input_register_device

编者提示：本例中使用了GPIO和中断。关于GPIO子系统的详细操作和中断处理程序的深入设计，请分别参考第20章《GPIO与Pinctrl子系统》和第5章《中断处理程序设计》。

8.4 上报不同类型的事件

输入子系统支持多种事件类型，你需要根据硬件特性选择正确的上报函数。如果你看到应用层读取到的坐标全是0或者键值混乱，大概率是上报函数用错了。

按键事件 (EV_KEY)

  • 函数: input_report_key(dev, code, value)
  • 参数: value=1按下，value=0释放
  • 示例: input_report_key(input, KEY_VOLUMEUP, 1);

相对位移 (EV_REL)

  • 函数: input_report_rel(dev, code, value)
  • 参数: value为相对变化量
  • 示例: input_report_rel(input, REL_X, 5); (鼠标右移5单位)

绝对坐标 (EV_ABS)

  • 函数: input_report_abs(dev, code, value)
  • 参数: value必须在absmin~absmax范围内
  • 示例: input_report_abs(input, ABS_X, 500); (触摸点X坐标)

开关/状态 (EV_SW)

  • 函数: input_report_switch(dev, code, value)
  • 参数: value=1开启，value=0关闭
  • 示例: input_report_switch(input, SW_LID, 1); (笔记本合盖)

一个关键细节：每次完整的事件上报（例如一次触摸包含X、Y、压力）后，必须调用input_sync(dev)。这个函数会生成一个类型为EV_SYN、代码为SYN_REPORT的特殊事件，它告诉事件处理层：“之前的一系列报告属于同一个事件包，现在可以发送给用户空间了”。忘记调用input_sync是新手最常见的错误之一，会导致事件丢失或延迟。

触摸屏事件上报示例

对于一个支持多点触控的电容屏（假设为FT5x06芯片），在中断中上报第一个触摸点的代码可能如下：

// 从硬件寄存器读取坐标和触摸状态
int touch_status = read_reg(TS_REG_STATUS);
int x = read_reg(TS_REG_X1);
int y = read_reg(TS_REG_Y1);

if (touch_status & TOUCH_DOWN) {
    input_report_abs(input, ABS_MT_TRACKING_ID, 0); // 指定触点ID
    input_report_abs(input, ABS_MT_POSITION_X, x);
    input_report_abs(input, ABS_MT_POSITION_Y, y);
    input_report_abs(input, ABS_MT_PRESSURE, 100);
    input_mt_sync(input); // 同步单个触点信息（MT协议）
    input_report_key(input, BTN_TOUCH, 1); // 报告有触摸发生
} else {
    input_mt_sync(input); // 无触点，同步一个空帧
    input_report_key(input, BTN_TOUCH, 0);
}
input_sync(input); // 关键！同步整个事件包

8.5 调试与查看输入设备

驱动编写完成后，如何验证它是否工作正常？Linux提供了丰富的工具。最直接的方法是查看/proc/bus/input/devices文件，它会列出所有已注册的输入设备及其能力。

$ cat /proc/bus/input/devices
I: Bus=0019 Vendor=0001 Product=0001 Version=0100
N: Name="gpio-keys"
P: Phys=gpio-keys/input0
S: Sysfs=/devices/platform/gpio-keys/input/input2
U: Uniq=
H: Handlers=sysrq kbd event2
B: PROP=0
B: EV=3
B: KEY=100000 0 0 0

这里B: EV=3表示设备支持EV_KEY (0x01)和EV_SYN (0x02)类型的事件。Handlers=event2表示该设备匹配到了evdev处理程序，并在用户空间创建了节点/dev/input/event2。你可以使用evtest或hexdump工具实时监听这个节点的事件流，这是验证驱动是否正确上报事件的终极方法。

动手试一试

1. 虚拟输入设备：在不连接真实硬件的情况下，编写一个内核模块，创建一个虚拟鼠标设备。在模块初始化时，模拟一次鼠标向右下方的移动（同时上报REL_X和REL_Y事件）。加载模块后，使用evtest /dev/input/eventX观察事件上报。
2. 分析真实设备：在你的开发板或PC上，运行cat /proc/bus/input/devices，选择一个设备（如内置键盘），记录其Name、EV和KEY字段的值。使用evtest监听对应的事件节点，按下几个键，观察终端输出的原始事件数据结构。

检验你的理解

  1. 输入子系统的三层架构中，设备驱动层的主要职责是什么？事件处理层的主要产出是什么？
  2. 在配置一个支持KEY_VOLUMEUP和KEY_VOLUMEDOWN的按键设备时，除了设置这两个键值位图，还必须设置哪个事件类型位图？
  3. 为什么在连续调用多个input_report_xxx()函数后，必须调用一次input_sync()？如果不调用会有什么后果？
  4. 对于一款分辨率为800x480的电阻触摸屏，在驱动中调用input_set_abs_params()设置X轴参数时，minimum、maximum、fuzz和flat参数应如何设置？

本章小结

  • 输入子系统采用设备驱动层、核心层、事件处理层三层架构，实现了输入设备的统一抽象与事件上报。
  • 编写输入设备驱动的核心是创建并配置struct input_dev，使用set_bit()或input_set_capability()明确声明设备支持的事件类型和具体键值/坐标轴。
  • 在中断或轮询例程中，使用input_report_key/rel/abs()等函数上报具体事件，并在一个完整事件包结束后必须调用input_sync()。
  • 通过/proc/bus/input/devices和evtest工具可以有效地调试和验证输入设备驱动是否工作正常。
  • 区分EV_REL（相对位移，如鼠标）和EV_ABS（绝对坐标，如触摸屏）是正确上报事件的关键。
