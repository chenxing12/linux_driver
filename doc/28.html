核心概念工作队列工作者线程

一个真实的故障现象：某个触摸屏驱动在中断处理函数里进行复杂的坐标校准计算，导致系统响应变慢，甚至偶尔出现“soft lockup”警告。中断上下文要求快速返回，不能睡眠，也不能进行可能阻塞的操作。这时，你需要一个机制，把耗时的任务“挪出去”执行。内核工作队列就是为解决这类问题而生的。

生活类比： 就像餐厅的后厨。顾客点单（中断发生）非常急迫，服务员（中断处理程序）立刻接下订单，但复杂的炒菜工作（耗时任务）会被写成菜单（工作项）贴到后厨的公告板（工作队列）上，由专门的厨师（工作者线程）来按顺序处理。

技术定义： 工作队列（workqueue）是Linux内核用于将任务（工作）的执行推迟到进程上下文的一种异步执行框架。它创建内核线程（工作者线程）来从队列中取出工作项并执行，允许任务睡眠、使用信号量等。

工作队列的两种创建方式与选择

内核提供了两种使用工作队列的路径：共享的全局工作队列和自定义的专用工作队列。选择哪一种，直接关系到驱动的性能和资源隔离性。

对比项共享工作队列 (keventd_wq)自定义工作队列
创建方式使用 schedule_work(&my_work)使用 create_workqueue(“my_queue”)
资源开销零开销，使用内核预创建线程池需要为每个CPU核心创建专属内核线程
执行延迟可能较高，需与系统其他任务排队较低，独占线程，响应及时
隔离性无隔离，一个任务阻塞可能影响其他驱动强隔离，一个队列阻塞不影响其他
适用场景任务不频繁、执行快、不阻塞的轻量级任务任务频繁、耗时、可能阻塞或要求低延迟的任务

我的经验是，对于99%的驱动场景，使用共享工作队列就足够了。除非你像开发一个高性能存储驱动，需要频繁提交大量I/O完成处理工作，否则创建自定义工作队列带来的复杂性往往得不偿失。内核默认的共享工作队列线程名为`kworker/uX:Y`，你可以用`ps aux | grep kworker`查看。

核心数据结构与执行流程

理解工作队列，需要抓住三个核心：工作（struct work_struct）、工作队列（struct workqueue_struct）和工作者线程。它们的关系构成了完整的执行链路。

一个典型的工作提交与执行流程如下：

定义工作结构体与处理函数
→
初始化工作（INIT_WORK）
→
在合适时机提交工作（schedule_work）
→
内核线程从队列取出并执行
→
函数执行完毕，工作项从队列移除

让我们看一个最基础的代码示例，它在中断处理函数中调度一个工作：

// 1. 定义工作结构体和工作处理函数
static struct work_struct my_work;
static void my_work_handler(struct work_struct *work) {
    printk(KERN_INFO "Work handler is running in process context, pid: %d\n", current->pid);
    // 这里可以调用可能睡眠的函数，如 kmalloc(GFP_KERNEL)
}

// 2. 在驱动初始化函数中初始化工作
static int __init my_init(void) {
    INIT_WORK(&my_work, my_work_handler);
    // ... 其他初始化，如申请中断
    return 0;
}

// 3. 中断处理函数（上半部）
static irqreturn_t my_interrupt(int irq, void *dev_id) {
    // 快速处理硬件相关操作
    clear_interrupt_flag();

    // 将耗时任务调度到工作队列
    schedule_work(&my_work);
    return IRQ_HANDLED;
}

关键点： 工作处理函数 `my_work_handler` 运行在进程上下文，其 `current` 宏指向的是 `kworker` 内核线程，而不是触发中断的用户进程。这意味着你不能直接通过 `current` 来访问用户空间数据。

⚠️ 常见误区

  • 在工作函数中访问用户空间： 工作函数由内核线程执行，与触发中断的用户进程无关联，直接 `copy_from_user` 会导致内核崩溃。需要在中断上半部将用户数据保存到内核空间。
  • 误用 `INIT_WORK` 导致重复初始化： 一个 `work_struct` 只能被初始化一次。如果工作已经在队列中 pending，再次调用 `INIT_WORK` 是危险的操作。通常只在模块初始化时调用一次。
  • 认为 `schedule_work` 会立即执行： 它只是将工作项放入队列并唤醒一个工作者线程。执行是异步的，可能有毫秒级的延迟，具体取决于系统负载。

延迟工作与更精细的控制

有时你不仅想把任务推迟到进程上下文，还想让它延迟一段时间再执行。这就是 `delayed_work` 的用途。它内部包含一个 `work_struct` 和一个定时器。

#include <linux/delay.h>
static struct delayed_work my_delayed_work;

static void delayed_handler(struct work_struct *work) {
    printk(KERN_INFO "Delayed work executed after 2 seconds.\n");
}

// 初始化延迟工作
INIT_DELAYED_WORK(&my_delayed_work, delayed_handler);

// 调度延迟工作，2000毫秒后执行
schedule_delayed_work(&my_delayed_work, msecs_to_jiffies(2000));

对于自定义工作队列，你拥有更强的控制力，比如创建、刷新和销毁。刷新操作 (`flush_workqueue`) 会阻塞当前进程，直到该队列中所有已提交的工作执行完毕。这在模块卸载时确保没有残留工作正在运行至关重要。

场景：网络数据包后处理

  • 网卡中断收到数据包。
  • 中断上半部快速将数据包DMA到内存环。
  • 调度一个工作到自定义工作队列 `net_rx_wq` 进行协议栈处理（如IP分片重组）。

场景：传感器数据批量上报

  • 传感器每10ms产生一次中断。
  • 中断中读取原始数据存入缓冲区。
  • 调度一个延迟工作，200ms后执行，对缓冲区的所有数据进行滤波、校准后一次性上报给用户层。

场景：硬件初始化探活

  • 驱动加载时，硬件可能未就绪。
  • 调度一个延迟工作，5秒后执行。
  • 在工作函数中检查硬件状态，若未就绪则尝试重新初始化或报错。

编者提示： 工作队列经常与第27章介绍的内核定时器进行比较。核心区别在于执行上下文：定时器回调在中断上下文执行，不能睡眠；工作队列函数在进程上下文执行，可以睡眠。对于需要获取锁或分配内存的延迟任务，工作队列是更安全的选择。

在实际项目中，我遇到过一个坑：使用 `schedule_delayed_work` 实现一个轮询任务，但在模块退出函数中只调用了 `cancel_delayed_work`。结果模块卸载后，之前调度的延迟工作有时仍会执行，导致访问已释放的内存而oops。正确的做法是使用 `cancel_delayed_work_sync`，它会取消并等待可能正在运行的工作完成。

工作队列的内部优化：并发管理队列（CMWQ）

在较新的内核版本（2.6.36之后），工作队列机制被重写为并发管理队列（Concurrency Managed Workqueue, CMWQ）。它自动管理工作者线程的数量和并发度，开发者无需关心。但你需要了解两个关键概念：

  • WQ_UNBOUND： 创建的工作者线程不绑定到特定CPU，可以在CPU间迁移，适用于CPU密集型且不关心缓存局部性的任务。
  • WQ_HIGHPRI： 高优先级工作队列，其工作者线程以更高的调度优先级运行（`MAX_RT_PRIO/2`），适用于低延迟要求的任务。

创建具有这些属性的自定义工作队列：

struct workqueue_struct *my_wq;
my_wq = alloc_workqueue("my_highpri_wq", WQ_HIGHPRI | WQ_UNBOUND, 0);
// 向该队列提交工作
// queue_work(my_wq, &my_work);

动手试一试

1. 基础实验： 编写一个内核模块，在初始化时调度一个延迟工作，5秒后向内核日志打印“Hello from delayed work!”。观察 `dmesg` 输出，并使用 `ps` 命令查看执行此工作的 `kworker` 线程。

2. 进阶实验： 修改上述模块，在中断处理函数中（可以模拟一个定时器中断或使用一个GPIO中断）调度一个普通工作。在工作处理函数中，尝试调用 `msleep(100)`，验证其可以睡眠。对比一下，如果把这个 `msleep` 放在中断上半部会发生什么？

3. 排错实验： 故意在工作处理函数中加入一个错误：`*(int *)0 = 1;` 制造一个内核oops。分析oops信息，看看崩溃发生在哪个进程上下文（`kworker` 线程），并与直接在中断处理函数中制造oops的情况进行对比。

本章小结

  • 工作队列是将任务延迟到进程上下文执行的核心机制，解决了中断上下文不能睡眠和耗时的问题。
  • 优先使用共享工作队列（`schedule_work`）处理轻量级任务；仅在任务频繁、耗时或需隔离时创建自定义工作队列。
  • 掌握延迟工作（`delayed_work`）用于定时调度，并在模块清理时使用 `*_sync` 系列函数确保工作完成，防止访问释放的内存。
  • 工作处理函数中不能直接访问用户空间内存，数据传递需在内核空间完成。
  • 理解CMWQ模型下的 `WQ_UNBOUND` 和 `WQ_HIGHPRI` 标志，用于优化CPU绑定和任务优先级。
