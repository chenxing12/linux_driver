核心概念复合设备驱动IIO触发器

想象一下智能家居的温湿度光照传感器，它内部可能集成了三个芯片：一个I2C接口的温湿度传感器（如SHT30），一个SPI接口的光照传感器（如VEML7700），还有一个GPIO控制的状态指示灯。为这样一个“综合传感器”写驱动，不是把三个独立驱动拼起来那么简单。你需要一个顶层驱动来协调它们，管理统一的电源、提供合并的数据上报接口。我在一个工业物联网项目中就踩过坑，最初写了三个独立模块，结果系统启动时加载顺序错乱，传感器初始化失败。

综合传感器驱动架构设计

第一个关键决策是驱动架构。你不能让三个传感器在系统中表现为三个独立的设备节点，那样用户空间应用需要分别打开`/dev/iio:device0`、`device1`、`device2`来读取数据，效率低下且难以同步。我们的目标是创建一个`/dev/iio:device0`，它能同时提供温度、湿度、光照三个通道的数据。

技术选型：为什么用IIO？

工业I/O子系统是管理传感器的不二之选。它提供了标准化的数据通道（channel）、采样频率、触发缓冲等机制。对于我们的综合传感器，可以定义三个IIO通道：

// 通道定义示例
static const struct iio_chan_spec sensor_channels[] = {
    {
        .type = IIO_TEMP,
        .info_mask_separate = BIT(IIO_CHAN_INFO_PROCESSED),
        .scan_index = 0,
        .scan_type = { // 定义数据在缓冲区的格式
            .sign = 's',
            .realbits = 16,
            .storagebits = 16,
            .endianness = IIO_CPU,
        },
    },
    // 湿度、光照通道定义类似...
};

这样，用户空间通过`/sys/bus/iio/devices/iio:deviceX/in_temp_input`就能读取温度，或者使用`read()`系统调用从缓冲区一次性获取所有通道的同步数据。

多总线设备管理与初始化

我们的虚拟设备连接在三个不同的总线上，设备树描述是关键。实际开发中，大部分团队会采用一个“聚合节点”来引用各个子节点。

// 设备树示例 (DTS)
&sensor_hub {
    compatible = “vendor,multi-sensor-1.0”;
    status = “okay”;
    // 引用I2C子设备
    temp-humidity = &i2c2 {
        sht30@44 {
            compatible = “sensirion,sht3x”;
            reg = <0x44>;
            status = “okay”;
        };
    };
    // 引用SPI子设备
    ambient-light = &spi1 {
        veml7700@0 {
            compatible = “vishay,veml7700”;
            reg = <0>;
            spi-max-frequency = <1000000>;
        };
    };
    // 使用GPIO
    led-gpio = <&gpio4 12 GPIO_ACTIVE_HIGH>;
};

在驱动初始化函数`probe()`中，你需要依次获取这些资源：

这里有个关键函数：`of_find_i2c_device_by_node()`和`of_find_spi_device_by_node()`。它们通过设备树节点找到内核中已经实例化的I2C或SPI设备（即子设备驱动`probe`后生成的`struct device`）。拿到子设备的`struct device`后，你就能通过`dev_get_drvdata()`获取到子设备驱动提供的操作句柄。如果看到报错`“OF: ERROR: could not get /sensor_hub/temp-humidity/sht30@44 device”`，大概率是子设备驱动加载失败或兼容字符串不匹配。

常见误区

  • 重复注册IIO设备：子设备驱动（如`sht3x`）本身可能也会注册一个IIO设备。你需要修改子设备驱动（或使用它的私有API），使其在主驱动模式下不自行注册，而是将操作函数集交给主驱动。
  • 电源管理混乱：三个传感器可能共用同一个电源引脚（如`VCC_SENSOR`）。在驱动中简单调用各自子驱动的`pm_runtime_put()`可能导致电源被意外关闭。应由主驱动统一管理电源，使用`regulator` API。
  • 并发数据访问不同步：当用户空间线程同时读取温度和光照时，两个I2C/SPI传输可能同时发生，造成总线冲突。必须在主驱动层用互斥锁保护对底层总线的访问序列。

数据同步与触发器模式

对于需要严格同步采样的场景（例如同时记录温湿度计算露点），IIO的触发器模式是必备的。你可以配置一个硬件定时器或GPIO中断作为触发器源。

在`read_raw`回调中，你需要根据通道类型调用对应的子设备驱动函数。我在项目中抽象了一个子设备操作表：

struct sensor_ops {
    int (*read_sample)(void *priv, int *val);
    void *priv_data; // 子设备驱动的私有上下文
};

static int multi_sensor_read_raw(struct iio_dev *indio_dev,
                             struct iio_chan_spec const *chan,
                             int *val, int *val2, long mask) {
    struct multi_sensor_data *data = iio_priv(indio_dev);
    int ret;
    const struct sensor_ops *ops;

    switch (chan->type) {
        case IIO_TEMP:
            ops = &data->temp_ops; // 指向SHT30的读函数
            break;
        // ... 其他通道
    }

    mutex_lock(&data->bus_lock);
    ret = ops->read_sample(ops->priv_data, val);
    mutex_unlock(&data->bus_lock);
    return ret;
}

编者提示：关于IIO子系统更详细的通道定义、缓冲区和触发器设置，请回顾第18章《IIO工业IO子系统》。本章重点在于如何将多个IIO设备聚合管理。

高精度数据采集

  • 启用IIO硬件缓冲（kfifo）
  • 配置定时器触发器，设置采样率（如10Hz）
  • 用户空间使用`libiio`库异步读取批量数据

低功耗物联网设备

  • 主驱动实现`pm_runtime_ops`
  • 无读取时自动关闭传感器电源（regulator）
  • 利用GPIO中断唤醒，而非轮询

带状态反馈的设备

  • 通过`sysfs`导出GPIO控制接口（如`/sys/class/multi_sensor/led`）
  • 在驱动中实现`sysfs_ops`，允许用户控制指示灯
  • 采样异常时，驱动内部自动闪烁LED告警

最后，别忘了电源管理。对于电池供电的设备，你需要在驱动中实现`pm_runtime`回调。当所有传感器都空闲超过预设时间（比如5秒），主驱动依次调用子设备的挂起例程，然后关闭公共的`regulator`。这个协同关闭的逻辑必须放在主驱动，因为子设备不知道彼此的状态。

动手试一试

基于QEMU的`vexpress-a9`虚拟开发板，模拟一个综合传感器。假设其包含：一个虚拟I2C温度计（地址0x48，返回固定值25°C）、一个虚拟SPI气压计（返回1013 hPa）、一个虚拟GPIO连接的绿色LED。

  1. 编写设备树片段，描述这个虚拟复合设备。
  2. 编写主驱动，在`probe`函数中打印成功找到子设备的信息。
  3. 实现IIO的`read_raw`回调，当读取温度通道时，模拟一次I2C传输（打印日志即可）；读取气压通道时，模拟SPI传输。
  4. 添加一个`sysfs`属性文件`/sys/bus/iio/.../led_brightness`，写入0或1可以控制虚拟LED的开关状态（打印日志）。

提示：你可以使用内核的`dummy_i2c`和`spi-dummy`模块来模拟底层总线设备。

本章小结

  • 架构是核心：综合传感器驱动应采用“主-从”架构，主驱动作为IIO设备注册，统一管理子设备并聚合数据流。
  • 设备树是粘合剂：通过设备树节点引用，将分散在不同总线上的物理设备在逻辑上聚合为一个复合设备，驱动据此获取所有子设备句柄。
  • IIO提供标准化接口：利用IIO的通道、缓冲区和触发器机制，可以高效、同步地暴露多传感器数据，并支持复杂的采集模式。
  • 并发与电源需统一管理：主驱动必须用互斥锁序列化对底层总线的访问，并统一协调所有子设备的电源状态，避免冲突和功耗浪费。
  • 调试关注加载顺序与资源获取：驱动加载失败通常源于子设备未就绪或资源获取函数（如`of_find_i2c_device_by_node`）返回NULL，需按依赖关系调整模块加载顺序或检查设备树。
