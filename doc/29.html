核心概念file_operationsinode

内核日志里出现“Unable to handle kernel NULL pointer dereference at virtual address 00000000”，而你正在操作一个自己创建的虚拟文件——这通常是file_operations结构体中的某个函数指针未初始化。VFS（Virtual Filesystem Switch）是Linux内核中一个精妙的抽象层，它让上层的系统调用（如open、read、write）能够以统一的方式访问下层千差万别的具体文件系统（如ext4、proc、tmpfs）或驱动设备。对于驱动开发者而言，理解并实现VFS接口，意味着你的设备可以通过熟悉的文件路径（如/dev/mydevice或/proc/driver/status）被用户空间程序访问。

什么是VFS？

生活类比：就像手机上的“文件”应用，无论你查看的是内部存储的照片、SD卡里的文档，还是网盘里的视频，都用统一的界面进行打开、删除、重命名操作。VFS就是这个统一的“文件”应用，它定义了一套标准接口，让内核屏蔽了底层存储介质的差异。

技术定义：VFS是Linux内核中的一个软件层，它提供了一组通用的数据结构和函数调用接口（主要是file_operations），用于抽象所有文件系统的共同操作。当用户程序执行read系统调用时，VFS会根据文件路径找到对应的inode，然后调用该文件所属文件系统或驱动提供的具体read函数。

我在一个传感器数据采集驱动中踩过坑：在open函数里直接访问了struct file的f_inode成员，却忽略了dentry可能为NULL的情况（比如通过某些特殊路径访问时）。正确的做法是使用file_inode(file)这个辅助宏，它内部做了安全检查。

驱动开发者的核心：file_operations

对于驱动开发者，与VFS打交道最主要的方式就是实现一个struct file_operations结构体，并将其赋值给inode或file的i_fop/f_op指针。这个结构体定义了你所“虚拟”的文件支持哪些操作。

// 一个简单的只读设备驱动示例
static const struct file_operations mydev_fops = {
    .owner = THIS_MODULE,
    .open = mydev_open,
    .release = mydev_close,
    .read = mydev_read,
    // .write 未定义，因此写操作将返回 -EBADF
    .llseek = no_llseek, // 告知VFS此设备不支持寻址
};

static int mydev_open(struct inode *inode, struct file *file)
{
    struct mydev_data *data = container_of(inode->i_cdev, struct mydev_data, cdev);
    file->private_data = data; // 关键：存储设备私有数据
    return 0;
}

static ssize_t mydev_read(struct file *file, char __user *buf, size_t count, loff_t *ppos)
{
    struct mydev_data *data = file->private_data;
    if (copy_to_user(buf, data->buffer, min(count, data->size)))
        return -EFAULT;
    return min(count, data->size);
}

关键点：file->private_data

这是驱动开发中最重要的字段之一。在open函数中，你将代表设备实例的数据结构（通常包含硬件寄存器地址、缓冲区、锁等）赋值给file->private_data。在后续的read、write、ioctl等函数中，你可以直接从中取出，避免了每次都要通过inode或全局链表去查找设备实例的开销和竞争。实际开发中，95%的驱动都采用这种模式。

常见误区

  • 混淆inode和file：inode代表文件本身（唯一的），file代表一次打开实例（可多个）。修改文件属性（如权限）应操作inode；记录当前读取位置（f_pos）应操作file。
  • 忽略并发访问：你的read/write函数可能被多个进程同时调用。如果操作共享硬件或缓冲区，必须使用第3章介绍的互斥锁（mutex）或自旋锁（spinlock）进行保护。
  • 错误处理不完整：ioctl命令中，如果传入的用户空间指针非法，必须使用access_ok()和copy_from_user()等函数检查并返回-EFAULT，而不是直接解引用。

创建文件：从inode到dentry

仅仅实现file_operations还不够，你需要让VFS知道在哪个路径下能找到你的文件。对于字符设备，这通常由mknod命令或udev在/dev下创建设备节点完成。但在内核中主动创建文件（例如在procfs中报告驱动状态），你需要理解创建过程。

以最常用的proc文件系统为例，创建一个只读状态文件：

// 定义文件操作
static int status_show(struct seq_file *m, void *v)
{
    seq_printf(m, "Interrupts: %lu\n", mydev_stats.interrupt_count);
    seq_printf(m, "Buffer usage: %d/%d\n", mydev_stats.buf_used, MAX_BUF);
    return 0;
}

static int status_open(struct inode *inode, struct file *file)
{
    return single_open(file, status_show, NULL);
}

static const struct file_operations status_fops = {
    .owner = THIS_MODULE,
    .open = status_open,
    .read = seq_read,
    .llseek = seq_lseek,
    .release = single_release,
};

// 在模块初始化时创建文件
struct proc_dir_entry *entry;
entry = proc_create("driver/mydev_status", 0444, NULL, &status_fops);
if (!entry) {
    pr_err("Failed to create proc entry\n");
    return -ENOMEM;
}

执行后，用户就能通过cat /proc/driver/mydev_status查看驱动状态。这里有个隐藏的坑：proc_create的第二个参数是权限模式（如0444表示只读），如果你错误地设置了可写位（如0666），但file_operations中没有定义.write函数，用户尝试写入时并不会出错，但写入的数据会被静默丢弃。

编者提示：关于procfs和sysfs更详细的操作，包括如何创建目录、读写文件的最佳实践，请参考第25章《Proc与Sys文件系统》。本章重点在于理解其底层共通的VFS接口机制。

高级话题：自定义文件系统类型

有时，驱动需要管理的不是单个文件，而是一整套具有复杂关系的虚拟文件目录结构（例如模拟一个FAT格式的ROM镜像）。这时，你可以注册一个自定义的文件系统类型。这涉及实现struct file_system_type和挂载时的struct super_operations。

场景：只读配置镜像

  • 硬件有一个SPI Flash，存储了设备的配置树（类似设备树）。
  • 驱动将其解析为/sys/firmware/mydevice/下的目录和文件。
  • 用户通过cat读取配置，但无法修改。
  • 实现建议：使用sysfs或configfs，而非从头实现一个文件系统。

场景：虚拟块设备

  • 驱动将一段加密的内存区域模拟成一个磁盘（如/dev/vault）。
  • 支持分区、格式化（ext4）、挂载到/mnt/vault。
  • 实现建议：基于第13章的块设备驱动框架，VFS部分由内核块层处理。

场景：调试信息瀑布流

  • 驱动产生大量实时调试日志，像“tail -f”一样持续读取。
  • 多个进程可以同时打开文件，各自独立读取流。
  • 实现关键：在read函数中实现等待队列（wait_queue），当无数据时让进程休眠。

我参与过一个视频采集卡项目，需要将多个摄像头的实时状态（分辨率、帧率、信号强度）通过一个目录（/sys/class/video4linux/videoX/status/）下的多个文件暴露出去。我们选择了sysfs，因为它与内核设备模型（kobject）天然集成，自动处理了文件创建和销毁，比直接操作VFS底层接口省心得多。

动手试一试

1. 基础练习：编写一个内核模块，在/proc下创建一个名为hello_vfs的文件。当用户读取该文件时，返回字符串“Hello from VFS layer!\n”以及当前的jiffies值（内核时间戳）。确保模块加载时创建文件，卸载时删除。

2. 进阶挑战：扩展上述模块，使/proc/hello_vfs文件可写。当用户向该文件写入一个数字N（如echo 100 > /proc/hello_vfs）时，模块内部记录这个值。当再次读取时，除了问候语和jiffies，额外输出“You wrote: N”。注意：用户传入的是字符串，你需要用kstrtoint()将其转换为整数，并处理错误。

排错线索：如果写入后读取不到值，检查你的.write函数是否正确更新了模块内的全局变量或通过private_data传递的结构体。确保使用了正确的内存屏障或锁来防止并发读写冲突。

本章小结

  • VFS是抽象层：它通过super_block, inode, dentry, file四个核心对象，为所有文件系统和设备驱动提供统一的文件操作视图。
  • 驱动接口是file_operations：实现这个结构体中的函数（如open, read, write, release）是驱动提供文件访问能力的主要方式。务必正确初始化所有用到的函数指针。
  • 利用file->private_data：这是在文件打开实例中存储设备特定数据的最标准、最有效的方式，贯穿整个文件操作的生命周期。
  • 创建文件有捷径：对于简单的信息导出，优先使用procfs、sysfs等现有虚拟文件系统，它们提供了更简单的API（如proc_create），避免直接操作复杂的VFS底层接口。
  • 并发与安全是底线：VFS接口函数天生可重入，必须考虑多进程并发访问，使用锁保护共享数据，并对用户空间指针进行严格校验。
