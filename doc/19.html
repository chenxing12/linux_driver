核心概念LED ClassLED Trigger

想象一下家里的智能音箱，它有多个指示灯：电源、网络、麦克风状态。如果每个灯都由不同的程序直接控制GPIO，代码会混乱不堪。Linux内核的LED子系统就是为统一管理这些“灯”而设计的管家，它提供了标准化的接口，让应用层可以轻松控制，也让驱动开发者无需重复造轮子。

LED子系统是什么？

生活类比：就像一个智能家居的灯光控制中心，你不需要知道客厅灯具体连在哪个电路开关上，只需通过手机App选择“阅读模式”或“影院模式”。
技术定义：Linux内核中一个为发光二极管（LED）设备提供统一抽象、管理和控制接口的框架。它将具体的硬件操作（如GPIO翻转）与上层逻辑（如闪烁模式）解耦。

在项目中，我曾遇到一个需求：让一个LED根据系统CPU负载闪烁。如果自己从头实现定时器和亮度计算会很麻烦，而LED子系统内置的`heartbeat` trigger一行配置就解决了。这个框架的价值在于，它把常见的LED行为模式（称为Trigger）都标准化了。

LED子系统架构全景

整个子系统采用典型的分层设计，核心目标是隔离硬件差异，提供统一用户接口。驱动开发者只需关注最底层的硬件操作。

用户空间 (Userspace)
↓
sysfs接口
↓
LED Class Core
⇄
LED Triggers
↓
LED硬件驱动
↓
实际硬件 (GPIO/PWM等)

从上到下看：用户通过`/sys/class/leds/`目录下的文件控制LED；LED核心层管理所有注册的LED设备；Trigger定义了LED的行为逻辑；最底层是你的驱动，负责将“亮”、“灭”或“亮度值”翻译成具体的硬件操作。

关键数据结构：struct led_classdev

这是驱动开发者需要填充的核心结构体，定义在``中。它代表了一个具体的LED设备。

struct led_classdev {
    const char *name; // LED名字，如 "power:green"
    int brightness; // 当前亮度 (0~max_brightness)
    int max_brightness; // 最大亮度，简单LED设为1
    int flags; // 标志位，如 LED_CORE_SUSPENDRESUME
    /* 核心函数：设置亮度 */
    void (*brightness_set)(struct led_classdev *led_cdev,
                          enum led_brightness brightness);
    /* 更多可选函数... */
};

你需要实现`brightness_set`回调函数，当用户修改sysfs中的`brightness`文件时，内核就会调用它。

编写一个GPIO LED驱动

我们从最简单的场景开始：控制一个连接到GPIO的LED。假设在RK3568开发板上，用户LED连接在GPIO0_B7这个引脚上。

以下是完整的驱动代码示例：

// 必要的头文件
#include 
#include 
#include  // 使用GPIO描述符API
#include 

struct gpio_led_data {
    struct led_classdev cdev;
    struct gpio_desc *gpiod;
};

static void gpio_led_set(struct led_classdev *led_cdev,
                       enum led_brightness value) {
    struct gpio_led_data *led_data =
        container_of(led_cdev, struct gpio_led_data, cdev);
    // 将亮度值转换为GPIO电平：非0亮，0灭
    gpiod_set_value(led_data->gpiod, value ? 1 : 0);
}

static int gpio_led_probe(struct platform_device *pdev) {
    struct device *dev = &pdev->dev;
    struct gpio_led_data *led_data;
    int ret;

    led_data = devm_kzalloc(dev, sizeof(*led_data), GFP_KERNEL);
    if (!led_data)
        return -ENOMEM;

    // 从设备树获取GPIO，名字为"led"
    led_data->gpiod = devm_gpiod_get(dev, "led", GPIOD_OUT_LOW);
    if (IS_ERR(led_data->gpiod))
        return PTR_ERR(led_data->gpiod);

    // 填充LED设备结构
    led_data->cdev.name = "user-led";
    led_data->cdev.max_brightness = 1; // 只有开和关
    led_data->cdev.brightness_set = gpio_led_set;
    led_data->cdev.default_trigger = "heartbeat"; // 默认心跳闪烁

    // 注册到LED子系统
    ret = devm_led_classdev_register(dev, &led_data->cdev);
    if (ret) {
        dev_err(dev, "Failed to register LED: %d\n", ret);
        return ret;
    }

    platform_set_drvdata(pdev, led_data);
    dev_info(dev, "GPIO LED driver probed successfully\n");
    return 0;
}

// 设备树匹配表（省略模块加载和平台驱动注册的样板代码）
static const struct of_device_id gpio_led_of_match[] = {
    { .compatible = "mycompany,gpio-led" },
    {},
};
MODULE_DEVICE_TABLE(of, gpio_led_of_match);

对应的设备树节点应该这样写：

// 在RK3568的DTS文件中
user_led: led {
    compatible = "mycompany,gpio-led";
    label = "User LED";
    gpios = <&gpio0 RK_PB7 GPIO_ACTIVE_HIGH>; // GPIO0_B7，高电平亮
    /* 可选：指定默认trigger */
    /* linux,default-trigger = "heartbeat"; */
};

常见误区

  • 混淆brightness值与GPIO电平：在`brightness_set`函数中，参数`value`是亮度值（0~max_brightness），你需要根据硬件逻辑将其转换为具体的电平。对于简单的开关LED，`value > 0`通常意味着点亮。
  • 忘记处理max_brightness：如果你的LED支持PWM调光（比如max_brightness=255），但驱动只实现了开关，用户设置中间亮度值（如128）时，你的驱动可能无法正确处理。实际开发中，大部分支持PWM的LED芯片驱动会使用`brightness_set_blocking`函数。
  • 资源泄漏：使用`devm_led_classdev_register`和`devm_gpiod_get`等托管函数（managed API）可以自动在设备注销时释放资源，避免在`remove`函数中忘记调用`led_classdev_unregister`。

强大的Trigger机制

生活类比：就像给灯设置了一个智能定时器或感应器，你可以设定“有人移动时亮”、“每隔5秒闪一下”、“CPU忙时快闪”，而无需自己不断写程序去控制开关。
技术定义：LED Trigger是内核中预定义的一些规则，它们可以自动改变LED的亮度状态，触发条件可以是定时器、磁盘活动、网络流量、CPU负载等。

Trigger是LED子系统最实用的功能之一。内核已经内置了许多有用的trigger，你可以在`/sys/class/leds//trigger`文件中查看和选择。

常用内置LED Trigger及其用途
Trigger名称触发条件/行为典型应用场景是否需要驱动额外支持
none无自动触发，完全由用户控制手动开关指示灯否
timer固定频率闪烁（可设置延迟时间）设备待机指示、报警闪烁否
heartbeat模拟心跳（双闪，间隔随系统负载变化）系统运行状态指示否
disk-activity磁盘有读写操作时闪烁硬盘活动指示灯否
cpu0对应CPU核心忙碌时亮起多核CPU负载可视化否
netdev网络设备有数据收发时闪烁网络活动指示灯需要配置网络设备名
gpio监视另一个GPIO的电平按键背光、外部事件指示需要指定GPIO号

你可以在驱动中通过`led_cdev.default_trigger`指定默认trigger，用户也可以在系统运行时通过sysfs动态切换。例如，让一个LED显示CPU0的负载：

# 查看当前trigger
$ cat /sys/class/leds/user-led/trigger
[none] timer heartbeat cpu0 disk-activity ...

# 切换到cpu0 trigger
$ echo cpu0 > /sys/class/leds/user-led/trigger
$ cat /sys/class/leds/user-led/trigger
none timer heartbeat [cpu0] disk-activity ...

编者提示：Trigger机制依赖于内核的其他子系统。例如，`disk-activity`需要块设备层支持，`netdev`需要网络子系统。理解这些关联有助于调试，当某个trigger不工作时，可能是相关内核配置未开启（如`CONFIG_LEDS_TRIGGER_DISK_ACTIVITY`）。

进阶：支持PWM调光的LED驱动

许多现代开发板（如树莓派、i.MX8MP）的LED支持PWM调光，可以实现呼吸灯效果。这时，你需要处理的是连续的亮度值，而不仅仅是0和1。

简单GPIO LED

  • max_brightness = 1
  • 实现 brightness_set
  • 仅开关两种状态

PWM调光LED

  • max_brightness = 255
  • 实现 brightness_set_blocking
  • 需获取PWM设备，设置占空比

多色LED (RGB)

  • 注册多个led_classdev
  • 如：red, green, blue
  • 或使用multicolor类

硬件复杂LED

  • 如LED驱动芯片（LP50xx, AW20xx）
  • 通过I2C/SPI通信
  • 需实现模式寄存器配置

对于PWM LED，核心是使用`brightness_set_blocking`回调，因为它可能涉及睡眠操作（PWM配置）。驱动需要获取PWM设备并设置占空比。一个常见的坑是：PWM的周期（period）设置不当会导致肉眼可见的闪烁。我在一个使用i.MX6ULL的项目中，将PWM周期设置为5ms（200Hz）以上，才消除了低频闪烁感。

static int pwm_led_set(struct led_classdev *led_cdev,
                      enum led_brightness brightness) {
    struct pwm_led_data *led_data = ...;
    struct pwm_device *pwm = led_data->pwm;
    int duty_cycle;

    // 计算占空比：亮度值 / 最大亮度 * PWM周期
    duty_cycle = (brightness * led_data->period) / led_data->cdev.max_brightness;
    // 配置PWM，注意pwm_config可能引起调度（睡眠）
    pwm_config(pwm, duty_cycle, led_data->period);
    if (brightness == 0) {
        pwm_disable(pwm);
    } else {
        pwm_enable(pwm);
    }
    return 0;
}
// 在probe中赋值
led_data->cdev.brightness_set_blocking = pwm_led_set;

实际开发中，更推荐使用内核提供的`leds-pwm`通用驱动（`drivers/leds/leds-pwm.c`），它已经完美处理了PWM调光的细节，你只需要在设备树中正确引用PWM节点即可，无需自己编写驱动。

动手试一试

1. 在QEMU或真实开发板上实验：如果你有一个带有可编程LED的开发板（如树莓派、BeagleBone或任何一款主流嵌入式板），找到其用户LED对应的GPIO。编写一个最简单的LED驱动模块（参考本章的GPIO LED示例），编译并加载。观察`/sys/class/leds/`目录下是否出现你的设备。
2. 玩转Trigger：加载驱动后，尝试通过sysfs切换不同的trigger： 

echo timer > /sys/class/leds/user-led/trigger
cat /sys/class/leds/user-led/delay_on  # 查看亮的时间
echo 100 > /sys/class/leds/user-led/delay_on  # 设置为100ms

观察LED的闪烁模式变化。
3. 实现自定义Trigger（可选）：如果你有兴趣，可以查阅内核源码`drivers/leds/trigger/`目录，看看`ledtrig-heartbeat.c`是如何实现的。尝试模仿它，写一个最简单的`mytrigger`，让LED以固定的三短一长模式闪烁。

本章小结

  • LED子系统是内核统一管理指示灯的标准框架，采用“驱动-核心-Trigger-用户接口”的分层架构，极大简化了LED控制逻辑。
  • 编写LED驱动的核心是定义并注册一个`struct led_classdev`结构体，必须实现其`brightness_set`（或`brightness_set_blocking`）回调函数，将抽象的亮度值转换为具体的硬件操作。
  • Trigger机制是LED子系统的精髓，它允许LED根据系统事件（定时、心跳、磁盘活动、CPU负载等）自动改变状态，无需应用层持续干预，内核已内置多种实用trigger。
  • 对于复杂LED（如PWM调光、多色RGB、I2C控制芯片），驱动需要处理连续亮度值或寄存器配置，但核心注册流程与GPIO LED一致，应优先考虑使用内核已有的通用驱动（如`leds-pwm`）。
  • 通过sysfs接口（`/sys/class/leds//`）可以动态查询和修改LED的亮度、最大亮度、当前trigger及trigger参数，这是用户空间与LED驱动交互的标准方式。
