凌晨三点，你的手机响了。测试同事告诉你，为RK3568开发板编写的I2C触摸屏驱动，在最新的Linux 6.1内核上编译失败，报错implicit declaration of function ‘i2c_new_device’。你打开电脑，意识到这不是一个简单的bug修复，而是驱动移植的典型开局。

为什么需要关注移植与兼容性？

Linux内核是活着的代码，平均每2-3个月发布一个主版本，API和内部数据结构持续演进。同时，嵌入式硬件平台（如NXP i.MX、TI AM62x、瑞芯微RK系列）层出不穷。一个商业驱动项目，其生命周期内几乎必然面临内核升级或硬件更换。我在一个车载项目里，就经历过从Linux 4.19到5.10再到5.15的三次内核大版本移植，每次都是对驱动稳定性的严峻考验。

内核API变更：从追踪到适配

内核API变更主要有三类：函数/宏被移除或改名、参数列表改变、数据结构成员变化。盲目搜索错误信息往往效率低下，你需要系统性的方法。

首先，官方ChangeLog是最高效的线索。以开头的错误为例，搜索Linux内核源码的`Documentation/process/changes.rst`或发布公告，你会发现在5.17版本后，`i2c_new_device`被标记为已弃用，推荐使用`i2c_new_client_device`。新函数返回的是指针而非整数，这直接影响了错误处理逻辑。

Linux 4.x 到 6.x 常见驱动API变更示例
变更内容影响版本替代方案关键差异
dma_alloc_coherent 参数变化5.0+使用 struct device *dev 参数不再需要 gfp_t gfp 单独参数

常见误区

  • 盲目添加 `#ifdef`：为了兼容旧内核，在代码里到处添加 `#if LINUX_VERSION_CODE < KERNEL_VERSION(5,17,0)` 是最糟糕的做法。这会让代码迅速变成难以维护的“补丁墙”。
  • 忽略编译警告：内核升级后，`deprecated` 警告是API即将变更的明确信号。我曾因为忽略一个关于 `create_singlethread_workqueue` 的警告，导致在下一个内核版本驱动直接崩溃。
  • 只测试编译，不测试运行：API变更有时是语义上的。比如 `platform_get_irq` 的返回值逻辑改变，编译能过，但运行时中断注册会失败。

正确的做法是构建一个集中的兼容层。在驱动源文件中（或一个独立的头文件如 `compat.h`）封装这些差异。例如，处理上述I2C API变更：

// 兼容层头文件 compat.h 中的示例
#if LINUX_VERSION_CODE < KERNEL_VERSION(5, 17, 0)
static inline struct i2c_client *compat_i2c_new_device(struct i2c_adapter *adap, struct i2c_board_info const *info) {
    return i2c_new_device(adap, info);
}
#else
static inline struct i2c_client *compat_i2c_new_device(struct i2c_adapter *adap, struct i2c_board_info const *info) {
    return i2c_new_client_device(adap, info);
}
#endif

// 在你的驱动代码中，统一使用 compat_i2c_new_device
struct i2c_client *client = compat_i2c_new_device(adapter, &board_info);
if (IS_ERR(client)) { // 注意：新老API都需要用IS_ERR判断
    return PTR_ERR(client);
}

平台抽象：硬件差异的隔离

生活类比：就像给不同品牌的电视机（硬件）配同一个型号的万能遥控器（驱动），遥控器内部有芯片（平台抽象层）来翻译通用指令为特定电视的红外信号。技术定义：平台抽象是通过一组标准化的接口和数据结构，将驱动核心逻辑与具体的硬件寄存器操作、中断号、时钟源、DMA通道等底层细节分离开。

假设你有一个基于TI AM335x的触摸屏驱动，现在要移植到NXP i.MX6ULL上。两者都是ARM Cortex-A内核，但外设控制器（I2C、GPIO）、时钟树、电源管理寄存器完全不同。硬编码的寄存器地址和操作序列会让移植变成重写。

时钟与电源

  • AM335x使用 `cm_per` 模块管理外设时钟
  • i.MX6ULL使用CCM时钟控制器
  • 抽象接口：`enable_clk()`, `set_clk_rate()`

引脚复用

  • AM335x通过 `padconf` 寄存器配置
  • i.MX6ULL通过IOMUXC模块配置
  • 抽象接口：`config_pin_mux()`

中断控制器

  • AM335x使用INTC，中断号固定
  • i.MX6ULL使用GPC，支持层级中断
  • 抽象接口：`request_irq()` (内核已抽象)

寄存器访问

  • 不同基地址和位域定义
  • 可能涉及字节序差异
  • 抽象接口：`read_reg()`, `write_reg()` 宏

实现平台抽象的核心是定义一个 `struct platform_ops` 或类似的数据结构，将硬件相关操作作为函数指针填入。驱动核心代码只调用这些接口。

struct ts_platform_ops {
    int (*init)(struct device *dev);
    void (*exit)(struct device *dev);
    int (*read_reg)(struct device *dev, u8 addr, u16 *val);
    int (*write_reg)(struct device *dev, u8 addr, u16 val);
    int (*reset)(struct device *dev);
    int (*power)(struct device *dev, bool on);
};

// AM335x 的具体实现
static const struct ts_platform_ops am335x_ts_ops = {
    .init = am335x_ts_init,
    .read_reg = am335x_i2c_read_reg,
    // ... 填充其他函数
};

// i.MX6ULL 的具体实现
static const struct ts_platform_ops imx6ull_ts_ops = {
    .init = imx6ull_ts_init,
    .read_reg = imx6ull_i2c_read_reg,
    // ...
};

// 驱动探测函数中，根据设备树兼容性字符串选择操作集
if (of_device_is_compatible(node, "ti,am335x-tsc"))
    ts_dev->ops = &am335x_ts_ops;
else if (of_device_is_compatible(node, "fsl,imx6ull-tsc"))
    ts_dev->ops = &imx6ull_ts_ops;

编者提示：设备树（DTS）是平台抽象的关键支撑技术，它通过 `compatible` 字符串将硬件描述与驱动绑定。关于如何编写和设备树匹配的驱动，请回顾 第7章《设备树DTS语法解析》 和 第6章《平台设备驱动模型》。

构建可维护的兼容性代码

兼容性代码的质量直接决定了驱动长期维护的成本。你需要遵循几个原则：集中管理、明确版本边界、持续测试。

→

首先，集中管理意味着所有针对内核版本或平台差异的代码，都应放在少数几个文件中，比如 `compat.h` 和 `platform_xxx.c`。避免在业务逻辑代码中散落 `#ifdef`。

其次，明确版本边界。为你的驱动定义一个明确支持的内核版本范围，例如 `Linux 4.19 ~ 6.6`。在 `Kconfig` 或 `README` 中写明，并使用 `LINUX_VERSION_CODE` 进行条件编译，确保在不支持的内核上直接编译失败，而不是产生运行时错误。

// 在驱动主文件顶部进行版本检查
#if LINUX_VERSION_CODE < KERNEL_VERSION(4, 19, 0) || LINUX_VERSION_CODE >= KERNEL_VERSION(6, 7, 0)
#error "This driver is only supported on Linux kernel versions 4.19 to 6.6"
#endif

最后，持续测试是生命线。实际开发中，我建议使用Docker或虚拟机搭建一个简单的CI环境，针对你支持的每个主要内核版本（如5.4, 5.10, 5.15, 6.1, 6.6）进行编译测试。对于关键驱动，甚至需要准备不同架构（arm, arm64, x86）的测试镜像进行冒烟测试。

实战技巧：如何查找API变更历史

除了官方文档，最直接的方法是使用内核源码树中的 `git log`。例如，想知道 `timer_setup` 何时引入：
git log --oneline --grep="timer_setup" -- drivers/ | head -5
或者使用 `coccinelle` 语义补丁工具，它能帮你自动将旧API用法转换为新API。内核源码的 `scripts/coccinelle/` 目录下有很多现成的例子。

动手试一试

1. 创建兼容层：找一个你之前写的简单字符设备驱动（或使用一个开源示例）。在驱动目录下创建一个 `compat.h` 文件，将其中用到的 `copy_to_user`/`copy_from_user` 函数用兼容性宏包装，模拟它们在未来内核中返回值类型从 `long` 变为 `int` 的情况（这只是一个练习场景）。

2. 模拟平台差异：编写一个虚拟的“GPIO操作”平台抽象层。定义 `struct gpio_ops`，包含 `set_direction`, `get_value`, `set_value` 函数指针。然后创建两个实现文件：`gpio_simulated.c`（用打印信息模拟）和 `gpio_fake_hw.c`（用一个全局数组模拟硬件寄存器）。在主驱动中，通过模块参数选择使用哪个平台实现。

3. 版本边界测试：使用 `LINUX_VERSION_CODE` 宏，在你的驱动模块初始化函数里打印当前内核版本，并根据版本不同，选择调用一个旧的内核API（如 `class_create()` 的旧签名）或新的API（`class_create()` 的新签名）。尝试用不同版本的内核头文件编译它，观察结果。

检验你的理解

  1. 你的驱动在 Linux 5.10 上正常工作，但在 5.15 上编译报错 “`struct file_operations` has no member named `ioctl`”。你应该首先查阅什么资料来定位问题？
  2. 在设计一个需要支持三种不同品牌传感器芯片（均通过I2C通信，但寄存器定义不同）的驱动时，是应该写三个独立的 `.c` 文件，还是在一个 `.c` 文件里用大量的 `if (chip_id == A)` ？为什么？
  3. 你为驱动添加了兼容层以支持内核 4.19 和 5.15。在代码提交前，除了确保在两个版本上都能编译通过，还应该进行什么关键测试来避免运行时错误？

本章小结

  • 内核API是流动的：必须通过官方ChangeLog、`git log` 和 Coccinelle 等工具系统性地追踪和适配API变更，而不是被动地应对编译错误。
  • 抽象隔离硬件差异：通过定义平台操作集（`struct xxx_ops`）将驱动核心逻辑与具体的硬件操作解耦，这是实现跨平台移植的关键设计模式。
  • 兼容代码需要工程化管理：集中管理兼容层代码、明确定义支持的内核版本边界、并建立针对多版本内核的持续集成测试流程，是保证驱动长期可维护性的三大支柱。
  • 设备树是重要桥梁：利用设备树的 `compatible` 属性来匹配不同的硬件平台和对应的操作集，是实现平台抽象的通用且标准的方法。
