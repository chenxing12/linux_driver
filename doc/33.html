核心概念Crypto API硬件加密引擎

你的嵌入式设备需要加密存储用户数据，或者网络芯片需要硬件加速的SSL/TLS。直接调用用户态的OpenSSL？性能开销和上下文切换会让你望而却步。Linux内核提供了一套完整的加密框架——Crypto API，它允许驱动直接在内核空间高效、安全地使用软件或硬件加密服务。

为什么需要内核加密？

用户态加密库数据需要多次在用户空间和内核空间之间拷贝，对于网络数据包加密或块设备全盘加密这类高频操作，性能损耗巨大。内核加密实现了“零拷贝”或最少拷贝，并且是许多内核子系统（如DM-Crypt、IPsec）的基石。

33.1 内核加密框架（Crypto API）全景

Crypto API是一个分层、抽象的框架。它向上对内核其他模块（如网络栈、文件系统）提供统一的加密接口，向下管理并调度各种加密算法实现，无论是纯软件的（如ARMv8的CRC指令优化），还是特定硬件的（如NXP的CAAM、Intel的QAT）。

对于驱动开发者，我们主要与“算法实现层”打交道，目标是向Crypto API注册一个硬件加密设备的驱动。这里有两个关键概念：转换（Transform）和算法（Algorithm）。

生活类比

转换（Transform）：就像一个多功能厨房料理机（硬件），它提供“搅拌”、“切碎”、“榨汁”等具体功能。算法（Algorithm）：就是“搅拌”这个功能本身，它有明确的输入（食材）、输出（糊状物）和操作规范。

技术定义：一个crypto_alg结构体描述了一个加密算法（如AES-CBC）的属性和函数指针。一个硬件设备可能实现多个算法，注册多个crypto_alg实例。

33.2 编写一个硬件加密引擎驱动

假设我们为一块集成了AES-256-GCM和SHA-256加速器的SoC（例如NXP i.MX8QM的CAAM模块）编写驱动。整个过程遵循标准的内核驱动模型。

首先，你需要定义并填充算法描述结构。这是驱动的核心数据结构。

// 定义AES-256-GCM算法
static struct aead_alg my_aes_gcm_alg = {
    .base = {
        .cra_name = "gcm(aes)",
        .cra_driver_name = "gcm-aes-myhw",
        .cra_priority = 300, // 硬件加速优先级高于软件(100)
        .cra_module = THIS_MODULE,
        .cra_blocksize = 1,
        .cra_ctxsize = sizeof(struct my_hw_ctx),
    },
    .ivsize = GCM_AES_IV_SIZE,
    .maxauthsize = AES_BLOCK_SIZE,
    .setkey = my_aes_gcm_setkey, // 设置密钥
    .setauthsize = my_aes_gcm_setauthsize,
    .encrypt = my_aes_gcm_encrypt, // 加密
    .decrypt = my_aes_gcm_decrypt, // 解密
    .init = my_aes_gcm_init,
    .exit = my_aes_gcm_exit,
};

cra_priority是关键参数。当多个算法（如软件AES和你的硬件AES）同名时，内核选择优先级高的。硬件驱动通常设为200-400，确保被优先使用。

⚠️ 常见误区

  • 混淆算法名与驱动名：cra_name是标准算法名（如“cbc(aes)”），用户通过它请求服务；cra_driver_name是你的驱动唯一标识。把驱动名当算法名注册会导致crypto_alloc_aead失败。
  • 忽视异步操作：硬件加密通常是异步的（触发后立即返回，通过中断或轮询得知完成）。必须在回调函数中正确调用aead_request_complete来通知上层请求完成，否则调用方会永远阻塞。
  • 密钥管理不当：在setkey回调中，必须将用户提供的密钥拷贝到驱动内部结构或硬件寄存器。直接保存用户空间指针是严重的安全漏洞和内核Oops的根源。

在驱动的probe函数中，你需要注册这个算法：

int my_hw_probe(struct platform_device *pdev) {
    ...
    int ret = crypto_register_aead(&my_aes_gcm_alg);
    if (ret) {
        dev_err(&pdev->dev, "Failed to register AES-GCM: %d\n", ret);
        return ret;
    }
    // 同样注册SHA-256算法: crypto_register_ahash()
    ...
}

33.3 实战：在内核驱动中使用加密服务

你的另一个驱动（比如一个存储驱动）需要加密数据，可以直接调用Crypto API，而无需关心底层是硬件还是软件实现。

场景：加密写入的数据块

  • 使用对称加密（如AES-CBC）
  • 流程：分配转换对象 -> 设置密钥 -> 执行加密

场景：计算文件哈希

  • 使用散列算法（如SHA-256）
  • 用于数据完整性校验

场景：网络包认证

  • 使用AEAD模式（如AES-GCM）
  • 同时提供加密和完整性保护

下面是一个在驱动函数中使用AES-CBC加密一段内存的典型代码片段：

#include 
#include 

int my_driver_encrypt_block(const u8 *key, const u8 *iv, const u8 *plain, u8 *cipher, size_t len) {
    struct crypto_skcipher *tfm = NULL;
    struct skcipher_request *req = NULL;
    struct scatterlist src_sg, dst_sg;
    int ret = 0;

    // 1. 分配转换对象（“料理机”）
    tfm = crypto_alloc_skcipher("cbc(aes)", 0, 0);
    if (IS_ERR(tfm)) { ret = PTR_ERR(tfm); goto out; }

    // 2. 设置密钥（“放入食谱”）
    ret = crypto_skcipher_setkey(tfm, key, 32); // AES-256密钥长度32字节
    if (ret) { goto out_tfm; }

    // 3. 创建并配置请求
    req = skcipher_request_alloc(tfm, GFP_KERNEL);
    skcipher_request_set_callback(req, 0, NULL, NULL);
    sg_init_one(&src_sg, plain, len);
    sg_init_one(&dst_sg, cipher, len);
    skcipher_request_set_crypt(req, &src_sg, &dst_sg, len, iv);

    // 4. 执行加密操作（同步方式）
    ret = crypto_skcipher_encrypt(req);

    skcipher_request_free(req);
out_tfm:
    crypto_free_skcipher(tfm);
out:
    return ret;
}

我在一个基于i.MX6UL的项目中踩过坑：crypto_alloc_skcipher返回-ENOENT。排查后发现内核配置中CONFIG_CRYPTO_AES和CONFIG_CRYPTO_CBC没有启用，导致算法不存在。务必检查内核的.config文件。

编者提示：处理大量数据或频繁操作时，考虑使用异步接口和DMA，这与第21章《DMA引擎驱动》中描述的原则一致。异步加密可以避免阻塞进程，提升系统整体吞吐量。

33.4 内核密钥保留服务与完整性校验

密钥不能硬编码在驱动代码里。Linux内核提供了密钥保留服务（Key Retention Service），允许安全地生成、存储和使用密钥。

除了加密，数据完整性同样重要。例如，确保固件镜像在加载前未被篡改。你可以使用内核的crypto_shash_* API计算SHA-256哈希，并与预置的合法哈希值比对。

实际开发中的选择

大部分团队对于固件校验会在驱动加载早期（probe阶段）使用同步的crypto_shash计算哈希，因为此时数据量不大且操作一次性。对于传输中的数据流（如网络驱动），则会注册并使用异步的ahash接口，并配合DMA。

一个常见的完整性校验错误是忽略了哈希值的比较需要是常数时间的。使用简单的memcmp可能会被时序攻击推测出差异位置。内核提供了crypto_memneq函数用于安全比较。

动手试一试

1. 环境准备：在QEMU模拟的ARM64虚拟机（或你的开发板）上，确保内核配置启用了CONFIG_CRYPTO_USER_API_SKCIPHER和CONFIG_CRYPTO_AES。
2. 编写测试模块：创建一个内核模块，在init函数中调用crypto_alloc_skcipher("ecb(aes)", 0, 0)，如果成功，打印算法驱动名（通过crypto_skcipher_driver_name获取）和优先级，然后释放它。
3. 观察输出：加载模块，查看dmesg。你会看到类似“ecb-aes-aes-generic”或“ecb-aes-caam”的名字，这取决于你的系统使用的是软件实现还是硬件实现。

检验你的理解

  1. 当内核中同时存在软件实现的“sha256”和你的硬件实现的“sha256”时，Crypto API默认会使用哪一个？由什么参数控制？
  2. 在硬件加密驱动的encrypt回调函数中，加密操作完成后，必须调用哪个关键函数来通知请求完成？
  3. 如果你的驱动需要使用一个128位的密钥，你在setkey回调中，应该直接从用户提供的key指针读取，还是先进行拷贝？为什么？

本章小结

  • Linux内核Crypto API提供了统一的加密服务抽象层，驱动开发者通过注册crypto_alg结构体将硬件加密引擎接入系统。
  • 编写硬件加密驱动需实现算法描述结构（如aead_alg）并注册，关键步骤包括设备探测、硬件初始化、设置算法优先级和实现异步操作回调。
  • 在内核其他驱动中，可以通过crypto_alloc_xxx()、setkey()、encrypt()等标准流程使用加密服务，无需感知底层实现细节。
  • 安全编程至关重要，需避免密钥管理漏洞，对完整性校验使用常数时间比较，并合理选择同步/异步接口以平衡性能与复杂度。
