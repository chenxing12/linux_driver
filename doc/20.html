核心概念引脚控制通用输入输出

深夜，你正在调试一块基于RK3568的开发板，想点亮一个连接到GPIO1_B5引脚的LED。你直接调用gpio_set_value(45, 1)，但LED毫无反应。用示波器一测，引脚根本没有电平变化。这不是代码逻辑问题，而是你忽略了最关键的一步：这个引脚可能还处于I2C1_SDA的复用模式。在复杂的SoC上，直接操作GPIO寄存器已成过去，内核的GPIO与Pinctrl子系统才是现代驱动开发的正确入口。

20.1 为什么需要Pinctrl和GPIO子系统？

你可以把SoC的引脚想象成多功能瑞士军刀上的一个工具头。这个“工具头”本身是金属片（物理引脚），但它可以是小刀（GPIO）、开瓶器（I2C功能）、螺丝刀（SPI功能）。Pinctrl子系统就是决定当前弹出哪个“工具头”的锁定机构，而GPIO子系统则是当它被锁定为“小刀”模式后，你来控制它开合的具体操作手柄。

技术定义上，Pinctrl（Pin Control）子系统负责管理芯片引脚的复用（Mux）、电气属性（如上拉/下拉、驱动强度、施密特触发）配置。GPIO（General Purpose Input/Output）子系统则在引脚被配置为GPIO功能后，提供一套统一的API来设置方向、读写电平、申请中断。两者分工明确：Pinctrl管“它是什么”，GPIO管“它做什么”。

20.2 子系统架构与数据流

理解数据流向能帮你快速定位问题。整个配置过程始于设备树，由内核解析并驱动各子系统协作。

设备树源文件 (.dts)
→
内核解析
→
Pinctrl驱动
↓
GPIO控制器驱动
⇄
你的设备驱动
↓
实际硬件引脚

以点亮一个LED为例，具体步骤是：1）在设备树中，通过Pinctrl节点将指定引脚复用为GPIO功能；2）同样在设备树中，你的设备节点引用该Pinctrl配置，并声明使用的GPIO号；3）驱动加载时，内核自动应用Pinctrl配置；4）你的驱动调用GPIO子系统API控制电平。

常见误区

  • 跳过Pinctrl直接操作GPIO：这是新手最常踩的坑。如果引脚复用功能不对，GPIO操作完全无效。务必先确认引脚状态。
  • 混淆逻辑GPIO号和硬件引脚号：驱动中使用的gpio属性值是由GPIO控制器定义的逻辑编号，并非芯片手册上的物理引脚号（如“GPIO1_B5”）。需要通过计算或查表映射。
  • 忘记配置电气属性：对于输入引脚，特别是按键，不配置上拉/下拉会导致电平不稳定，读取的值随机跳动。

20.3 设备树配置详解

设备树是连接硬件描述和驱动代码的桥梁。配置错误，后续所有工作都是徒劳。一个完整的配置涉及Pinctrl和GPIO两部分。

Pinctrl节点配置

Pinctrl配置通常放在SoC级别的.dtsi文件中，由芯片厂商提供。但你需要在自己的板级.dts文件中覆盖或添加自定义状态。最常见的两个状态是default和sleep。

/* 在板级.dts文件中 */
&pinctrl {
    // 定义一个名为‘my_led_pin’的引脚配置组
    my_led_pin: my-led-pin {
        rockchip,pins = <<1 RK_PB5 RK_FUNC_GPIO &pcfg_pull_none>;
    };
    my_key_pin: my-key-pin {
        rockchip,pins = <<2 RK_PA3 RK_FUNC_GPIO &pcfg_pull_up>;
    };
};

以Rockchip平台为例，<1 RK_PB5 RK_FUNC_GPIO &pcfg_pull_none>表示：GPIO1组，B5引脚，复用为GPIO功能，上下拉配置为无。这里的“1”是GPIO组号，“RK_PB5”是宏定义的引脚偏移。

接下来，在你的设备节点中引用这些配置，并声明GPIO。

/* 你的设备节点 */
my_led_device {
    compatible = "vendor,my-led";
    status = "okay";

    // 引用Pinctrl配置，驱动probe时会自动应用‘default’状态
    pinctrl-names = "default";
    pinctrl-0 = <&my_led_pin>;

    // 声明使用的GPIO，‘0’是GPIO索引，‘&gpio1’指定控制器，‘GPIO_ACTIVE_HIGH’定义有效电平
    led-gpios = <&gpio1 RK_PB5 GPIO_ACTIVE_HIGH>;
};

my_key_device {
    compatible = "vendor,my-key";
    pinctrl-names = "default";
    pinctrl-0 = <&my_key_pin>;
    key-gpios = <&gpio2 RK_PA3 GPIO_ACTIVE_LOW>;
    interrupts-extended = <&gpio2 RK_PA3 IRQ_TYPE_EDGE_FALLING>;
};

编者提示：设备树语法是驱动开发的基础。如果对compatible、status、interrupts-extended等属性感到陌生，建议回顾第7章 设备树DTS语法解析，那里有从零开始的详细讲解。

20.4 GPIO子系统API实战

设备树配置好后，驱动中就可以使用GPIO子系统提供的简洁API了。我推荐使用描述符(descriptor) API（如gpiod_get），它比老旧的整数编号API（如gpio_request）更安全、更面向设备。

GPIO描述符API核心函数
函数作用备注
gpiod_get() / devm_gpiod_get()获取GPIO描述符后者为设备管理版本，自动释放
gpiod_direction_input() / gpiod_direction_output()设置输入/输出方向输出时可同时设置初始值
gpiod_get_value() / gpiod_set_value()读/写GPIO电平值非0即1
gpiod_to_irq()将GPIO描述符转换为中断号用于申请GPIO中断
gpiod_set_debounce()设置按键防抖时间（ms）不是所有GPIO控制器都支持

来看一个完整的驱动代码片段，它控制一个LED并检测一个按键：

#include <linux/gpio/consumer.h>
#include <linux/interrupt.h>

struct my_device_data {
    struct gpio_desc *led_gpio;
    struct gpio_desc *key_gpio;
    int irq;
};

static irqreturn_t key_irq_handler(int irq, void *dev_id) {
    struct my_device_data *data = dev_id;
    int key_val = gpiod_get_value(data->key_gpio);
    // 按键处理逻辑，注意中断上下文中不能调用可能睡眠的函数
    printk(KERN_INFO "Key pressed, value: %d\n", key_val);
    return IRQ_HANDLED;
}

static int my_driver_probe(struct platform_device *pdev) {
    struct device *dev = &pdev->dev;
    struct my_device_data *data;
    int ret;

    data = devm_kzalloc(dev, sizeof(*data), GFP_KERNEL);

    // 1. 获取LED GPIO，设备树中属性名为‘led-gpios’
    data->led_gpio = devm_gpiod_get(dev, "led", GPIOD_OUT_LOW);
    if (IS_ERR(data->led_gpio)) {
        return PTR_ERR(data->led_gpio);
    }

    // 2. 获取按键GPIO，配置为输入
    data->key_gpio = devm_gpiod_get(dev, "key", GPIOD_IN);
    if (IS_ERR(data->key_gpio)) {
        return PTR_ERR(data->key_gpio);
    }

    // 3. 将按键GPIO转换为中断，并申请中断
    data->irq = gpiod_to_irq(data->key_gpio);
    ret = devm_request_irq(dev, data->irq, key_irq_handler,
                           IRQF_TRIGGER_FALLING | IRQF_ONESHOT,
                           "my-key", data);
    if (ret) {
        dev_err(dev, "Failed to request IRQ: %d\n", ret);
        return ret;
    }

    // 4. 尝试设置按键防抖（硬件支持时）
    gpiod_set_debounce(data->key_gpio, 20); // 20ms防抖

    // 5. 点亮LED
    gpiod_set_value(data->led_gpio, 1);

    platform_set_drvdata(pdev, data);
    return 0;
}

场景：控制LED

  • 使用devm_gpiod_get(..., GPIOD_OUT_LOW/HIGH)一步获取并设为输出。
  • 调用gpiod_set_value()控制亮灭。
  • 无需手动调用gpiod_direction_output()。

场景：读取按键

  • 使用devm_gpiod_get(..., GPIOD_IN)获取输入GPIO。
  • 电平变化触发中断更高效，轮询会浪费CPU。
  • 务必在设备树配置上拉/下拉，确保空闲电平稳定。

场景：复用引脚切换

  • 在运行时切换引脚功能（如GPIO切为UART）需谨慎。
  • 调用pinctrl_lookup_state()和pinctrl_select_state()。
  • 必须确保没有其他驱动正在使用该引脚。

场景：调试与排查

  • 检查/sys/kernel/debug/pinctrl/pinctrl-handles查看引脚状态。
  • 检查/sys/kernel/debug/gpio查看GPIO使用情况。
  • 使用devm_gpiod_get_index()获取同一属性中的多个GPIO。

我在一个车载项目上踩过一个坑：按键中断触发异常频繁。最后发现是设备树里只配置了GPIO_ACTIVE_LOW，但忘了在Pinctrl节点里配置&pcfg_pull_up，导致输入引脚处于浮空状态，任何噪声都引发了误触发。加上上拉配置后问题立刻消失。

20.5 高级主题：GPIO映射与中断性能

对于GPIO数量众多的复杂板卡，直接在设备树里写死GPIO号不利于维护。内核提供了gpio-leds、gpio-keys等现成的基于GPIO子系统的驱动框架，它们通过更抽象的“label”来映射GPIO，建议优先使用。

关于GPIO中断性能，有一个关键细节：GPIO中断是共享中断。同一个GPIO控制器的所有中断可能共享一个上级GIC（通用中断控制器）的硬件中断线。这意味着你的中断处理函数要尽可能快，否则会影响同一条线上其他GPIO的中断响应。如果处理逻辑复杂，应该像第28章介绍的那样，使用工作队列或线程化中断将任务推后处理。

动手试一试

1. 硬件准备：找一块树莓派4B（BCM2711）或类似开发板，将一个LED通过限流电阻连接到GPIO17（物理引脚11），将一个按键连接到GPIO27（物理引脚13）并接地。
2. 修改设备树：在/boot/overlays/目录下创建或修改一个overlay文件，使用&gpio节点定义两个Pinctrl配置组（led_pins和key_pins），并将它们应用到两个新的设备节点（my_led和my_key）上，注意配置按键的上拉电阻。
3. 编写内核模块：参照20.4节的代码，编写一个平台驱动。在probe函数中点亮LED，并为按键申请下降沿中断，在中断处理函数中翻转LED的状态。
4. 测试：编译加载模块，按下按键，观察LED是否随之亮灭变化。通过cat /proc/interrupts查看你的中断是否被正确注册和触发。

检验你的理解

  1. 如果一个引脚在设备树中被配置为&uart2_pin（UART功能），你的驱动还能成功调用gpiod_get获取它并控制电平吗？为什么？
  2. 在驱动代码中，devm_gpiod_get(&pdev->dev, "foo", GPIOD_OUT_LOW)这行代码执行时，内核内部依次发生了哪些关键操作？（提示：从设备树查找开始）
  3. 你发现某个输入GPIO读取的值在0和1之间随机跳动，可能是什么原因造成的？请列出至少两种排查思路。

本章小结

  • 明确分工：Pinctrl子系统管理引脚“是什么”（复用、电气属性），GPIO子系统管理引脚被设为GPIO后“做什么”（方向、电平、中断）。现代驱动必须两者配合使用。
  • 配置始于设备树：在设备树中正确编写Pinctrl配置组（定义功能），并在设备节点中通过pinctrl-0引用和xxx-gpios属性声明，是驱动工作的前提。
  • 使用描述符API：优先使用gpiod_*系列函数（如devm_gpiod_get），它们更安全、更直观。获取GPIO时指定初始方向（GPIOD_OUT_LOW、GPIOD_IN）可简化代码。
  • 重视电气属性：输入引脚，特别是按键，必须在Pinctrl中配置明确的上拉或下拉电阻，避免浮空导致电平不稳定或误触发中断。
  • 调试有工具：遇到问题，首先查看/sys/kernel/debug/pinctrl/和/sys/kernel/debug/gpio下的调试信息，可以快速确认引脚状态和GPIO分配是否正确。
