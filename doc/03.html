核心概念竞态条件临界区

想象一下，一个共享的打印机被办公室里的多个人同时发送打印任务，如果不加管理，打印出来的文档会混杂在一起，这就是“竞态条件”。在Linux内核中，当多个执行路径（进程、中断、内核线程）同时访问和操作同一份共享数据（如设备寄存器、全局变量）时，如果缺乏保护，就会导致数据错乱、系统崩溃等严重后果。本章的目标就是为你装备应对这些“办公室打印混乱”的工具。

3.1 内核并发从何而来？

驱动代码运行在内核态，它面临的多线程环境比用户态程序复杂得多。主要并发源有三个：

  • 对称多处理（SMP）：你的设备可能插在双核的ARM Cortex-A53或四核的Intel i5平台上，多个CPU核心会同时执行内核代码。
  • 内核抢占：即使单核CPU，高优先级进程也可以抢占正在内核态运行的低优先级进程。
  • 中断：硬件中断（如网络包到达、按键按下）几乎可以在任何时刻打断当前执行流。

我在一个读取温度传感器的驱动项目中踩过坑：一个用户进程正在通过read系统调用读取传感器数据，此时传感器数据更新的硬件中断发生，直接修改了共享的数据缓冲区，导致用户读到了半新半旧的数据。这就是典型的由中断引起的竞态。

3.2 保护临界区：自旋锁 vs 信号量

临界区是指访问共享资源的代码段。保护它的两种最基本武器是自旋锁（spinlock）和信号量（semaphore）。它们的核心区别在于等待锁时的行为。

特性自旋锁 (spinlock_t)信号量 (semaphore)
等待行为忙等待（循环检查）睡眠等待
持有时间极短（纳秒/微秒级）可较长（毫秒级以上）
使用场景中断上下文、短临界区进程上下文、可能阻塞的操作
开销消耗CPU周期涉及进程调度开销
初始化APIspin_lock_init(&lock)sema_init(&sem, 1)
加锁APIspin_lock(&lock)down(&sem) 或 down_interruptible(&sem)
解锁APIspin_unlock(&lock)up(&sem)

关键决策点：如果你要保护的代码可能在中断处理函数中被调用，或者临界区执行时间极短（比如只是对一个整数做加减），必须使用自旋锁。因为中断上下文不能睡眠。反之，如果临界区内可能包含copy_from_user、kmalloc（可能触发内存回收）等可能导致进程睡眠的操作，必须使用信号量。

来看一个自旋锁的典型用法，保护一个简单的设备状态计数器：

static DEFINE_SPINLOCK(my_lock); // 静态初始化自旋锁
static int device_open_count = 0;

int my_device_open(struct inode *inode, struct file *filp) {
    unsigned long flags;
    // 获取锁，并保存当前中断状态到flags，然后禁用本地CPU中断
    spin_lock_irqsave(&my_lock, flags);
    if (device_open_count++ > 0) {
        // 设备已打开，处理错误...
        device_open_count--;
        spin_unlock_irqrestore(&my_lock, flags);
        return -EBUSY;
    }
    // 执行实际的设备打开操作...
    spin_unlock_irqrestore(&my_lock, flags); // 恢复锁和中断状态
    return 0;
}

这里使用了spin_lock_irqsave/spin_unlock_irqrestore这对宏。这是处理可能被中断打断的临界区的标准做法。它会在加锁的同时保存当前中断使能状态并禁用本地CPU中断，解锁时再恢复。如果你看到驱动代码里只用spin_lock，那多半是在确保不会有中断访问该临界区的情况下。

常见误区

  • 在持有自旋锁时调用可能睡眠的函数：这是致命错误，会导致系统死锁或内核崩溃。常见的睡眠函数包括kmalloc(GFP_KERNEL)、copy_from_user、msleep等。
  • 忘记解锁：特别是错误处理分支中容易遗漏解锁操作，导致锁被永久占用。
  • 锁的粒度不当：用一个“大锁”保护所有数据，会严重降低并发性能；锁的粒度太细，又会增加复杂度和死锁风险。实际开发中，大部分团队会为每个逻辑上独立的数据结构分配独立的锁。

3.3 原子操作：轻量级的计数器保护

对于简单的整型变量增减、位设置等操作，使用完整的锁机制好比用保险箱来保管一枚硬币，太重了。内核提供了原子操作（atomic operations），它们通过CPU提供的原子指令（如ARM的LDREX/STREX，x86的LOCK前缀）来保证单条指令执行的不可分割性。

原子增减

  • atomic_inc(&v) / atomic_dec(&v)
  • atomic_add(i, &v) / atomic_sub(i, &v)
  • 常用于引用计数

原子读写

  • atomic_read(&v) / atomic_set(&v, i)
  • atomic_cmpxchg(&v, old, new) (比较并交换)
  • 实现无锁算法基础

原子位操作

  • set_bit(nr, addr) / clear_bit(nr, addr)
  • test_and_set_bit(nr, addr)
  • 管理设备状态标志位

原子变量使用atomic_t类型定义。将之前的计数器改用原子操作实现，代码更简洁安全：

static atomic_t device_open_count = ATOMIC_INIT(0);

int my_device_open(struct inode *inode, struct file *filp) {
    // 如果原值是0，则设置为1并返回真；否则什么也不做返回假
    if (!atomic_add_unless(&device_open_count, 1, 0)) {
        return -EBUSY; // 设备已打开
    }
    // 执行实际的设备打开操作...
    return 0;
}

int my_device_release(struct inode *inode, struct file *filp) {
    atomic_dec(&device_open_count); // 引用计数减1
    return 0;
}

编者提示：原子操作虽然高效，但只能保护单个变量的单一操作。如果你需要保护一个涉及多个变量修改的复杂序列，或者临界区内有非原子操作（如对结构体多个字段赋值），仍然需要求助于自旋锁或信号量。关于内存分配（如kmalloc）可能引发的睡眠问题，我们将在第4章《内存管理与分配》中详细探讨。

3.4 内存屏障：指令重排的“栅栏”

现代CPU和编译器为了优化性能，会对指令进行重排序。这在单线程下没问题，但在多核并发环境下，可能导致一个CPU核心“看到”的操作顺序与另一个核心发出的操作顺序不一致。内存屏障（Memory Barrier）就是用来在代码中插入一个“栅栏”，确保屏障前的内存操作一定在屏障后的内存操作之前完成，并且对其他CPU可见。

CPU0 写入数据 A=1
→
写缓冲区/缓存
→
内存/其他CPU缓存
CPU0 写入标志 B=1
→
写缓冲区/缓存
→
内存/其他CPU缓存
❌ 无屏障时可能乱序
CPU1 读取标志 B==1
←
内存/其他CPU缓存
CPU1 读取数据 A (可能读到旧值0!)
←
内存/其他CPU缓存

内核提供了几种屏障：

  • mb() / rmb() / wmb()：通用/读/写内存屏障。
  • smp_mb() / smp_rmb() / smp_wmb()：在SMP系统上生效的屏障，在单核上是空操作。

一个典型的使用场景是“生产-消费”模式：生产者准备好数据后设置一个标志，消费者检查到这个标志后才去读数据。必须在写数据和写标志之间插入写屏障，在读标志和读数据之间插入读屏障：

// 生产者端
data_buffer[index] = new_data;
smp_wmb(); // 确保数据写入先于标志写入
ready_flag[index] = 1;

// 消费者端
if (ready_flag[index]) {
    smp_rmb(); // 确保读到标志后，再读到的数据是新的
    process_data(data_buffer[index]);
}

实际开发中，大部分内核同步原语（如自旋锁、原子操作）的内部已经包含了必要的内存屏障。但当你自己设计无锁数据结构或直接操作设备内存映射寄存器时，就需要显式使用它们。我在调试一个多核ARMv8（如Cortex-A72）平台的网卡驱动时，就曾因为遗漏写屏障，导致一个核心配置的DMA描述符地址未被另一个核心及时“看见”，引发数据丢失。

实用建议

对于驱动开发者，一个简单的原则是：在修改了硬件寄存器或共享内存，需要立即让其他执行路径（尤其是其他CPU或硬件DMA）看到效果的地方，考虑使用wmb()。在读取硬件寄存器或共享内存，且其值依赖于之前读到的另一个值（如状态标志）时，考虑使用rmb()。

动手试一试

1. 改造一个存在竞态的驱动：假设你有一个简单的字符设备驱动，其read函数直接返回一个全局整数counter，而一个内核定时器每隔1秒对counter加1。在不加保护的情况下，用户读取时可能正好遇到定时器更新，导致读到不完整的值。请分别用自旋锁和原子操作两种方式保护counter，并编写测试程序验证。

2. 分析锁的持有时间：使用ftrace的lock_stat功能（需要内核配置CONFIG_LOCK_STAT）跟踪你编写的驱动中自旋锁的竞争情况。命令示例：echo 1 > /sys/kernel/debug/tracing/events/lock/lock_acquire/enable，然后操作你的设备，最后查看/sys/kernel/debug/tracing/trace，观察锁的等待时间。

本章小结

  • 并发源：内核驱动面临SMP、内核抢占和中断三大并发源，必须对共享数据访问进行保护。
  • 锁的选择：自旋锁用于中断上下文和极短临界区（忙等待）；信号量用于进程上下文和可能阻塞的操作（睡眠等待）。切勿在持有自旋锁时调用可能睡眠的函数。
  • 原子操作：保护简单的整型或位操作的最高效方式，适用于引用计数、状态标志等场景。
  • 内存屏障：用于约束CPU和编译器的指令重排，确保多核间内存操作的可见性和顺序性，在操作设备寄存器或无锁编程时至关重要。
  • 核心思维：识别驱动中的所有共享数据（全局变量、静态变量、设备寄存器），为每个数据或相关联的数据集选择合适的保护机制，并明确临界区的边界。
