核心概念分散/聚集 (Scatter/Gather)传输描述符

你的网卡每秒要接收数百万个网络数据包，如果每个字节都让CPU来搬运，它早就累趴下了。这就像让公司CEO亲自去搬每一箱快递，效率极低。DMA（Direct Memory Access，直接内存访问）就是专门雇来的“快递员”，它能在CPU不干预的情况下，在设备和内存之间直接搬运数据。本章，我们深入Linux内核的dmaengine框架，看看如何驱动这位“超级快递员”。

21.1 DMA引擎框架：谁是老板，谁是员工

内核的dmaengine框架采用“生产者-消费者”模型进行抽象。你的设备驱动是“消费者”，它提出搬运需求（比如，把A处的1000字节搬到B处）。而SoC内部实际的DMA控制器硬件是“生产者”，它提供搬运能力。框架本身是“中介”，负责匹配需求与能力，并管理订单。

这个框架的核心是几个关键数据结构。首先，struct dma_device代表一个DMA控制器（比如TI的EDMA或ARM的PL330），它由控制器驱动注册。一个控制器可以有多个通道（struct dma_chan），每个通道独立执行传输任务。你的设备驱动需要申请一个通道，然后通过struct dma_async_tx_descriptor（传输描述符）来描述一次具体的传输。

关键点：为什么需要Scatter/Gather？

想象你要搬运的不是一整块连续内存，而是分散在物理内存各处的多个数据块（例如，一个网络数据包被拆分到多个SKB缓冲区）。Scatter/Gather（SG）DMA允许你用一个链表描述这些分散的块，DMA控制器能自动地依次搬运它们，这对处理网络数据流和文件I/O至关重要。在项目中，我常用它来处理摄像头传感器输出的非连续图像帧缓冲区。

21.2 发起一次DMA传输：四步流程

发起一次DMA传输，就像通过外卖平台点餐，有固定的步骤。下面这个流程适用于大多数内存到内存或内存到设备的传输。

1. 申请通道
→
2. 准备描述符
→
3. 提交并触发
→
4. 等待回调

我们来拆解每一步的代码实现。假设我们要从一块内核缓冲区src搬运len字节数据到设备（例如一个FIFO寄存器）。

/* 步骤1 & 2: 申请通道并准备描述符 */
struct dma_chan *chan;
struct dma_async_tx_descriptor *desc;
dma_cookie_t cookie;

// 通过设备树或名称申请通道。实际开发中，更推荐用设备树匹配。
chan = dma_request_chan(&your_device, "tx");
if (IS_ERR(chan)) {
    // 错误处理...
}

// 配置传输参数：源地址、目标地址、方向、长度
struct dma_slave_config config = {
    .direction = DMA_MEM_TO_DEV,
    .dst_addr = your_device_fifo_phys_addr, // 设备物理地址
    .dst_addr_width = DMA_SLAVE_BUSWIDTH_4_BYTES, // 总线宽度，例如32位
};
dmaengine_slave_config(chan, &config);

// 准备描述符。src是内核虚拟地址，框架会处理虚拟到物理的转换。
desc = dmaengine_prep_slave_single(chan, src, len, DMA_MEM_TO_DEV, 0);
if (!desc) {
    // 错误处理...
}

这里有个隐藏的坑：dst_addr必须是设备的物理地址，而不是IO映射后的虚拟地址（ioremap得到的结果）。我在一个音频驱动项目里踩过这个坑，错误配置导致数据全写到了未知的内存区域。

⚠️ 常见误区

  • 地址混淆：为DMA配置的地址必须是CPU物理地址或设备总线地址，不能直接使用内核虚拟地址（kmalloc返回的）或ioremap的地址。需要使用dma_map_single()等API进行映射。
  • 缓存一致性：如果源或目标内存是可缓存的（Cacheable），必须在传输前后调用dma_sync_*系列函数同步缓存，否则会读到脏数据或数据丢失。这是最易出错的地方之一。
  • 忘记释放：传输完成后，不仅要释放DMA通道（dma_release_channel），如果之前使用了dma_alloc_coherent分配了专用缓冲区，也必须对应释放。

继续完成后续步骤。我们需要设置一个回调函数，在传输完成后被调用，然后提交并启动传输。

/* 步骤3 & 4: 设置回调、提交并等待 */
void dma_callback_func(void *data) {
    struct completion *dma_complete = data;
    complete(dma_complete);
    printk(KERN_INFO "DMA transfer completed.\n");
}

DECLARE_COMPLETION(dma_complete);
desc->callback = dma_callback_func;
desc->callback_param = &dma_complete;

// 提交传输，获取一个cookie用于跟踪
cookie = dmaengine_submit(desc);
if (dma_submit_error(cookie)) {
    // 错误处理...
}

// 触发DMA控制器开始执行这个描述符
dma_async_issue_pending(chan);

// 等待传输完成。这里也可以用中断，而不是忙等待。
wait_for_completion_timeout(&dma_complete, msecs_to_jiffies(1000));

编者提示：回调函数中使用的complete()机制，是内核中常见的同步方式。如果你想深入了解内核中其他同步原语如信号量、互斥锁、等待队列的工作原理，可以回顾第3章：并发控制与同步机制。

21.3 关键API与缓冲区管理

DMA驱动开发中，选择正确的缓冲区分配和管理API至关重要。下表对比了三种主要方式：

API适用场景缓存一致性分配开销示例代码片段
dma_alloc_coherent需要长期存在的DMA缓冲区（如设备描述符环）一致性映射，无需手动同步高buf = dma_alloc_coherent(dev, size, &dma_handle, GFP_KERNEL);
dma_map_single/dma_unmap_single临时使用已有的普通内核缓冲区（如kmalloc分配）流式映射，必须手动同步缓存低dma_addr = dma_map_single(dev, kernel_buf, len, dir);
dma_pool_alloc需要分配大量小尺寸的DMA对象（如网络驱动中的描述符）取决于池的创建方式中buf = dma_pool_alloc(desc_pool, GFP_ATOMIC, &dma_handle);

我的经验是，对于高性能、数据频繁更新的场景（如网卡收发数据），使用dma_alloc_coherent最省心，但代价是可能损失一些CPU缓存性能。对于控制结构或不常更新的数据，这是首选。对于从通用内存分配器（如SLAB）获得的缓冲区进行一次性传输，必须使用dma_map_single，并在传输后立即dma_unmap_single。

生活类比：缓存一致性

假设CPU和DMA共享一个仓库（内存）。CPU有个手推车（缓存）用来临时存放常用货物。当CPU修改了仓库A区的货物，它可能只更新了手推车里的，仓库里的还没变（写缓存）。此时如果DMA直接从仓库A区搬运，搬走的就是旧货物。缓存同步操作（dma_sync_single_for_device）就是强制CPU把手推车里的新货物写回仓库，保证DMA拿到的是最新数据。

21.4 实战：SG DMA传输示例

让我们看一个更复杂的Scatter/Gather DMA示例，这在网络驱动中处理数据包散列（skb->data_frag）时是标准操作。

/* 假设我们有一个scatterlist数组sgl，包含nents个散列项 */
struct scatterlist sgl[MAX_ENTRIES];
struct dma_async_tx_descriptor *desc;
int i, nents;

// 1. 初始化scatterlist
sg_init_table(sgl, nents);
for (i = 0; i < nents; i++) {
    // 假设pages[i]是包含数据的页面，offset是页内偏移，len是长度
    sg_set_page(&sgl[i], pages[i], len, offset);
}

// 2. 映射SG列表，获取DMA地址
nents = dma_map_sg(dev, sgl, nents, DMA_TO_DEVICE);
if (nents == 0) {
    // 映射失败
}

// 3. 准备SG DMA描述符
desc = dmaengine_prep_slave_sg(chan, sgl, nents, DMA_MEM_TO_DEV, DMA_PREP_INTERRUPT);
if (!desc) {
    dma_unmap_sg(dev, sgl, nents, DMA_TO_DEVICE);
    // 错误处理...
}

// 4. 设置回调、提交、触发（同单缓冲区示例）
desc->callback = my_callback;
cookie = dmaengine_submit(desc);
dma_async_issue_pending(chan);

// 5. 在回调函数中，务必取消映射！
void my_callback(void *data) {
    dma_unmap_sg(dev, sgl, nents, DMA_TO_DEVICE);
    // ... 其他后处理
}

如果你在驱动中看到DMA-API: device driver tries to sync DMA memory it has not allocated这类内核告警，大概率是映射（map）和取消映射（unmap）没有成对调用，或者传错了设备指针dev。

音频Codec驱动

  • 场景：将PCM音频数据从内存搬运到I2S TX FIFO。
  • 要点：使用循环双缓冲区（Ping-Pong Buffer）和SG DMA实现连续播放。配置DMA为循环模式（Cyclic Mode）。
  • 参数：数据宽度16/24位，主时钟（MCLK）通常为12.288MHz或11.2896MHz。

MMC/SD卡驱动

  • 场景：读写SD卡扇区数据。
  • 要点：数据块大（通常512字节），使用SG DMA处理可能不连续的多个扇区请求。需要处理DMA与SD控制器内部FIFO的宽度对齐。
  • 参数：SDHC卡支持高达104MB/s（UHS-I）的传输。

图像传感器驱动

  • 场景：将CMOS传感器输出的RAW图像数据搬运到内存。
  • 要点：传输数据量大（一帧1080P RGB图像约6MB），对实时性要求高。通常使用专用视频DMA控制器（VDMA）。
  • 参数：像素时钟（PCLK）可能高达150MHz，需要精确计算DMA突发长度以匹配总线带宽。

动手试一试

1. 环境搭建：在QEMU中模拟一个ARM Versatile Express板（例如使用vexpress-a9机器模型），它包含一个PL330 DMA控制器。确保内核配置已启用CONFIG_DMA_ENGINE和CONFIG_AMBA_PL08X（PL330的驱动）。

2. 编写一个内核模块：实现一个简单的内存到内存的DMA拷贝。模块加载时，分配两个512字节的一致性缓冲区（dma_alloc_coherent），用数据填充源缓冲区，然后发起DMA传输将其拷贝到目标缓冲区。在传输完成的回调中，验证数据是否正确，并通过printk输出结果和耗时（可使用ktime_get）。最后在模块卸载时释放所有资源。

3. 进阶挑战：修改上述模块，使用kmalloc分配普通缓冲区，然后通过dma_map_single进行映射，体验缓存同步的必要性（尝试注释掉dma_sync_*调用，观察数据是否出错）。

本章小结

  • 框架核心：Linux dmaengine框架抽象了DMA控制器，通过通道（dma_chan）和描述符（dma_async_tx_descriptor）管理传输，支持高效的Scatter/Gather操作。
  • 标准流程：发起DMA传输遵循“申请通道→配置参数→准备描述符→设置回调→提交→触发→等待完成”的四步流程，务必在回调中清理资源。
  • 缓冲区管理是关键：根据缓冲区生命周期和性能需求，正确选择dma_alloc_coherent（一致性）或dma_map_single（流式）API，并严格遵守映射/取消映射、缓存同步的配对规则。
  • 实战要点：地址配置必须使用物理/总线地址；SG DMA是处理非连续数据的主流方式；不同外设（音频、网络、图像）对DMA的使用模式有特定优化点。
