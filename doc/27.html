核心概念hrtimerjiffies

驱动里需要“等一会儿再做事”或者“每隔固定时间做一次事”，比如按键按下后等待20ms再判断状态以防抖动，或者每秒钟读取一次传感器温度。内核提供了两种核心机制：标准定时器和高精度定时器。它们的区别就像手机的普通闹钟和秒表，一个适合分钟级的提醒，一个适合毫秒级的精确计时。

内核的时间基准：jiffies与clocksource

理解定时器前，先要明白内核如何计时。系统启动时，会初始化一个硬件时钟源（如ARM的ARMv7 Generic Timer，x86的TSC或HPET）。内核定义了一个全局变量jiffies，记录自系统启动以来的“滴答”数。每个滴答对应一次时钟中断，频率由CONFIG_HZ定义（通常为100、250或1000）。

两种定时器对比与选型

直接看表格，这是选择定时器类型的最快方式。

特性标准定时器 (timer_list)高精度定时器 (hrtimer)
精度毫秒级 (1/HZ)，HZ=1000时约1ms纳秒级 (可到几十纳秒)
底层时钟源基于jiffies的tick直接读取硬件时钟寄存器
API复杂度简单，接口稳定稍复杂，功能强大

典型应用场景秒/分钟级超时、简单轮询、兼容旧代码音频采样、PWM控制、精确延时、性能分析

启动函数mod_timer()hrtimer_start()

我的建议很明确：新驱动一律使用高精度定时器（hrtimer）。除非你维护的旧驱动或兼容的内核版本低于2.6.16。hrtimer精度更高，对电源管理更友好，是现代驱动的标配。

标准定时器（timer_list）实战

我们从一段有问题的代码开始，这是我在调试一个触摸屏驱动时遇到的真实案例：

// 错误示例：静态初始化定时器
static struct timer_list my_timer = TIMER_INITIALIZER(my_func, 0, 0);

static int probe(struct platform_device *pdev) {
    // 错误：重复初始化
    init_timer(&my_timer);
    my_timer.function = my_func;
    my_timer.data = (unsigned long)dev;
    // 计划1秒后触发
    mod_timer(&my_timer, jiffies + HZ);
    return 0;
}

这段代码在模块卸载时会导致内核oops。问题在于TIMER_INITIALIZER和init_timer重复初始化，破坏了内部结构。正确做法是使用timer_setup()（Linux 4.15+）或动态分配。

// 正确示例：动态初始化与使用
static void my_timer_callback(struct timer_list *t) {
    struct my_device *dev = from_timer(dev, t, timer);
    // 执行你的任务，例如读取GPIO状态
    int val = gpio_get_value(dev->gpio);
    // 重新激活定时器，实现周期性执行（单次触发则无需此步）
    mod_timer(&dev->timer, jiffies + msecs_to_jiffies(100));
}

static int probe(struct platform_device *pdev) {
    struct my_device *dev;
    // ... 设备初始化 ...
    // 使用 timer_setup 初始化，第三个参数是flags，通常为0
    timer_setup(&dev->timer, my_timer_callback, 0);
    // 启动定时器，100毫秒后触发
    mod_timer(&dev->timer, jiffies + msecs_to_jiffies(100));
    return 0;
}

static int remove(struct platform_device *pdev) {
    struct my_device *dev = platform_get_drvdata(pdev);
    // 必须同步删除，确保回调不会在模块卸载后执行
    del_timer_sync(&dev->timer);
    return 0;
}

常见误区与避坑指南

  • 竞态条件（Race Condition）：在定时器回调函数中，不要直接访问可能已被释放的设备结构体。务必使用from_timer宏安全获取容器结构，并确保在remove函数中先del_timer_sync再释放资源。
  • 耗时操作：定时器回调在软中断上下文执行，不能睡眠（不能调用kmalloc(GFP_KERNEL)、mutex_lock等）。如果需要长时间操作，应使用工作队列（workqueue），这是下一章的内容。
  • 精度误解：即使设置HZ=1000timer_list的精度也受系统负载和tick影响，不保证严格准时。对于精确间隔，必须用hrtimer。

高精度定时器（hrtimer）深入解析

高精度定时器直接基于硬件时钟，不依赖jiffies。它的核心是struct hrtimer和ktime_t时间类型。你可以把ktime_t想象成一个64位的纳秒计数器。

使用hrtimer的关键是选择正确的时钟模式和回调模式。时钟模式决定时间参考系：

CLOCK_MONOTONIC

  • 系统启动后经过的时间
  • 不受NTP调整影响
  • 推荐用于间隔测量

CLOCK_REALTIME

  • 墙上时钟时间
  • 会被NTP或用户修改
  • 用于需要对齐真实时间的任务

CLOCK_BOOTTIME

  • 包含系统休眠时间
  • 用于需要累计运行时间的场景
// hrtimer 完整使用示例：实现一个精确的10ms周期性任务
#include <linux/hrtimer.h>
#include <linux/ktime.h>

static struct hrtimer my_hrtimer;
static ktime_t interval;

// 回调函数，返回枚举值决定定时器行为
static enum hrtimer_restart hrtimer_callback(struct hrtimer *timer) {
    // 执行你的高精度任务，例如触发一个GPIO脉冲
    gpio_set_value(123, 1);
    udelay(5); // 模拟一个5微秒的脉冲
    gpio_set_value(123, 0);

    // 重新启动定时器，实现周期性
    hrtimer_forward_now(timer, interval);
    return HRTIMER_RESTART; // 返回 RESTART 表示继续
}

static int probe(struct platform_device *pdev) {
    // 设置间隔为10毫秒
    interval = ms_to_ktime(10);

    // 初始化hrtimer，选择CLOCK_MONOTONIC时钟和HRTIMER_MODE_REL相对模式
    hrtimer_init(&my_hrtimer, CLOCK_MONOTONIC, HRTIMER_MODE_REL);
    my_hrtimer.function = &hrtimer_callback;

    // 启动定时器，10ms后首次触发
    hrtimer_start(&my_hrtimer, interval, HRTIMER_MODE_REL);
    return 0;
}

static int remove(struct platform_device *pdev) {
    // 尝试取消定时器，如果回调正在执行则等待其完成
    int ret = hrtimer_cancel(&my_hrtimer);
    if (ret == 0)
        pr_info("Timer was not active.\n");
    return 0;
}

这里有个隐藏的坑：hrtimer_cancel()的返回值。返回0表示定时器未被激活（可能已触发或从未启动），返回1表示成功取消。不要假设返回值总是1。

编者提示：高精度定时器的回调同样在中断上下文执行。如果你需要在回调中进行可能睡眠或耗时的操作（如访问I2C/SPI设备），请结合第28章的工作队列（workqueue）机制，在定时器回调中调度一个工作项到工作队列执行。

内核延时函数：何时该“等一等”

除了定时器，驱动中经常需要主动等待一小段时间。内核提供了不同精度的延时函数，选择错误会严重影响性能。

延时函数精度是否让出CPU适用上下文典型最小延时
ndelay(ns)纳秒级否（忙等待）任何几十纳秒
udelay(us)微秒级否（忙等待）任何1微秒
mdelay(ms)毫秒级否（忙等待）任何1毫秒
usleep_range(min, max)微秒级是（睡眠）进程上下文10微秒
msleep(ms)毫秒级是（睡眠）进程上下文10毫秒
ssleep(s)秒级是（睡眠）进程上下文1秒

黄金法则：在中断上下文（包括定时器回调）中，只能使用忙等待的ndelay/udelay/mdelay。在进程上下文中，优先使用能让出CPU的usleep_range，它比msleep更精确且对电源管理更友好。

// 错误：在中断处理函数中调用可能睡眠的延时
irqreturn_t irq_handler(int irq, void *dev_id) {
    // 下面这行会导致内核告警或死锁
    // msleep(20); // WRONG!
    // 正确做法：使用忙等待
    udelay(200); // 等待200微秒
    return IRQ_HANDLED;
}

// 正确：在进程上下文中使用带范围的睡眠
ssize_t device_write(struct file *file, const char __user *buf, size_t count, loff_t *ppos) {
    // 等待15ms ± 10%，更省电且允许调度器运行其他任务
    usleep_range(13500, 16500);
    return count;
}

实际开发中，我见过最隐蔽的bug是在probe函数里用mdelay(1000)等待芯片上电稳定，导致整个启动过程卡住1秒。应该改用msleep，或者更好的办法是查询芯片的就绪状态寄存器。

  1. 使用高精度定时器（hrtimer）创建一个周期为50毫秒的定时任务。
  2. 在定时器回调函数中，通过printk输出当前精确时间戳，使用ktime_get_real_ns()获取纳秒级时间。
  3. 在模块的init函数中，分别用udelay(100)和usleep_range(9500, 10500)进行延时，并用ktime_get测量两者的实际耗时（提示：在延时前后调用ktime_get并计算差值）。

通过这个实验，你会直观感受到不同延时API的精度和行为差异。

本章小结

  • 定时器选型：新驱动优先使用高精度定时器（hrtimer），它提供纳秒级精度并支持tickless省电模式；标准定时器（timer_list）仅用于简单超时或兼容旧代码。
  • 核心API：掌握timer_setup()/mod_timer()/del_timer_sync()用于标准定时器；掌握hrtimer_init()/hrtimer_start()/hrtimer_cancel()以及HRTIMER_RESTART返回值用于高精度定时器。
  • 上下文约束：定时器回调在软中断上下文执行，禁止睡眠；耗时操作应移交工作队列。
  • 延时函数选择：中断上下文只能用忙等待（udelay）；进程上下文优先用可睡眠的usleep_range，它比msleep更精确、更省电。
  • 避坑关键：始终使用del_timer_sync或hrtimer_cancel在模块退出前安全停止定时器，防止回调访问已释放内存。
