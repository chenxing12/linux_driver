核心概念中断处理程序中断上下文

想象一下你在看书，突然手机响了，你会先做个书签标记当前页，然后去接电话，接完再回来继续看书。中断（Interrupt）就是硬件（比如手机）打断CPU（你）当前工作，要求CPU立即处理紧急事务（接电话）的机制。在技术定义上，中断是外部设备或内部异常事件向处理器发出的请求信号，迫使处理器暂停当前执行流，转而执行特定的服务程序。

中断处理的核心流程

一个完整的中断处理，内核将其分成了“上半部”（Top Half）和“下半部”（Bottom Half）两部分。上半部要求快速响应，只做最紧急的工作（比如读取硬件状态寄存器），然后调度下半部去处理耗时的任务（比如数据处理、唤醒进程）。

硬件中断发生
→
CPU跳转至ISR（上半部）
↓
紧急处理
（读寄存器、应答硬件）
↘
调度下半部
（工作队列、软中断等）
↓
耗时任务处理
（可睡眠、可调度）

中断处理程序（ISR）

中断处理程序（Interrupt Service Routine, ISR）是直接响应中断信号的函数。它运行在特殊的“中断上下文”中，这意味着它不能睡眠、不能调用可能引起阻塞的函数（如copy_from_user）。我在调试一个触摸屏驱动时，曾在ISR里误用了msleep，直接导致内核Oops（死机）。

如何注册一个中断处理程序

在Linux内核中，使用request_irq函数向系统申请一个中断号并绑定你的处理函数。这是驱动与硬件中断建立联系的关键一步。

// 典型的中断注册代码片段
static irqreturn_t my_irq_handler(int irq, void *dev_id) {
    // 1. 读取硬件中断状态寄存器，确认中断来源
    unsigned int status = readl(base_addr + REG_STATUS);
    if (!(status & INT_FLAG)) {
        return IRQ_NONE; // 不是本设备中断
    }
    // 2. 清除硬件中断标志（非常重要！）
    writel(INT_FLAG, base_addr + REG_STATUS);
    // 3. 调度下半部处理数据
    schedule_work(&my_work);
    return IRQ_HANDLED;
}

int probe_function(...) {
    int ret;
    int irq_num = platform_get_irq(pdev, 0); // 从设备树获取中断号
    if (irq_num < 0) return irq_num;
    // 注册中断，标志设为共享中断和边沿触发
    ret = request_irq(irq_num, my_irq_handler,
                     IRQF_SHARED | IRQF_TRIGGER_RISING,
                     "my_device", my_dev_ptr);
    if (ret) {
        dev_err(&pdev->dev, "Failed to request IRQ %d\n", irq_num);
        return ret;
    }
}

关键参数解析：request_irq的标志位

request_irq的第三个参数是标志位，它决定了中断的触发方式和行为。选错标志是新手常见错误，会导致中断无法触发或系统不稳定。

常用中断请求标志（IRQF_*）详解
标志含义典型应用场景错误使用后果
IRQF_SHARED共享中断线。多个设备共用同一个物理中断号。PCI设备、某些GPIO中断复用。非共享设备使用此标志，可能导致中断冲突，系统日志出现“IRQ handler type mismatch”错误。
IRQF_TRIGGER_RISING上升沿触发。信号从低到高跳变时触发。按键（按下瞬间）、某些传感器数据就绪信号。对于电平触发中断使用边沿触发，会丢失中断或重复触发。
IRQF_TRIGGER_HIGH高电平触发。只要信号为高电平就持续触发。UART接收数据、某些DMA完成信号。如果不清除中断源，会导致中断风暴，CPU被100%占用。
IRQF_ONESHOT单次触发。中断处理完成后才重新使能中断线。线程化中断（threaded IRQ）必须使用。在线程化中断中忘记设置，可能导致中断嵌套和死锁。

常见误区：中断上下文“三不能”

在中断处理程序（上半部）中，你必须牢记以下限制，否则会引发内核崩溃或死锁：

  • 不能睡眠/阻塞：禁止调用msleep()、wait_event()、带_GFP_KERNEL标志的kmalloc()。我曾在一个项目里，在ISR中尝试分配大块内存（未用GFP_ATOMIC），直接触发内核告警“BUG: scheduling while atomic”。
  • 不能与用户空间交换数据：禁止调用copy_to_user()/copy_from_user()，因为此时可能没有有效的进程上下文。
  • 不能调用可能引起阻塞的锁：慎用互斥锁（mutex），优先使用自旋锁（spinlock），且获取锁之前必须用spin_lock_irqsave()关闭本地中断。

下半部处理机制选择

由于上半部的限制，所有耗时、可能睡眠的操作都应放到下半部。Linux提供了多种下半部机制，你需要根据延迟要求和操作复杂度来选择。

任务小，要求快

  • 软中断/ Tasklet
  • 仍在中断上下文，不能睡眠
  • 执行延迟极低（微秒级）
  • 适用于简单数据标记、统计

任务中，可延迟

  • 工作队列（Workqueue）
  • 在进程上下文运行，可以睡眠
  • 延迟毫秒级，由内核线程调度
  • 最常用，适合大部分场景

任务复杂，需调度

  • 线程化中断
  • 整个ISR都在内核线程中运行
  • 可以睡眠，可以调用复杂函数
  • 优先级可调，但开销最大

实际开发中，大部分团队会优先选择工作队列，它在易用性和功能性上取得了很好的平衡。下面是一个使用工作队列的完整示例：

// 定义工作和处理函数
static struct work_struct my_work;
static void my_work_handler(struct work_struct *work) {
    // 这里可以安全地睡眠、分配内存、与用户空间交互
    msleep(10);
    process_data();
    wake_up_interruptible(&wait_queue);
}

// 在驱动初始化函数中
INIT_WORK(&my_work, my_work_handler);

// 在中断上半部中调度工作
static irqreturn_t my_irq_handler(...) {
    ...
    schedule_work(&my_work); // 将工作提交到系统默认工作队列
    return IRQ_HANDLED;
}

编者提示：工作队列、Tasklet等机制内部都依赖于内核的并发控制。如果你对自旋锁、信号量等同步原语还不熟悉，建议回顾第3章《并发控制与同步机制》的内容，这对于编写安全的下半部代码至关重要。

中断的调试与排错

中断驱动不工作，通常表现为设备无响应，或者系统出现奇怪告警。你可以通过以下线索定位问题。

中断未触发
→
检查/proc/interrupts计数是否增加
→
否：硬件/触发标志问题
中断触发一次后停止
→
检查ISR是否清除了硬件中断标志
→
否：补上清除操作
系统卡死或Oops
→
检查ISR是否违反“三不能”原则
→
是：将耗时操作移到下半部

查看/proc/interrupts是首要手段。这个虚拟文件列出了每个中断号被触发的次数。如果你的设备中断号那一行的计数在操作设备时不动，那说明中断根本没报到CPU，问题可能出在硬件连接、设备树配置或request_irq失败。

实战技巧：获取中断号

在新内核（设备树普及后），不再推荐手动指定硬编码的中断号。正确做法是从设备树中解析：

int irq = platform_get_irq(pdev, 0); // 获取第一个中断号
int irq2 = platform_get_irq(pdev, 1); // 获取第二个中断号（如果有）

对于GPIO中断，使用gpio_to_irq函数。我在一个基于i.MX6UL的项目中，曾因为直接写死中断号100，而实际设备树分配的是101，导致驱动调试了一整天。

动手试一试

1. 编写一个虚拟中断驱动：创建一个内核模块，使用timer（见第27章）模拟周期性的中断信号（例如，每秒一次）。在中断处理程序中，打印一条信息到内核日志（printk），并调度一个工作队列，在工作队列处理函数中打印另一条信息。观察上半部和下半部的执行顺序和上下文。

2. 观察真实中断：在开发板上，加载一个已有的按键驱动（或GPIO中断示例驱动）。在操作按键的同时，使用cat /proc/interrupts命令观察对应中断号的中断计数是否增加。尝试修改中断标志（如将边沿触发改为电平触发），观察行为变化。

3. 触发一个常见错误：在中断上半部（ISR）中故意加入一个msleep(1000)，编译加载模块，观察系统反应（很可能内核会panic）。通过这个实验，深刻理解中断上下文的限制。

检验你的理解

  2. 在中断上半部中，为什么不能调用copy_from_user()函数？
  3. 如果/proc/interrupts显示你的设备中断号计数始终为0，可能的原因有哪些？（至少列出三点）
  4. 共享中断（IRQF_SHARED）和非共享中断在注册和处理程序编写上有什么关键区别？

本章小结

  • 中断是异步事件通知机制，处理流程分为快速响应的上半部和处理耗时任务的下半部。
  • 使用request_irq注册中断，关键要正确设置触发标志（如IRQF_TRIGGER_RISING）和共享标志。
  • 中断上下文有严格限制：不能睡眠、不能与用户空间交互、慎用锁。违反会导致内核崩溃。
  • 下半部机制首选工作队列，它允许睡眠，适合大多数需要延迟处理的场景。
  • 调试中断首先查看/proc/interrupts，并确保在ISR中正确清除硬件中断标志，避免中断丢失或中断风暴。
