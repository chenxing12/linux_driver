核心概念PWM子系统背光设备

开发板上的LCD屏幕背光突然只能全亮或全灭，无法调节亮度；蜂鸣器要么不响，要么只能发出一种单调的声音。这些问题通常是因为驱动没有正确利用PWM（脉冲宽度调制）信号。PWM就像是一个高速开关的水龙头，通过控制“开”和“关”的时间比例，来模拟出不同的“水流”大小，从而实现从亮度调节到电机调速的各种功能。

15.1 内核PWM子系统剖析

Linux内核将PWM控制器抽象为一个子系统，为驱动开发者提供了统一的编程接口。这类似于文件系统，无论底层是HDD还是SSD，上层都用统一的`open`、`read`、`write`接口。PWM子系统屏蔽了不同芯片（如i.MX6ULL的PWM1、RK3288的PWM2）的硬件差异。

背光/蜂鸣器驱动
↓
PWM核心层 (pwm.c)
↓
PWM芯片驱动 (如pwm-imx27.c)
↓
硬件PWM控制器

要使用一个PWM，首先需要从设备树中获取它。内核用`struct pwm_device`对象代表一个PWM通道。核心操作只有三个：配置、使能、关闭。

// 关键API示例
struct pwm_device *pwm_get(struct device *dev, const char *con_id);
int pwm_config(struct pwm_device *pwm, int duty_ns, int period_ns);
int pwm_enable(struct pwm_device *pwm);
void pwm_disable(struct pwm_device *pwm);

💡 参数单位：纳秒（ns）

`pwm_config`的参数单位是纳秒，而不是百分比。`period_ns`是周期（频率的倒数），`duty_ns`是高电平时间。例如，要产生一个频率1KHz（周期1,000,000 ns）、占空比50%的方波，应调用`pwm_config(pwm, 500000, 1000000)`。

⚠️ 常见误区

  • 混淆duty_ns与duty_cycle：`duty_ns`是绝对时间值，而占空比（duty cycle）是`duty_ns/period_ns`的比值。新手常误以为`duty_ns`是百分比。
  • 未检查返回值：`pwm_get`可能返回`ERR_PTR(-ENODEV)`，如果直接使用会导致内核Oops。必须用`IS_ERR()`判断。
  • 使能前未配置：调用`pwm_enable`前必须先调用`pwm_config`，否则PWM输出是未定义的，可能导致背光闪烁或蜂鸣器异响。

15.2 编写PWM背光驱动

背光驱动将PWM的占空比映射为屏幕的亮度等级。假设我们的硬件连接是：i.MX6ULL的PWM1通道0（引脚GPIO1_IO08）控制一个背光LED。

首先，在设备树中描述这个硬件关系：

/* 在设备树源文件 (.dts) 中 */
&backlight {
    compatible = "pwm-backlight";
    pwms = <&pwm1 0 50000>; // 使用pwm1的通道0，默认周期50000ns (20kHz)
    brightness-levels = <0 1 2 4 8 16 32 64 128 255>;
    default-brightness-level = <8>; // 默认亮度等级索引，对应128
    status = "okay";
};

&pwm1 {
    #pwm-cells = <2>;
    status = "okay";
};

驱动代码的核心是注册一个`platform_driver`，并在其`probe`函数中，解析设备树节点，创建`backlight_device`。内核已经提供了`pwm_backlight`驱动框架，大多数情况下我们只需配置设备树。但在定制化需求中，可能需要自己实现`probe`函数。

驱动模块初始化
→
platform_driver.probe被调用
→
从设备树获取PWM设备
→
创建并注册backlight_device
→
通过sysfs暴露亮度控制节点

驱动加载后，你会看到`/sys/class/backlight/backlight/`目录，其中的`brightness`和`max_brightness`文件用于控制亮度。用户空间可以通过`echo 5 > brightness`来设置亮度。

15.3 驱动无源蜂鸣器

无源蜂鸣器与背光LED不同，它需要特定频率的方波才能发声。改变频率就是改变音调，改变占空比可以微调音量或音色。这里有个关键点：蜂鸣器通常对频率有范围要求，比如2KHz到5KHz。用错误的频率驱动，要么不响，要么声音很小。

播放单音

  • 固定频率（如2.5KHz）
  • 固定占空比（通常50%）
  • 用于简单报警提示

播放简单旋律

  • 动态改变`period_ns`切换频率
  • 配合内核定时器控制节拍
  • 可播放《小星星》等简单曲目

下面是一个让蜂鸣器以2.5KHz频率鸣叫1秒的驱动代码片段：

static void buzzer_beep(struct buzzer_dev *dev)
{
    // 计算周期：1秒 / 2500Hz = 400,000 纳秒
    int period_ns = 1000000000 / 2500;
    int duty_ns = period_ns / 2; // 50%占空比

    pwm_config(dev->pwm, duty_ns, period_ns);
    pwm_enable(dev->pwm);

    // 使用内核定时器，1秒后关闭
    mod_timer(&dev->stop_timer, jiffies + msecs_to_jiffies(1000));
}

static void buzzer_stop_timer(unsigned long data)
{
    struct buzzer_dev *dev = (struct buzzer_dev *)data;
    pwm_disable(dev->pwm);
}

我在一个智能家居网关项目里踩过坑，蜂鸣器声音像蚊子叫。排查后发现是PWM控制器输出的驱动电流不足。解决方案是在PWM输出引脚后增加一个三极管进行电流放大。所以，如果驱动后蜂鸣器声音小，先查硬件电路，再查软件频率和占空比。

15.4 性能考量与调试

PWM驱动的性能瓶颈通常不在CPU，而在PWM控制器本身的精度和响应延迟。选择参数时需要考虑硬件限制。

参数典型范围影响调试建议
频率 (1/period_ns)几十Hz ~ 几MHz背光：>100Hz避免闪烁
蜂鸣器：2KHz-5KHz		用示波器测量实际输出波形
占空比精度取决于控制器位数
(如8位/16位)		8位精度只有256级亮度通过`pwm_get_state()`读取实际配置
极性 (polarity)PWM_POLARITY_NORMAL/INVERSED决定高电平有效还是低电平有效设备树中设置`pwm-invert = <1>`

调试时，最直接的工具是示波器。将探头接到PWM输出引脚，可以看到实际的波形、频率和占空比是否与代码配置一致。如果没有示波器，可以用万用表测量电压，PWM输出在使能时，平均电压应为 `VCC * duty_cycle`。

🔧 生活类比：PWM调光 vs. 电阻调光

老式台灯用可变电阻调光，电阻发热浪费能量。PWM调光则像快速开关灯，肉眼因视觉暂留感觉亮度变化，但开关本身几乎不耗能。这就是为什么现代LED背光都用PWM，高效且亮度控制线性好。

内核也提供了调试信息。通过`cat /sys/kernel/debug/pwm`可以查看系统中所有已注册的PWM设备及其状态（使能、周期、占空比）。这是排查“PWM申请失败”或“配置未生效”问题的第一站。

动手试一试

假设你有一块搭载全志H3芯片的开发板，其PWM0引脚（PA05）连接了一个背光LED。请完成以下任务：

  1. 编写设备树片段，为背光设备创建一个节点，命名为`backlight`，指定使用PWM0，默认频率为10KHz，亮度等级为0到10共11级，默认亮度为5。
  2. 编写一个简单的内核模块，在`module_init`函数中，通过`pwm_get()`获取名为`“backlight”`的PWM设备，并配置其输出一个频率1KHz、占空比30%的方波，然后使能它。
  3. 加载模块后，使用`cat /sys/kernel/debug/pwm`命令验证你的PWM配置是否已生效。

提示：全志H3的PWM设备树节点通常为`&pwm { status = “okay”; }`。

检验你的理解

  1. 如果调用`pwm_config(pwm, 300, 1000)`，产生的PWM信号频率是多少？占空比是多少？
  2. 在背光驱动的设备树中，`brightness-levels`数组的数值代表什么？它们是如何与PWM占空比关联的？
  3. 无源蜂鸣器在驱动时完全没有声音，列出三个可能的原因（分别从硬件、设备树、驱动代码角度）。

本章小结

  • PWM子系统是硬件抽象层：它通过`pwm_get`、`pwm_config`、`pwm_enable/disable`等核心API，为驱动提供统一的PWM控制接口，屏蔽了底层芯片差异。
  • 背光驱动是PWM的典型应用：通过设备树绑定PWM通道与背光设备，并利用内核的`pwm-backlight`框架或自行实现`probe`，将PWM占空比线性映射为屏幕亮度等级。
  • 蜂鸣器驱动关注频率：驱动无源蜂鸣器本质是输出特定频率的方波，通过动态改变`period_ns`来改变音调，并需注意硬件驱动能力。
  • 调试依赖硬件工具与内核信息：示波器是验证PWM波形的终极工具，而`/sys/kernel/debug/pwm`则是内核层排查配置问题的有效手段。
